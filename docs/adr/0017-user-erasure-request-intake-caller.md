# ADR-0017: Admin Service Is the Erasure-Request Intake Caller — Extends ADR-0010's Closed Category Enumeration With a Fifth Business Category

## Status

Accepted

## Context

`kart-user-service`'s own `architecture.md` (Architecture Agent pass) resolved every part of ADR-0016's GDPR/erasure policy it could resolve unilaterally — the intake mechanism is a synchronous internal REST call (`POST /internal/users/{userId}/erasure-requests`), never an event, for the same "a compliance-audited write needs a request-time confirmation" reasoning `kart-admin-service/architecture.md` already applies to its own audited back-office actions. What that document explicitly could not resolve, and named as "a genuine open item... a human should decide," is **who calls it**: no BRD section names a compliance/data-subject-request role, workflow, or owning service anywhere, and BRD §24.1's four platform-wide roles (Customer, Support Agent, Admin, Partner API) give no explicit grant for "process a right-to-erasure request."

This is not local to User Service. The one existing platform precedent for a back-office actor calling a user-scoped internal endpoint is Admin Service's `POST /internal/users/{userId}/lock`/`unlock` (ADR-0010's "user suspension" category) — but ADR-0010 fixed Admin Service's back-office operation set as **a closed enumeration for v1**, explicitly excluding anything outside its four named categories (catalog management, coupon issuance, user suspension, inventory replenishment) and explicitly requiring "a new, explicit extension to this table" before a fifth is added, rather than a silent expansion. Identity Service is the other plausible candidate (it owns the account/credential lifecycle and is itself a named `UserDataErased` consumer per ADR-0016's Consequences section), but neither `kart-identity-service/requirement-spec.md` nor its `architecture.md` names a customer-support/compliance-intake responsibility today, and Identity's own boundary (`kart-identity-service/architecture.md`) is scoped to authentication/authorization/session lifecycle, not back-office customer-request handling.

Whatever this ADR decides changes what `kart-admin-service` (or `kart-identity-service`) must expose and what `kart-user-service`'s own erasure-intake endpoint is called by — the same "affects another service's required contract" bar that already justified ADR-0006, ADR-0010, and ADR-0014.

## Decision

**Admin Service is the erasure-request intake caller, added as a fifth business category to ADR-0010's closed enumeration: "Compliance / erasure request intake."**

- Exercised via the BRD's existing `Support Agent` role (§24.1: "Read + limited write across customer-facing services"), not a new role — processing a verified data-subject erasure request is the same class of "a support operator acts on a specific customer's account/data on the customer's own request" action BRD §24.1 already grants Support Agent for Payment's capped refunds. No new platform role is invented.
- Mechanically identical to Admin's existing four categories: Admin Service invokes `kart-user-service`'s own `POST /internal/users/{userId}/erasure-requests` endpoint synchronously (internal, client-credentials service-to-service call, per `kart-user-service/architecture.md`'s already-resolved intake mechanism) — Admin's own PostgreSQL database gains no new stored data beyond its existing `admin_permission_grants`/`AdminActionPerformed` rows; it does not become a second owner of erasure state, the same non-negotiable property ADR-0010 already established for every other category.
- Grant-gated the same way as Admin's other four categories: a `compliance-erasure` value is added to `admin_permission_grants.category` (`kart-admin-service/requirement-spec.md` §6 Decision item 1's existing default-deny, category-scoped grant model already accommodates a new category value without a schema change — it is a `TEXT` column, not a fixed enum at the database layer).
- **Why Admin over Identity:** a customer-support-initiated data-subject request is a back-office/compliance operation — squarely BRD §2.1 item 20's "back-office operations, RBAC" framing — not an authentication concern. Identity already becomes a `UserDataErased` *consumer* under ADR-0016 (redacting its own credential rows) but that is a downstream redaction responsibility, not an intake/triggering one; conflating the two would make Identity both the trigger and a consumer of its own trigger's fan-out, an awkward self-referential shape Admin-as-caller avoids entirely. Admin already exercises an analogous "operator acts on a specific user account" pattern for user suspension (ADR-0010 category 3) — keeping every "an operator acts on a user account" back-office action under one category owner is simpler than splitting user-account-related operator actions across two different callers.
- **Why not a new, not-yet-scoped dedicated compliance-tooling surface:** inventing a 19th platform service/surface for one endpoint call would be substantial net-new scope no BRD section asks for. Admin Service's existing "back-office operations" framing (BRD §2.1 item 20) already generically covers exactly this class of action without expanding the platform's service count.
- **Not decided here:** the exact `/admin/*` route/payload shape for submitting an erasure request, and how Admin itself verifies a request is genuinely the account holder's own (identity-proofing of the request itself) — both are the API Design Agent's / Admin Service's own future documentation pass's job, the same way ADR-0010's original four categories left their own endpoint-level contracts to that stage.

## Consequences

- `kart-user-service/architecture.md`'s "genuine open item" (erasure-request intake caller) is closed by citation — its `POST /internal/users/{userId}/erasure-requests` endpoint's caller is Admin Service, matching the mechanism (synchronous, client-credentials-gated) that document already fixed independently of who the caller turned out to be.
- `kart-admin-service`'s own `requirement-spec.md` (§6 Decision items 1-2) and `architecture.md` need this fifth category and the new outbound edge to `kart-user-service` added in a future documentation pass — not applied to those files by this ADR itself, the same "decide the integration pattern, don't edit the sibling service's docs directly" boundary ADR-0010 and ADR-0016 already observed for their own cross-service consequences.
- Admin Service's synchronous outbound fan-out grows from five peers (Product, Category, Offer, Identity, Inventory, per ADR-0010) to six (adding User) — no new distributed-monolith risk, since `kart-admin-service/architecture.md` already established that each category's circuit breaker is independent (bulkhead-isolated) and a User Service outage would only block the new compliance-erasure category, not any of Admin's other four.
- Accepted risk: like ADR-0010's original four categories, "who is authorized to process an erasure request" is gated only by the coarse `Support Agent`/`Admin` role plus Admin's own category-scoped grant — no additional identity-proofing-of-the-requester step is specified here, since the BRD gives no basis to design one; a real compliance/legal review may add such a step later without changing this ADR's caller decision.

## Epic: kart-identity-service v1

Platform AuthN/token-issuance, session/refresh-token lifecycle, TOTP MFA, RBAC role/claim issuance, and SSO/identity federation (enterprise SAML/OIDC + customer social OIDC) — the platform's single token issuer and sole point of contact with any external IdP (BRD §24.1/§24.2; `architecture.md` Boundary Rationale). Eighteen vertical slices below map 1:1 to `api-contract.yaml`'s eighteen paths; no endpoint in the contract is left undecomposed and no ticket invents a route the contract doesn't already have.

## Flagged Gaps — Not Decomposed Into Tickets (would invent scope, not decompose it)

- **No public endpoint for native role elevation (grant `support_agent`/`admin` on a `user_roles` row).** `database-design.md` itself flags this ("an out-of-band administrative process no stated FR defines a public endpoint for") — no ticket, consistent with that flag.
- **No public endpoint for managing `idp_group_role_mappings`.** `database-design.md` itself flags this ("written by an out-of-band operator process; no endpoint for it exists yet... flagged here as a gap for a later API Design Agent pass") — no ticket, consistent with that flag.
- **RBAC-role-change-triggers-revocation enforcement** (`edge-cases.md`, "RBAC Role Change Outlives an Already-Minted JWT") has no ticket of its own because there is no role-change endpoint to trigger it (see the role-elevation gap above) — the mechanism itself (extend the Redis revocation list IDN-9/IDN-10 already write to) requires no new infrastructure once a role-change endpoint exists; wiring it in is a small addition to whichever future ticket implements that endpoint, not a standalone task today.
- **Gateway-side revocation-list consumption is out of scope for this service's own ticket list.** `architecture.md`'s Dependencies table states the API Gateway reads Identity-owned Redis keys directly as shared infrastructure, not via an API/event call — that read-side implementation belongs in the Gateway's own repo/ticket list. IDN-9, IDN-10, and IDN-13 above are responsible only for the write side (populating the revocation list).
- **Federation Single Logout / SLO propagation** — requirement-spec.md §6 explicitly carries this forward as a non-blocking, later-pipeline-stage item, and `architecture.md`'s "Sync vs. Async Resolution" section confirms no SLO webhook/SCIM integration is added to Identity's boundary in this pass (the 24-hour federated-refresh-token absolute cap, built as part of IDN-15/IDN-16, is the accepted v1 mitigation). No ticket — correctly out of v1 scope per both docs.

## Notes for Sprint Planner Agent

- IDN-1 (`GetJwks` / RS256 key provisioning) is the single foundation task — every token-issuing slice (IDN-2, IDN-3, IDN-7, IDN-14, IDN-17) depends on it directly, and everything else depends on one of those transitively. Start it first, alone.
- IDN-15 (`EnterpriseSamlAssertionConsumer`) and IDN-16 (`EnterpriseOidcCallback`) share the same underlying JIT-provisioning + fail-closed IdP-group→role-mapping domain logic (requirement-spec.md §2, resolved Q7) but are independent REST entry points with no lifecycle dependency on each other — recommend building IDN-15 first as the reference implementation (BRD §24.2 names SAML before OIDC) and having IDN-16 reuse its domain logic, but this is a sequencing recommendation, not a hard dependency edge.
- IDN-4 → IDN-5 → IDN-6 is this graph's longest chain from IDN-3 (enroll → confirm → verify-at-login), 3 deep; combined with IDN-1 → IDN-3 at the front, the longest chain overall is 5 nodes (IDN-1 → IDN-3 → IDN-4 → IDN-5 → IDN-6).
- IDN-10 → IDN-11 (lock → unlock) and IDN-12 → IDN-13 (reset-initiate → reset-confirm) are natural sequential pairs, mirroring `kart-offer-service/tickets.md`'s OFF-5/OFF-6 activate/deactivate precedent.
- IDN-8 (`RefreshToken`), IDN-9 (`Logout`), IDN-12 (`InitiatePasswordReset` via IDN-2), IDN-14 (`StartEnterpriseFederation`), and IDN-17 (`StartSocialLogin`) each depend on only one prior task and can otherwise be parallelized once their single dependency lands.
- IDN-19 (`UpdateProfile`) depends only on IDN-3 (must be authenticated to call it) and can be built any time after login exists. IDN-20 (`ConsumeUserDataErased`) has no dependency on any other ticket in this graph — it is a standalone event consumer triggered externally by User Service, not by another Identity endpoint — and can be built in parallel with everything else.
- No circular dependencies in this graph.

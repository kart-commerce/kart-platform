# Requirement Spec: kart-identity-service

## 1. Scope

Covers the single BRD service **Identity Service** (BRD §2.1 item 1: "AuthN, tokens, sessions, MFA, SSO federation, RBAC role/claim issuance"). No merge; no ADR applies. This is the only service in the BRD's core-modules table that no other row's Scope note or event catalog entry references as a merge candidate.

Unlike the prior draft of this spec, Identity is **not** limited to the condensed §5.4 row and scattered §24 mentions. The BRD gives it two dedicated cross-cutting subsections — §24.1 "Cross-Cutting RBAC Model" and §24.2 "SSO / Identity Federation" — that state, in explicit prose (not implication), that Identity is the platform's single role issuer and the sole termination point for both enterprise SSO federation and customer social login. This redraft incorporates both; the prior draft predated them and treated their content as unresolved open questions (see §6). This spec draws on §2.1, §5.4, §10, §18, §24, §24.1, and §24.2, plus ADR-0006 and ADR-0007 for the two event-catalog gaps those ADRs closed — it does not invent detail beyond what those sources state; remaining gaps are named explicitly in §6.

## 2. Functional Requirements

**Core AuthN / token issuance**

- Authenticate a user and issue tokens (`POST /auth/login`, BRD §5.4).
- Refresh an access token (`POST /auth/refresh`, BRD §5.4).
- Act as the platform's single token issuer: mint the JWT the API Gateway validates at the edge (BRD §18: "JWT validated at gateway edge; downstream services trust a signed internal header, not the raw client token") and that every other service trusts. This is now stated directly, not just implied by §18 — BRD §24.1 says outright: "Identity Service is the single issuer of roles; every other service is a **consumer** of the claim, never a second source of truth." Resolves what was previously Open Question #4.
- Terminate both OAuth2 flows named in BRD §24's AuthN row — Authorization Code (web/mobile clients) and Client Credentials (service-to-service) — at Identity Service. §24.1's Issuance bullet describes minting for a "user/service-principal," covering both flows, and §24.2 confirms federated logins "land on the same Identity Service token issuance path" as native login, i.e. there is exactly one issuance path, owned by Identity. Resolves what was previously Open Question #5.
- Support multi-factor authentication (BRD §2.1 item 1: "...MFA") using **TOTP (RFC 6238)** as the concrete mechanism. **Decision:** TOTP over SMS OTP (carrier delivery is phishable/unreliable and no SMS gateway is named anywhere in the BRD), WebAuthn (stronger, but assumes a hardware-key/platform-authenticator baseline the BRD never states), and email code (shares a delivery channel with password reset, weakening factor separation) — TOTP needs no new BRD-unstated infrastructure and is the industry-default second factor. **Enforcement default:** required at every login for the Admin and Support Agent roles (BRD §24.1's higher-privilege roles); optional, customer-elected, off by default for the Customer role; not applicable to Partner API (client-credentials flow is non-interactive). **Trade-off accepted:** Identity must store a per-user encrypted TOTP secret (covered by the existing AES-256-at-rest PII invariant, §4) rather than taking a stateless phone-OTP dependency. Resolves prior Open Question #1 (blocking); see also edge-cases.md, "Partial-Auth Window During MFA."
- Provide the session-termination and account-recovery endpoints necessary to make already-stated responsibilities usable, none of which appear in the BRD's condensed §5.4 table: `POST /auth/logout` (adds the current token to the revocation list — the only way BRD §24's "revocation list for logout" control and Domain Invariant §4's revocation invariant can actually be triggered), `POST /auth/mfa/verify` (completes the MFA challenge the bullet above requires, and which edge-cases.md's "Partial-Auth Window During MFA" already assumes exists), and `POST /auth/password/reset-initiate` / `POST /auth/password/reset-confirm` (a password-based native account model, which BRD §24.2 states exists alongside federation, is not supportable without a reset path). **Decision:** these are additive implementations of responsibilities the BRD already states, not new scope — resolves prior Open Question #6. Exact route names/verbs/versioning remain the API Design Agent's job (§5); only their existence as FRs is decided here.

**RBAC role/claim issuance (BRD §24.1)**

- At token-mint time, resolve the authenticated user or service-principal to a role set and embed it as scoped claims (`roles: [...]`, `scopes: [...]`) in the same JWT described above — this is the exact mechanism BRD §24.1's "Issuance" bullet describes, and is a new functional requirement this redraft adds; the prior draft had no FR for role-claim issuance at all.
- The four roles named in §24.1's table (Customer, Support Agent, Admin, Partner API) are the role vocabulary Identity resolves against; each role's actual scope/grant enforcement happens downstream (Gateway coarse-grained, owning service fine-grained per §24.1's "Enforcement" bullet) — Identity's own responsibility stops at correct role resolution and claim embedding, not downstream enforcement.

**SSO / Identity Federation (BRD §24.2)**

- Terminate SAML 2.0 or OIDC federation against the organization's enterprise IdP (Okta/Azure AD/Google Workspace) for Admin/back-office users, exchanging the external assertion for Kart's own short-lived JWT carrying Kart-native role claims (BRD §24.2). New FR this redraft adds.
- Terminate OIDC social login (Google/Apple/etc.) for customers, alongside native email/password signup, exchanging the external token for Kart's own JWT via the same issuance path as native login (BRD §24.2). New FR this redraft adds.
- Both federation paths terminate exclusively at Identity Service — no other service validates a SAML assertion or holds an external IdP client secret (BRD §24.2, explicit). See also Domain Invariants (§4).
- **External IdP role/group mapping (resolves prior Open Question #7 — blocking).** BRD §24.1/§24.2 state that Identity resolves every login (native or federated) to one of the four platform roles but never state how an enterprise IdP's own group/claims map onto those roles. **Decision:** customer social login always resolves to the `Customer` role only — the social IdP's own claims/groups, if any, are never consulted for role elevation, so there is no path from a Google/Apple login to `Support Agent`/`Admin`/`Partner API`. Enterprise IdP federation (SAML/OIDC for Admin/back-office) resolves role via a config-driven mapping table Identity maintains, keyed on the IdP's own group/claim value (e.g., an Okta/Azure AD group name) to exactly one Kart role — never hardcoded, never inferred from claim naming conventions. Absent an explicit mapping entry for a claimed group, the login still succeeds (account is JIT-provisioned per the FR below) but is granted **zero** platform roles (fail-closed), not a fallback to `Customer` or, worse, `Admin`. The `Admin` role is never auto-granted by the mere fact of a successful federated assertion — only an explicit, pre-created group-mapping entry can grant it. **Why:** a hardcoded or auto-inferred mapping would let the enterprise IdP's own group-naming convention silently become a second, un-audited definition of what "Admin" means, contradicting BRD §24.1's single-issuer/single-source-of-truth invariant; a config-driven mapping keeps Identity the sole authority while still letting the org's IdP admins manage group→role assignment without a Kart code deploy per change. **Trade-off accepted:** a legitimate new hire whose IdP group has no mapping row yet gets zero access until an operator creates that mapping — an intentional fail-closed cost, preferred over fail-open (auto-granting a role) for a table that includes `Admin`.
- **Federated login with no matching Kart account (resolves the "Federated Login With No Matching Kart Account" edge case, previously escalated in edge-cases.md).** **Decision:** just-in-time (JIT) provisioning on first successful federation, for **both** federation paths — a new Kart account/identity record is created automatically rather than requiring a separate out-of-band signup step. **Why:** for customer social login this matches the BRD's own self-service registration intent (§24.2: "reduces signup friction"); for enterprise federation, JIT-provisioning the account record (not the role) keeps the org's IdP as the single source of truth for "who is currently employed here" without a manual, out-of-sync Kart-side pre-provisioning step, while the role-mapping decision immediately above (fail-closed, zero roles absent an explicit mapping) is what actually prevents the governance risk the escalation was worried about — auto-granting `Admin` is impossible under that rule regardless of whether the account itself is JIT-created. **Trade-off accepted:** a JIT-provisioned enterprise account with no matching role mapping is a real account that exists but can access nothing, rather than a rejected login with a clearer error — considered acceptable since it fails closed either way, and avoids Identity having to distinguish "reject this login" from "reject this specific privilege" as two different code paths.

**Session / revocation**

- Create a session on successful authentication, publishing `SessionCreated` (BRD §5.4).
- Maintain a token revocation list, checked at logout (BRD §24: "...revocation list for logout").
- Support session lifecycle management implied by "sessions" (BRD §2.1 item 1) and `SessionCreated` (§5.4). The BRD states no session termination/expiry event (no `SessionExpired` or `SessionRevoked` in the event catalog, §10); §4's Domain Invariant resolves this deliberately as "no new event added" — natural TTL expiry (§4's concrete refresh-token TTLs) plus the existing revocation-list check cover this without a new Event Catalog row.

**Cross-service event publishing**

- Register a new user identity, publishing `UserRegistered` on success (BRD §5.4; consumed by User, Notification, and Analytics per the Event Catalog, §10, as extended by ADR-0007).
- Publish `UserAccountUpdated` (payload: `userId`, `email`, `displayName`) whenever Identity's copy of login email or display name changes **after** registration — not just at registration time — so User Service can reconcile its denormalized profile copy. This event and its Identity→User direction are stated directly in BRD §5.4's Identity row and §10's Event Catalog; the underlying data-ownership split (Identity owns login-credential-adjacent fields; User owns the broader profile and keeps a denormalized copy) is resolved by **ADR-0006**, cited here rather than re-derived. The prior draft of this spec had no FR for this event at all — its own Consumed-Events note ("§5.4 shows Identity consuming nothing") was correct and remains correct, but the omission was on the **publish** side, not consume.

**Admin-triggered account suspension (resolves prior Open Question #4 — "no consumed events")**

- Expose an internal, service-to-service-only write endpoint (`POST /internal/users/{userId}/lock`, `POST /internal/users/{userId}/unlock` — exact route naming is the API Design Agent's job) that locks/unlocks a user's ability to authenticate. **Decision:** this is a synchronous call **Admin Service invokes against Identity's own API**, not an event Identity consumes — per ADR-0010 (`docs/adr/0010-admin-service-scope-and-integration.md`), which already assigns "user suspension" to Identity as the owning service ("auth-identity fields are Identity's domain per ADR-0006's boundary, not User's") and states the integration pattern platform-wide is the owning service's write API invoked by Admin as a caller, never a duplicated copy of another service's data inside Admin. This closes what the prior draft flagged as "surprising" about Identity's empty Consumes list (§5.4): the reason Identity consumes no events is that the account-suspension trigger is a synchronous internal call, not an asynchronous one — Identity's Consumes list legitimately stays empty. A locked user's outstanding tokens must be invalidated via the same Redis-backed revocation-list mechanism already required for logout and role-change (see Domain Invariants, §4, and edge-cases.md's "RBAC Role Change Outlives an Already-Minted JWT").
- This endpoint must only be callable by Admin Service's service-principal (Client Credentials flow, `Admin`-scoped claim) — never reachable by a Customer- or Support-Agent-scoped token, and never exposed through the public API Gateway routes a browser/mobile client can reach.

**GDPR erasure consumption (resolves a gap the Architecture stage flagged, closed via an update to ADR-0016).** Identity consumes `UserDataErased` (published by User Service, ADR-0016). On receipt: overwrite `email`/`display_name` with tombstone values, clear `password_hash`, delete the corresponding `mfa_credentials` row, delete every `federated_identities` row for that user, and revoke every outstanding session/refresh-token family for that user via the existing Redis-backed revocation-list mechanism (the same one logout/role-change/admin-lock already use). **Decision:** ADR-0016 originally named Order, Notification, Analytics, Review, Recommendation, and Wishlist as consumers but not Identity, despite Identity storing exactly the kind of userId-linked PII that ADR governs (login email, password hash, encrypted TOTP secret) — this was flagged, not silently resolved, by `architecture.md` during this service's Architecture stage. **Why now:** the gap was a real omission, not an intentional exclusion — ADR-0016 has been updated to name Identity explicitly and specify its redaction shape, closing the gap at the source rather than inventing scope unilaterally in a downstream doc. **Trade-off accepted:** none material — this is the same tombstone pattern already established platform-wide, applied to Identity's own tables.

**Cross-cutting Security obligations added by BRD §24.1.2, §24.1.4, §24.1.5, §24.3 (cross-reference only — the detailed "why" lives in the BRD itself and in this service's own ddd-model.md/database-design.md, not re-derived here):**

- **§24.1.2 (CanRead/CanWrite/CanDelete, service-owned):** Identity is itself a resource-owning service for its own domain data (accounts, credentials, sessions, role grants, federation links) and defines its own fine-grained permission mechanism for that data, independently of its separate role as the platform's coarse-role *issuer* (§24.1.1) — being the issuer of the coarse claim does not exempt Identity's own tables from this tier, the same point BRD §24.1.2 makes generally. The concrete mechanism (ownership comparison for self-service resources, plus a narrow inline Admin-service-principal-only check for the two internal lock/unlock endpoints) is recorded as a Domain Invariant in ddd-model.md.
- **§24.1.4 (row-level security):** Identity's write model is 100% PostgreSQL, no MongoDB read model (database-design.md), so native RLS (`ENABLE ROW LEVEL SECURITY` / `CREATE POLICY`, keyed on a session-scoped `current_setting('app.current_principal')`) is the mechanism used, not the MongoDB query-builder alternative. See database-design.md's "Row-Level Security Policy" subsection for the per-table policies and the tables carved out as having no per-row end-user ownership concept.
- **§24.1.5 (column-level security):** Identity owns some of the platform's most sensitive columns by construction — credential hashes, encrypted TOTP secrets, refresh/reset token hashes, and SSO-federation subject identifiers. See database-design.md's "Sensitive / PII Column Classification" subsection for the per-column visibility rule.
- **§24.3 (default audit fields):** every mutable table in database-design.md now carries `created_at`/`updated_at`/`created_by`/`updated_by`, auto-injected by the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor reading the JWT principal (§24.3's own stated mechanism — one platform-wide implementation, not built locally by this service) — never a value a request handler sets directly. System-initiated mutations (self-registration/JIT provisioning, the Outbox poller, the `UserDataErased` consumer, reuse-detected token revocation) stamp a well-known `system:*` principal id, never `NULL`.

## 4. Domain Invariants

- A refresh token must be single-use and rotated on every refresh (BRD §24: "rotating refresh tokens"). **Reuse response (resolves prior Open Question #3 — blocking):** replay of an already-rotated-out refresh token must revoke that token's entire rotation family (every session descended from it), not just deny the single stale request or tear down every session the user holds elsewhere — formalizing edge-cases.md's "Refresh Token Replay After Rotation" Decision (token-family reuse detection) as a platform invariant rather than leaving it as an edge-case-local choice. **Why:** reject-and-continue leaves a live coexisting-compromise path open; full teardown of every session for the user is disproportionate collateral damage for unrelated, uncompromised sessions. **Trade-off accepted:** requires storing rotation lineage per token, not just a flat single-use flag.
- A revoked token (post-logout, forced role change, or admin-triggered lock — see §2's admin-suspension FR) must never be accepted again, regardless of its stated expiry (BRD §24: "revocation list for logout"), via the Redis-backed revocation-list check the Gateway consults (edge-cases.md, "Stale Revocation Under Stateless JWT Validation").
- An access token's validity window must not exceed the BRD's stated ~15 minutes (BRD §24, Token Lifecycle row).
- **Refresh-token / session TTL (resolves prior Open Question #2 — blocking).** **Decision:** a native (non-federated) refresh token uses a 30-day sliding window (each successful refresh resets the 30-day clock) bounded by a 90-day absolute cap from the session's original login, whichever comes first — the standard sliding-plus-absolute-cap pattern, chosen because a pure sliding window with no ceiling lets a continuously-active session live forever, while a pure fixed window forces disruptive re-logins on genuinely active users. A refresh token issued via a federated login (SAML/OIDC) instead gets a **24-hour absolute cap, no sliding extension**, per edge-cases.md's "Federated Session Can Outlive IdP-Side Revocation Indefinitely" — deliberately far shorter than the native cap because federation corresponds to the higher-privilege back-office population (§24.1) and Identity has no live signal of IdP-side deprovisioning (see Open Questions §6, item on SLO). **No new `SessionExpired`/`SessionRevoked` event is added to the Event Catalog for this** — natural token expiry plus the already-required revocation-list check are sufficient for the Gateway's enforcement needs, and the BRD names no consumer that needs a session-termination event for its own purposes; this is called out explicitly as a scoping decision, not a silent omission, and can be revisited if a concrete downstream consumer need is identified later.
- **External IdP role/group mapping (resolves prior Open Question #7 — blocking).** No other service may be granted a role via federation except through Identity's own explicit, config-driven group→role mapping (enterprise federation) or the fixed `Customer`-only default (social login) — see §2's SSO FR for the full decision. `Admin` is never auto-granted merely by a successful federated assertion.
- `UserRegistered` must be published exactly once per successful registration and must be idempotent-safe for its consumers (BRD §3 general Reliability rule; §5.4/§10 — User, Notification, and Analytics all consume it per ADR-0007).
- `UserAccountUpdated` must be published on every login-email or display-name change after registration (not only at registration) and must be idempotent-safe for User Service's consumer, mirroring the reliability bar already set for `UserRegistered` (BRD §5.4, §10; **ADR-0006**).
- Identity Service is the platform's single issuer of role/permission claims; no other service may maintain its own independent role or permission table as a second source of truth for authorization (BRD §24.1, explicit: "one issuer + one claim shape is what makes RBAC auditable across the whole platform"). This is a new invariant this redraft adds — the prior draft had no domain invariant covering RBAC at all.
- Identity Service is the platform's sole point of contact with any external IdP: no other service may hold an external IdP client secret or validate a SAML assertion / external OIDC token directly (BRD §24.2, explicit). New invariant this redraft adds.
- PII (at minimum, credentials and any stored personal identifiers) must be encrypted at rest with AES-256 (BRD §24, Encryption row) and never logged in plaintext (implied by BRD §23's structured-logging discipline, though not stated specifically for Identity).
- On consuming `UserDataErased` (ADR-0016, updated to name Identity), Identity must render the account permanently unusable for future authentication (`password_hash` cleared, `mfa_credentials` row deleted, `federated_identities` rows deleted) and must revoke every outstanding session/refresh-token family for that user — an erased account must never remain reachable via a not-yet-expired token. This is a new invariant this redraft adds, closing the gap `architecture.md` flagged rather than leaving it open.

## 6. Open Questions / Flagged Ambiguities

**Closure pass.** Every item the prior draft carried as open has now been resolved one of three ways: (a) an existing ADR already answered it, (b) it is a single-service engineering default decided directly in this spec/edge-cases.md, or (c) — for exactly one item — it is a legitimate non-blocking hand-off to a later pipeline stage, kept below labeled as such. None of the resolved items are re-litigated here; each links to where its resolution now lives.

**Resolved by existing ADR / this spec's own FRs (no longer open):**

- *Token issuer identity* and *OAuth2 flow termination point* — resolved by BRD §24.1/§24.2 directly; see §2.
- *Retry/DLQ policy for `UserRegistered`/`SessionCreated`* — resolved by **ADR-0007**; see §3, §5.
- *MFA mechanism* (prior #1, blocking) — resolved in §2: TOTP (RFC 6238), required for Admin/Support Agent, optional for Customer, N/A for Partner API.
- *Session TTL* (prior #2, blocking) — resolved in §4: access token 15 min; native refresh token 30-day sliding / 90-day absolute cap; federated refresh token 24-hour absolute cap, no sliding.
- *Refresh-token reuse/theft response* (prior #3, blocking) — resolved in §4, formalizing edge-cases.md's token-family reuse-detection Decision as a Domain Invariant.
- *No consumed events* (prior #4) — resolved in §2/§4/§5: the admin-lock/unlock path is a synchronous internal API call from Admin Service (per ADR-0010 (`docs/adr/0010-admin-service-scope-and-integration.md`)), not an event Identity consumes, so the empty Consumes list is confirmed correct by design, not an unexplained gap.
- *Availability tier* (prior #5) — resolved in §3: 99.99%, the tier matching Identity's platform-wide dependency shape.
- *No password-reset/logout/MFA-challenge endpoint* (prior #6) — resolved in §2/§5: all three now exist as stated FRs and API Surface rows.
- *External IdP role/group mapping* (prior #7, blocking) — resolved in §2: fixed `Customer`-only default for social login; config-driven group→role mapping, fail-closed, for enterprise federation; `Admin` never auto-granted.
- *Federated Login With No Matching Kart Account* (the edge-cases.md escalation tied to prior #7) — resolved in §2: JIT provisioning for both federation paths, governed by the role-mapping decision above so no ungoverned privilege can result.

**Carried forward — non-blocking hand-off to a later pipeline stage:**

1. **Federation session termination / Single Logout (SLO).** BRD §24.2 describes federation as a one-way exchange (external assertion → Kart JWT) at login time only; nothing states what happens if the enterprise IdP's own session ends or a SAML SLO request arrives. This is **not blocking**: §4's 24-hour absolute federated-refresh-token cap already bounds the worst-case exposure window today (a defensible interim mitigation, not the real fix). Real-time SLO/deprovisioning propagation (SCIM push, SLO endpoint, or polling) genuinely depends on which specific enterprise IdP capabilities are chosen, which is Architecture Agent-level integration design, not a BRD-derivable requirement-spec decision — carried forward deliberately, the same way a final numeric latency target would be.

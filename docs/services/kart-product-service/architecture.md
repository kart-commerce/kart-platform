# Architecture: kart-product-service

## Boundary Rationale

`kart-product-service` is the bounded context that answers "what can be sold, at what price, and with what attributes" — it owns product identity, the variant/SKU model, catalog attributes, and the canonical base/list price (BRD §2.1 item 3; requirement-spec §1, §4). The ubiquitous-language glossary (`docs/ddd/ubiquitous-language.md`) already records "SKU / Product Price" as owned here and only *referenced* by `kart-offer-service` via an ACL — this doc formalizes the owning side of that same boundary.

Two structural facts drive its architecture more than anything else:

1. **It is the platform's highest-fan-out catalog publisher.** `ProductCreated` and `ProductPriceChanged` each reach four downstream consumer groups (Search, Recommendation/Wishlist, Offer/Pricing, Analytics — requirement-spec §2, §5), so publish reliability (Outbox + per-consumer-group queues) is a first-class architectural concern here even though Product itself has no synchronous outbound dependency on any of those four.
2. **It is explicitly *not* on the Order Saga's critical path.** BRD §5.5's saga diagram only reaches Product from the Gateway, never from a saga step, and NFR §3 confirms the read path sits in the 99.9% secondary-availability tier, not the 99.99% order-path tier. This mirrors `kart-offer-service`'s own boundary posture (see its `architecture.md`) — read-heavy and checkout-adjacent, never a saga participant.

CQRS split: PostgreSQL is the strongly-consistent write side (product/variant/SKU rows, unique-constraint-enforced); MongoDB (`product_read_model`, sharded by `category.id`) is the eventually-consistent read side, rebuildable from the write side plus replayed events (requirement-spec §4, BRD §6.1/§6.2/§7) — the same platform-wide CQRS safety property `kart-offer-service` and every other CQRS-split service already commit to.

The variant/attribute data model itself (hybrid EAV/JSONB — requirement-spec §2) and the exact aggregate boundary between `Product` and `Variant` are **not** re-litigated here; per requirement-spec §2/§6 and `design-decisions.md`'s "Not Decided Here," that aggregate-boundary call is explicitly the DDD Agent's job. This document only fixes *where Product's boundary sits relative to other services*, not its internal aggregate shape.

### Category — no confirmed dependency edge today (resolved this stage)

The platform's "Recommended Build Order" (`docs/services/README.md`) lists Product as depending on Category, and BRD §6.2's `product_read_model` example embeds a denormalized `category: { id, name }` object. It would be tempting to read this as a synchronous validation call (Product → Category, confirming `category_id` exists and fetching its current name at write time). That call is **not** added here, because `kart-category-service/requirement-spec.md` (already approved, reviewed to the same bar as this doc) states this plainly and explicitly: *"Neither Product's nor Search's own approved `requirement-spec.md`/`edge-cases.md` names itself as a consumer of `CategoryUpdated` today... if Product or Search later need live sync... that is a new finding for their own docs"* — i.e., Category's own docs already establish that **no live sync mechanism, sync or async, connects the two services today**, and explicitly reserve the decision to add one for whichever side needs it later.

Inventing a synchronous Product→Category call here would silently resolve a question Category's own approved spec deliberately left open for Product's side to answer, the same anti-pattern this pipeline's ADRs (e.g., ADR-0008 §"a new, separate finding for that service's own docs") consistently avoid. Resolved reading, consistent with that boundary:

- The Recommended Build Order's "depends on Category" is a **sequencing** statement (category IDs must exist before a product can reference one), not a **runtime integration** statement.
- Product's write API accepts a caller-supplied `category_id` and `category` name (the caller — Admin's catalog-management UI or the Partner bulk-upload feed — is expected to have already resolved a valid category through Category's own read surface before submitting); Product does not independently validate or refresh it against Category.
- **Known, accepted gap, carried forward, not silently dropped:** if a category is later renamed via Category's own `CategoryUpdated`-emitting write path, Product's denormalized `category.name` copy will not update until a future revision of *this* service's docs adds `CategoryUpdated` consumption — exactly the staleness window Category's own `edge-cases.md` ("Taxonomy write vs. Search facet staleness") already budgets for on the Search side. No new decision is invented here beyond stating this explicitly for Product's side too.

## Rating-Projection Staleness Bound (resolves requirement-spec §6, item 7)

Requirement-spec §6 carried forward one non-blocking numeric question: how quickly must `product_read_model.ratingSummary` reflect a new `ReviewSubmitted` event? Resolved here using the platform's own already-stated general CQRS default rather than inventing a Product-specific number: BRD §7 states the platform's Eventual Consistency Window is **"typically sub-second"** for every CQRS-split read model. Nothing in requirement-spec, edge-cases.md, or the BRD names a reason the rating projection should diverge from that default (it uses the same Outbox → RabbitMQ → MongoDB-projector pipeline as every other Product projection) — so the sub-second platform default applies unchanged. If a future load-test under the DDD/Database Design Agent stage finds this SLA unachievable at 100M-SKU scale for this specific projector, that is a capacity finding for that stage, not a reason to re-derive the number here.

## Distributed-Monolith Risk

- **No risk on the read path.** `GET /products/{id}` never calls out synchronously to any other service — it reads only from Product's own MongoDB read model. Product's read-path availability is fully self-contained.
- **Narrow, write-path-only coupling: none added.** Because the Category dependency above is resolved as "no live call" rather than a synchronous validation call, Product's write path (`POST`/`PUT`/`PATCH /products/{id}`) also has **zero synchronous outbound dependency** on any other service. Admin and the Partner API are inbound callers *of* Product, not dependencies *of* Product.
- **High fan-out on publish is a known, already-mitigated risk, not a new one.** `ProductCreated`/`ProductPriceChanged` reaching four consumer groups each is exactly the failure mode edge-cases.md's "High fan-out on every publish risks one slow consumer blocking others" already addresses via per-consumer-group queues/DLQ under `ecommerce.events` (BRD §8) — cited here for completeness of the dependency graph, not re-decided.
- **Genuine residual risk to flag: Admin/Partner write availability is coupled to Product's own write-path uptime**, which is unavoidable (they call Product's own API, per ADR-0010's stated pattern — there is no alternative path that wouldn't create a second writer to Product's tables, which the platform forbids everywhere). This is an acceptable, standard "caller depends on the service whose data it's mutating" coupling, not a distributed-monolith anti-pattern — it does not chain any further (Product's write path does not, in turn, depend synchronously on anything else, per the point above).
- **Overall assessment:** Product is architecturally clean relative to the rest of the platform — a pure event source for four other bounded contexts, with no synchronous dependency of its own on any of them, and (per the Category resolution above) no hidden synchronous dependency on Category either.

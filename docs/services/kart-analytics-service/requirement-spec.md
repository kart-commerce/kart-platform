# Requirement Spec: kart-analytics-service

## 1. Scope

Covers a single BRD service: **Analytics Service** (BRD §2.1 item 19, "Event ingestion, dashboards, funnels"). No merge — unlike Offer, there is no existing ADR bundling Analytics with another bounded context, and nothing in the BRD suggests one.

Analytics is unusual among the platform's services in that the BRD gives it no public write API and, per §5.4, no request-driven read API either ("(ingestion only)") — its entire external surface is consuming events and serving dashboards/funnels from the resulting warehouse. The BRD does not name a concrete technology for the dashboard/funnel query surface (API, BI tool, internal query layer). **Decision (D4b):** this is served as an **internal-only reporting/query layer over the warehouse** (e.g. an internal REST/GraphQL surface or a BI-tool connection), not exposed through the public API Gateway — consistent with this service having no public inbound endpoint at all per BRD §5.4. The concrete query technology choice is ordinary downstream design work for the Architecture/API Design Agents, not a blocking gap; it is carried forward like any other implementation-shape decision (see §5 and §6).

## 2. Functional Requirements

- Ingest domain events published across the platform's event bus into a data warehouse via Kafka topics (BRD §5.4: "Kafka topics → warehouse").
- Consume the **full fan-in of every published platform event** (BRD §5.4, as amended: "All events, full fan-in — resolved scope, see ADR-0004; §10 lists Analytics as a consumer on every row"). **Resolved:** the earlier §5.4-vs-§10 scope contradiction is closed by **ADR-0004** (Analytics Ingests Full Event Fan-In) — §2.2's forcing rule ("Analytics ingests every domain event") is authoritative, §10's Event Catalog has since been corrected to list Analytics as a consumer on every row (including the events added later by ADR-0007/ADR-0008: `OrderCompensationTriggered`, `InventoryReleased`, `InventoryReplenished`, `RefundIssued`, `CartCheckedOut`, `WishlistPriceAlertTriggered`, `AdminActionPerformed`, `UserRegistered`, `SessionCreated`, `CategoryUpdated`, `PriceQuoteIssued`, `PromotionActivated`, `UserProfileUpdated`), and ADR-0004 itself states "every future new event automatically has Analytics as a consumer by default." This is no longer an open question — see §6.
- Produce dashboards and funnels (BRD §2.1 item 19) from ingested event data. **Decision (D4a):** an initial concrete set is enumerated in §6 (Resolved Decisions), grounded directly in BRD-named events; the BRD does not enumerate these itself, so this spec makes the call rather than leaving the phrase "dashboards, funnels" undefined.
- Support reprocessing (replay) of historical events from the durable event log — the BRD's own justification for putting Analytics on Kafka is a concrete replay scenario: "Analytics needs to reprocess 30 days of `OrderCreated` after a bug fix" (BRD §14).
- Be the first consumer migrated in the RabbitMQ→Kafka strangler migration, ahead of Recommendation, "since it's the one hitting RabbitMQ's replay limitation" (BRD §15).
- Publish nothing back to the bus — the BRD lists no event published by Analytics anywhere in §5.4 or the §10 Event Catalog (verified against the current, ADR-0007/0008-corrected catalog: Analytics never appears in a Publisher column). Analytics is consumer-only.

## 4. Domain Invariants

- Every ingested event must carry (or be resolvable to) a schema version, and the ingestion pipeline must tolerate multiple schema versions of the same event type in flight at once — the direct consequence of BRD §2.2's "Analytics ingests every domain event → forces a durable event bus and schema versioning discipline." **Resolved (D2):** concrete scheme is a Confluent-compatible schema registry in `BACKWARD` compatibility mode; see §6.
- The event log Analytics reads from must be durable and replayable (not consume-and-discard) — inferred from §14's stated RabbitMQ limitation ("no native replay... queues are consumed and gone") being the explicit reason Analytics moved to Kafka.
- Reprocessing historical events (replay) must not corrupt or double-count already-computed dashboard/funnel aggregates — inferred from the combination of the §14 replay scenario and the global "at-least-once delivery + idempotent consumers" NFR (§3), which applies to Analytics as a consumer like any other. This is the domain invariant edge-cases.md's "Replay Correctness" edge case resolves (idempotent upserts on raw storage, aggregates always recomputed from raw storage, never incrementally mutated).
- Analytics must not be able to affect any other service's state or trigger side effects — it publishes no events (§10/§5.4 list none), so ingestion failures on Analytics' side must never block or slow the publishers whose events it fans in from. This is why Analytics' own consumer offset is only advanced after a successful warehouse write or a successful hand-off to `analytics.dlq` (D5, §6) — never left uncommitted in a way that would stall its consumer group and cause redelivery pressure on upstream brokers.

## 6. Resolved Decisions and Carried-Forward Items

Every item previously listed under "Open Questions / Flagged Ambiguities" has been closed by one of: citing an existing ADR, making a single-service engineering-default call directly in this spec, or (for two items) confirming no cross-cutting ADR is warranted and deciding directly. Nothing here blocks sign-off; the two items explicitly marked **non-blocking** at the end are ordinary downstream handoffs to later pipeline stages, the same pattern `kart-search-service` and `kart-recommendation-service` already use for their own unresolved numeric SLAs.

1. **("All events" vs. the Event Catalog) — RESOLVED via ADR-0004.** Analytics consumes every published platform event, full fan-in. BRD §2.2's forcing rule is authoritative; §10's Event Catalog has been corrected (ADR-0007, ADR-0008) to list Analytics on every row. No remaining ambiguity — see §2, §5.
2. **(Schema versioning/evolution policy) — DECIDED (D2), single-service engineering default.** Analytics adopts a Confluent-compatible schema registry (Avro payloads) in `BACKWARD` compatibility mode as the enforcement point: a publisher's schema change is rejected at publish time if it isn't backward-compatible with the prior schema, before it ever reaches Analytics. Each message's wire format carries the registry-assigned schema ID (the de facto version pointer — no separate hand-maintained version field is needed). The registry's subject metadata additionally exposes a human-readable `MAJOR.MINOR` label per logical event type: `MINOR` bumps are additive-only changes (new optional field, new enum value) compatible in place; `MAJOR` bumps are breaking changes that require a new topic/version namespace and a dual-publish transition window, mirroring the platform's own dual-publish precedent from the RabbitMQ→Kafka strangler migration (BRD §15). *Why:* this closes the exact residual gap edge-cases.md's "Schema Evolution" edge case had flagged as escalated (registry/serialization format, version-field convention) without inventing a new cross-cutting governance process — the registry *pattern* itself was already chosen in edge-cases.md; only the concrete format was open. *Trade-off:* Avro requires every publisher to adopt registry tooling and CI-time compatibility checks — real process coupling across independent service teams — but this coupling was already accepted in principle by BRD §2.2's own "forces... schema versioning discipline" language and by edge-cases.md's prior decision; this only makes it concrete instead of hand-wavy. This is a single-service (Analytics-side ingestion contract) decision, not a new ADR: it doesn't reopen any BRD self-contradiction and Analytics is the sole consumer setting its own ingestion tooling requirements, exactly as the edge-case-analyzer-agent rubric already treats registry choice as an Analytics-local engineering default rather than a platform governance question.
3. **(Warehouse/topic retention window) — DECIDED (D3), single-service engineering default.** Kafka topic retention for Analytics' own topics (`kart.analytics.*`, per `docs/standards/kart-conventions.md`'s naming convention) is set to **30 days**, matching BRD §14's own stated replay scenario exactly ("reprocess 30 days of `OrderCreated`") — no need to over-provision broker storage beyond the business scenario the BRD itself names. The warehouse's raw-event layer retains ingested events **indefinitely** (append-only, partitioned by ingestion date, tiered hot/cold storage as a later cost-optimization detail): this is not an arbitrary extra choice, it is required by edge-cases.md's already-adopted "Replay Correctness" design, which recomputes all aggregates from raw storage rather than incrementally — that design breaks silently if raw storage doesn't outlive the 30-day Kafka window. *Trade-off:* unbounded warehouse storage growth over time, mitigated by storage tiering (an implementation detail, not a policy question) rather than by deleting raw history.
4. **(Dashboards/funnels not enumerated) — DECIDED (D4a), single-service engineering default.** Initial concrete set, each grounded in BRD-named events (§10):
   - **Order conversion funnel:** `CartCheckedOut` → `OrderCreated` → `OrderConfirmed` → `PaymentCompleted` → `OrderDelivered`, with drop-off rate per stage. Directly grounded in BRD §10's own annotation of `CartCheckedOut` as "(funnel/conversion tracking)."
   - **Revenue dashboard:** aggregated from `OrderCreated`/`PaymentCompleted` payload totals over time, by SKU/category.
   - **Fulfillment performance dashboard:** time-to-ship/time-to-deliver, from `ShipmentDispatched` → `DeliveryStatusUpdated` → `OrderDelivered`.
   - **Inventory movement dashboard:** stock reservation/release/replenishment trends from `InventoryReserved`, `InventoryReservationFailed`, `InventoryReleased`, `InventoryReplenished`.
   - **Catalog/pricing dashboard:** `ProductCreated`, `ProductPriceChanged`, `CategoryUpdated` volume and change frequency.
   - **Promotions/coupon effectiveness dashboard:** `CouponRedeemed`, `PriceQuoteIssued`, `PromotionActivated` — redemption rate and attributable order volume.
   - **User growth & engagement dashboard:** `UserRegistered`, `SessionCreated`, `UserAccountUpdated`/`UserProfileUpdated` — signups, logins, profile-change volume over time.
   - **Reviews & ratings dashboard:** `ReviewSubmitted` volume and rating distribution.
   - **Admin audit/compliance dashboard:** `AdminActionPerformed` trail, for back-office accountability.
   - **Notification delivery dashboard:** `NotificationSent`, `WishlistPriceAlertTriggered` — delivery-status volume by channel.
   This is a starting catalogue for the DDD/Architecture Agents to model read schemas against, not a final BI spec.
5. **(Analytics' own consumer-side failure handling) — DECIDED (D5), single-service engineering default.** A warehouse write failure gets 3x exponential-backoff retry, then the raw event is parked in a dedicated `analytics.dlq` (following the platform's existing per-domain DLQ naming convention, e.g. `order.dlq`/`catalog.dlq`) rather than dropped. A scheduled reprocessor drains `analytics.dlq` using the same replay tooling BRD §14 already implies for the 30-day reprocessing scenario. The consumer's offset is only committed after a successful write **or** a successful DLQ hand-off — never left uncommitted — so a stuck event cannot stall the consumer group or create redelivery pressure back on upstream publishers (Domain Invariant, §4). *Why 3x, not 5x or 1x:* a warehouse write failure isn't money-critical the way `PaymentCompleted` is (nothing blocks on it synchronously), but it isn't pure fire-and-forget audit either since every downstream dashboard in D4a depends on it — 3x matches the platform's existing "standard business event" tier (e.g. `OrderCreated`, `ProductCreated`) rather than either extreme.
6. **(Availability/latency SLA tier) — Availability DECIDED (D6a, see §3); latency CARRIED FORWARD, non-blocking.** Availability is 99.9% (secondary tier), per the same reasoning already used by `kart-search-service`/`kart-recommendation-service`. Latency has no numeric target in the BRD and none is invented here: Analytics has no public request/response API for the read-path P95/P99 NFR to bind to, and neither the ingestion-lag budget (event published → visible in the warehouse) nor the dashboard-query latency budget (once D4b's query layer is chosen) is stated anywhere in the BRD. **This is carried forward to the Architecture Agent to propose concrete numbers**, with human sign-off before it's load-bearing — the same non-blocking handoff pattern `kart-search-service` uses for its own consistency-lag SLA and `kart-recommendation-service` uses for its staleness bound. It does not block this spec's approval because no BRD text asserts a number that this spec would otherwise be silently guessing at.

### Payload Resolution: Deliberately Not Expanded Beyond `{trackingId, status}`

requirement-spec §2 explicitly hands this stage the call on whether to grow this payload beyond the BRD's fixed `trackingId, status` ("expanding a payload two other services already consume is the Event Design Agent's call, not this spec's to make unilaterally"). **Resolved: no expansion.** Neither `eta` nor a timestamp field is added:

- **`eta` stays out**, consistent with requirement-spec's own reasoning already applied to itself: ETA is exposed only via `GET /tracking/{id}`'s response body (`api-contract.yaml`'s `TrackingResponse`), never this event's payload, because it changes independently of `status` (recomputed on every accepted update, edge-cases.md "ETA Going Stale Mid-Transit") and none of Notification/Analytics/Order's own documented needs require it — Notification composes its own message copy from `status` alone, Analytics logs `status` transitions, and Order filters on `status == Delivered` only (`kart-order-service/requirement-spec.md` §2). Adding it would be carrying a field no consumer asks for.
- **No `occurredAt`/timestamp field is added either**, unlike `CategoryUpdated`'s payload growth (`kart-category-service/event-contract.md`), because that expansion was *structurally required* to make an already-locked edge-case decision implementable (a coarse subtree-move event needs a path to describe). Nothing here is structurally blocked by the BRD's original two fields: `ddd-model.md`'s `TrackingStatusHistory`/`WebhookDedupEntry` aggregates already carry `eventTimestamp`/`receivedAt` internally for ordering and dedup, and none of that internal bookkeeping needs to leak into this external payload for any consumer's stated purpose. Adding a field "because other services do" would be inventing scope requirement-spec never asked for.
- **This is additive-safe to revisit later** (`event-standards.md`'s versioning rule: additive fields are non-breaking) if a future consumer states an explicit need — not a permanent close-out, just not invented here without one.

### DLQ Correction: `tracking.shipment-dispatched.dlq`, Not the Shared `shipping.dlq` Label

`kart-shipping-service/requirement-spec.md` §3 and [ADR-0015](../../adr/0015-shipping-shipment-creation-failed-event.md) both name `shipping.dlq` as the retry/DLQ label for **two different events** Shipping publishes — `ShipmentDispatched` and `ShipmentCreationFailed` — consumed by up to four different services (Order, Notification, Delivery Tracking, Analytics for the former; Order, Notification, Analytics for the latter). That is exactly the shared/global-DLQ pattern `event-standards.md` forbids ("every consumer queue has its own DLQ — never a shared/global DLQ"), the identical pattern this repo's precedent already corrects on the consumer side (`kart-cart-service/event-contract.md`'s `InventoryReservationFailed` → `cart.inventory-reservation-failed.dlq`, overriding the BRD's shared `inventory.dlq`). `kart-shipping-service` has not yet authored its own `event-contract.md`, so there is no more-authoritative publisher-side source to defer to yet (unlike `kart-offer-service/event-contract.md`, which Analytics' own contract already treats as authoritative over BRD §10). Tracking's own consumption therefore gets its own scoped DLQ, `tracking.shipment-dispatched.dlq`, independent of whatever Order's/Notification's own future event contracts eventually name for their own consumption of the same event. The retry **count** (3x) is unchanged from Shipping's own stated tier — only the shared queue label is corrected, and only for this service's own queue.

- **Why 3x, not escalated to the `Payment*`/5x-paged tier:** a lost/DLQ'd `ShipmentDispatched` consumption means this service never materializes a `TrackingRecord` for that `trackingId` — the customer-visible consequence is `GET /tracking/{id}` continuing to return `202`/`PENDING` indefinitely for that shipment (edge-cases.md's already-resolved pending-race response), not a hard error, an oversell, or a financial double-charge. Standard business-event tier is the correct fit, the same reasoning `kart-category-service/event-contract.md` and `kart-offer-service/event-contract.md` already apply to their own non-money-moving consumed/published events.
- **Why not de-escalated to a 1x fire-and-forget tier:** unlike `AdminActionPerformed` (whose underlying fact stays durable in Admin's own PostgreSQL table regardless of event delivery), a permanently un-materialized `TrackingRecord` has **no independent durability fallback** — nothing else creates this aggregate. A permanently stuck "pending" state directly contradicts BRD §2.1's stated "real-time tracking" purpose (the same purpose the polling-fallback edge case and the non-regression invariant both already protect), so this needs a real retry budget before parking, not a single attempt.

### Correcting requirement-spec §6 item 4(b)'s Phrasing — `CarrierStatusIngested` Gets Its Own DLQ, Not a Literal Share of `tracking.dlq`

requirement-spec §6 item 4(b) states a durably-accepted webhook "gets the same at-least-once/`tracking.dlq` machinery as any other internally-published event." Read literally, that would mean `CarrierStatusIngested` shares the physical `tracking.dlq` queue with `DeliveryStatusUpdated`'s downstream consumers (Notification, Analytics, Order) — a genuine violation of "never shared," since those are entirely different consumers from this service's own internal processing pipeline. **Resolved here, not a contradiction to escalate:** requirement-spec's phrasing is read as meaning the same *type* of machinery (durable RabbitMQ queue + TTL-retry-ladder + DLQ + manual-replay tooling — `event-standards.md`), not the literal same physical queue. `CarrierStatusIngested` gets its own dedicated DLQ, `tracking.carrier-status-ingested.dlq`, satisfying requirement-spec's actual intent (this internal hop is not left with weaker guarantees than any other internally-published event) without violating the never-shared rule.

- **Why 3x, matching `DeliveryStatusUpdated`'s sibling tier:** a permanently-failed `CarrierStatusIngested` processing attempt produces the same class of customer-facing consequence as a lost `ShipmentDispatched` consumption or a lost `DeliveryStatusUpdated` publish — a stale/unprocessed status, not a financial or oversell risk. Standard business tier, not `Payment*`.
- **Why not escalated further despite sitting on this service's single busiest processing path:** a permanently DLQ'd `CarrierStatusIngested` message is bounded by the independent 6-hour polling-fallback backstop (edge-cases.md "Carrier Webhook Failure or Silence") — once a shipment's `lastUpdatedAt` goes stale regardless of *why* (carrier silence or this service's own processing failure), the per-carrier polling client re-fetches status directly from the carrier's API and re-establishes correctness without needing this specific message replayed. This safety net is why 3x-then-park is sufficient rather than a higher-retry/paged tier.
- **Not de-escalated to 1x** for the same reason `ShipmentDispatched`'s consumer-side tier isn't: `TrackingRecord`'s only source of state-update attempts, absent the (much slower, 6-hour-triggered) polling backstop, is this event.

### `UnmappedCarrierStatusFlagged`: Standard Tier Plus Mandatory Alerting on Final Failure

- **Why 2x, not 3x:** this event's underlying fact does not disappear if this event is ultimately lost — `ddd-model.md`/`database-design.md` both confirm the raw unmapped `carrierStatusCode` is persisted to `tracking_status_history_entries` (with `canonicalStatus: null`) independently of whether this event ever successfully publishes, matching the durable-fact-already-exists reasoning `kart-admin-service/event-contract.md` uses for `AdminActionPerformed`'s 1x tier. A slightly lighter retry budget than the two events above is appropriate since the *raw record* is never at risk.
- **Why not the same 1x fire-and-forget tier as `AdminActionPerformed`, and why this needs its own DLQ alerting where Admin's doesn't:** `AdminActionPerformed`'s sole consumer (Analytics) tolerates an arbitrarily delayed/lost copy — nothing time-bound depends on it arriving. `UnmappedCarrierStatusFlagged` is different: it is the **sole trigger** for the operational triage queue that edge-cases.md's "Unmapped Carrier Status" decision requires (1-business-day routine SLA, escalated to a same-day page if the shipment's ETA window passes while still unmapped) — nothing else scans `tracking_status_history_entries` for null-`canonicalStatus` rows on its own. If this event is silently lost after exhausting retries with no further signal, the unmapped status could sit untriaged indefinitely, recreating the exact "goes stale with no signal" failure mode BRD §2.1's real-time-tracking purpose (and this whole edge case) exists to prevent. **Resolved:** a final `tracking.unmapped-carrier-status-flagged.dlq` delivery failure itself raises an operational alert (not paged on-call — this is not `Payment*`/compliance tier — but not silent either), so a DLQ hit here is triaged the same way an unmapped status itself is, rather than treated as a passive parking lot nobody watches.
- **Not escalated to `Payment*`/5x/paged:** no money-moving or compliance risk exists here (contrast `UserDataErased`, `kart-analytics-service/event-contract.md`'s citation of ADR-0016) — the risk is a delayed operational response to a carrier-integration gap, already itself bounded by edge-cases.md's own SLA escalation ladder (routine → same-day page once ETA passes), not this event's retry tier.

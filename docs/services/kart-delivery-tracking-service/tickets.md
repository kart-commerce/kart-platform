## Epic: kart-delivery-tracking-service v1

Real-time delivery tracking read-side service (BRD §2.1 item 17) with **no PostgreSQL write side** (`architecture.md` Boundary Rationale) — the entire `TrackingRecord` projection is fed by two external triggers only: consuming `ShipmentDispatched` (aggregate creation) and per-carrier webhooks/scheduled polling (state transitions). This service also defines and owns its own carrier-webhook ingestion contract as a single-service API-surface addition (requirement-spec.md §5, was Open Question #1), including HMAC-verified durable buffering, content-hash dedup, a canonical status-vocabulary mapping layer, and a distinguishable `202`/`PENDING` response for the pre-materialization race window instead of a `404` (requirement-spec.md §5/§6 item 6; edge-cases.md).

## Notes for Sprint Planner Agent

- TRK-1 and TRK-3 have no dependencies — can start in parallel in sprint 1. TRK-1 is the aggregate-creation foundation every read/update path needs; TRK-3 is an independent ingress endpoint (durable-enqueue only, no aggregate mutation of its own).
- TRK-2 (`GetTrackingStatus`) only needs TRK-1's schema/collection to exist — its `202` branch works even with zero materialized records, but the task as a whole is sequenced after TRK-1 since both read against the same `tracking_records` document shape.
- TRK-4 is this service's highest-value and highest-risk slice (three-aggregate pipeline: dedup, ordinal-guard, canonical mapping, history append, two possible outbound publishes) and depends on both TRK-1 (target document must be creatable/existing) and TRK-3 (its trigger event). Sequence it right after those two land.
- TRK-5 and TRK-6 both depend on TRK-4 (they reuse its apply-status-update pipeline / react to its terminal-status transitions) but are independent of each other — can run in parallel once TRK-4 is done.
- TRK-7 depends only on TRK-4 (needs `UnmappedCarrierStatusFlagged`/null-`canonicalStatus` history rows to exist) — independent of TRK-5/TRK-6, can run in parallel with them.
- No circular dependencies in this graph; longest chain is TRK-3 → TRK-4 → TRK-5 (or TRK-6 or TRK-7), 3 deep.
- TRK-4's `UnmappedCarrierStatusFlagged` final-DLQ-failure alerting (event-contract.md) is implemented as part of TRK-4 itself (it's the same publish call's failure path), not a separate ticket — kept distinct from TRK-7, which is the *successful*-publish/ETA-window escalation path, a different trigger and a different failure mode over the same underlying "don't let an unmapped status go untriaged" goal.

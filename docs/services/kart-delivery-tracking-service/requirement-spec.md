# Requirement Spec: kart-delivery-tracking-service

## 1. Scope

Covers a single BRD service: **Delivery Tracking** (BRD §2.1 item 17, "Real-time tracking, ETA"). No merge with another BRD service applies here — like Search and Product, Delivery Tracking maps 1:1 onto one bounded context, so no ADR is required to justify scope.

The BRD's treatment of this service is minimal: one API endpoint, one database, one published event, and one condensed consumed-event description (§5.4). Most of the "how" is left unstated — this spec surfaces those gaps rather than filling them in.

## 2. Functional Requirements

- Serve tracking status/detail reads (`GET /tracking/{id}`, BRD §5.4). Response contract (including the pre-first-event state) is resolved in §5/§6 below.
- Consume `ShipmentDispatched` (BRD §10: published by Shipping, consumed by Order, Notification, **Tracking**; payload `orderId, carrier, trackingId`). **Resolved (was Open Question):** this event is the aggregate-creation trigger — consuming it is what creates the addressable tracking record for that `trackingId`. This is a single-service interpretation decision (it changes nothing about what Shipping publishes or who else consumes it), made directly here rather than escalated: the BRD's Event Catalog gives no other source of a `trackingId` (§10), and `GET /tracking/{id}` has nothing to resolve against otherwise. The DDD Agent still owns the exact aggregate/entity shape; this spec only settles the triggering relationship.
- Consume carrier status updates via "carrier webhook → internal event" (BRD §5.4's condensed Consumes column for this service). **Resolved (was Open Question #1, self-labeled blocking):** see §5 for the concrete minimal internal ingestion contract this service defines for itself (carrier id, tracking id, canonical-pending carrier status code, event timestamp, raw-payload passthrough), plus the per-carrier adapter/mapping and dedup/ordering rules already decided in edge-cases.md. The BRD still cannot name a fixed list of carriers or their native payload shapes (that is inherently third-party and out of the BRD's control), but the internal contract every per-carrier adapter must produce is no longer unspecified.
- Publish `DeliveryStatusUpdated` (`trackingId, status`) on a status change (BRD §10), consumed by Order (terminal `"delivered"` status value only, added per [ADR-0005](../../adr/0005-unify-order-terminal-event.md)), Notification, and Analytics. Published `status` values are always the canonical internal enum (edge-cases.md's vocabulary-normalization decision), never a raw carrier-native string.
- Provide real-time tracking and an ETA (BRD §2.1's stated primary responsibility). **Resolved (was Open Question #2):** ETA computation is a single-service engineering default, decided here — pass through the carrier-supplied ETA when the carrier's webhook/API provides one; fall back to a static per-carrier, per-service-level SLA table (e.g. "carrier X ground: dispatch + 5 business days") when it doesn't. Recomputed on every incoming status update, never on an independent fixed schedule (matches edge-cases.md's "ETA Going Stale Mid-Transit" decision). ETA is exposed only via `GET /tracking/{id}`'s response body, not added to `DeliveryStatusUpdated`'s BRD-fixed payload (`trackingId, status`) — expanding a payload two other services (Notification, Analytics) already consume is the Event Design Agent's call, not this spec's to make unilaterally.
- Persist tracking state in MongoDB (BRD §5.4) — no read/write split is stated for this service the way it is for Product/Search (§6.1); the BRD does not name a PostgreSQL write side for Delivery Tracking at all. Status history (edge-cases.md) is persisted in the same MongoDB store, keyed by `trackingId`.

## 4. Domain Invariants

- A tracking record is addressable by `trackingId`, and that identity must exist before any carrier status can be recorded against it — inferred from `ShipmentDispatched`'s payload (`orderId, carrier, trackingId`, BRD §10) being the only stated source of a `trackingId`, and from `GET /tracking/{id}` requiring that id to already resolve to something.
- The exposed delivery status must reflect the service's stated purpose of "real-time tracking" (BRD §2.1) rather than a stale or long-superseded one — the BRD gives no staleness bound for this, unlike Search's (still numerically unspecified) "seconds, not milliseconds" callout at §2.2 (see Open Questions).
- `DeliveryStatusUpdated` publication must be idempotent under RabbitMQ's at-least-once delivery (BRD NFR §3, Reliability row) — the same idempotency demand made of every other publisher in the Event Catalog (§10).
- Once a later delivery lifecycle stage has been recorded for a `trackingId`, the exposed current status should not regress to an earlier stage — inferred at a general level from "real-time tracking" being the stated purpose combined with status originating from an external, not BRD-controlled carrier webhook source. The BRD does not state this explicitly, nor does it define a status vocabulary or lifecycle ordering at all; the concrete lifecycle-ordinal mechanism is decided in edge-cases.md ("Out-of-Order Carrier Status Updates").
- Every status this service exposes (`GET /tracking/{id}`) or publishes (`DeliveryStatusUpdated`) must be a member of a single canonical internal status enum, never a raw carrier-native string — inferred from `DeliveryStatusUpdated` having two independent consumers (Notification, Analytics, BRD §10) that both need one stable vocabulary to key logic off of. Concrete mapping/fallback mechanics are decided in edge-cases.md ("Inconsistent Status Vocabulary Across Carriers", "Unmapped Carrier Status").
- A query for a `trackingId` this service has not yet materialized a record for (the pre-first-event race window) must never be indistinguishable from a hard permanent error — resolved in §5/§6 and edge-cases.md ("Tracking Query Before the First Status Event Has Arrived"): the response is a distinguishable pending state, not a `404`.

## 5. API Surface (from BRD, starting point only)

| Endpoint/Event | Direction | Notes |
|---|---|---|
| `GET /tracking/{id}` | Inbound API | BRD §5.4. **Response contract resolved (was Open Question):** returns `200` with `{trackingId, status, eta, lastUpdatedAt}` once a record exists; returns `202 Accepted` with `{trackingId, status: "PENDING", message}` when no record has been materialized yet for that id (see §6 item 6 / edge-cases.md — covers both the genuine pre-`ShipmentDispatched` race and a never-existing id, since this service cannot distinguish the two from its own data). Final wire-level contract (headers, error envelope, pagination if history is exposed) is still the API Design Agent's job — this spec only fixes the `200`-vs-`202` semantic, not the full schema. |
| `ShipmentDispatched` | Consumed | Published by Shipping (BRD §10); also consumed by Order and Notification. **Resolved:** this is the aggregate-creation trigger for this service's tracking record (see §2). |
| `POST /internal/webhooks/carriers/{carrierId}` | Consumed → translated to internal event | **New — resolved (was Open Question #1, self-labeled blocking).** This service's own concrete minimal ingestion contract, defined here as a single-service API-surface addition (not a BRD-stated endpoint, since the BRD only describes the concept "carrier webhook → internal event", BRD §5.4): a per-carrier adapter (one per integrated carrier) parses that carrier's native payload and normalizes it into this internal shape before further processing: `{ carrierId: string, trackingId: string, carrierStatusCode: string, eventTimestamp: ISO-8601 UTC, rawPayload: string (opaque passthrough of the original carrier body, stored for audit/debugging) }`. Auth: per-carrier shared-secret HMAC-SHA256 signature in an `X-Carrier-Signature` header, verified before any processing; requests failing verification are rejected `401` with no history write and no dedup-store entry. `carrierStatusCode` is the raw, carrier-native value — it is mapped to the canonical status enum (edge-cases.md) *after* ingestion, not before, so an unmapped code can still be captured for audit (edge-cases.md, "Unmapped Carrier Status"). What the BRD genuinely cannot define — the fixed list of carriers and each one's native payload shape — remains carrier-dependent and out of scope for this spec; only the internal contract every adapter must normalize into is fixed here. |
| `DeliveryStatusUpdated` | Published | Consumed by Order (terminal `"delivered"` status value only), Notification, Analytics (BRD §10; Order added per [ADR-0005](../../adr/0005-unify-order-terminal-event.md)). Payload stays `{trackingId, status}` per BRD §10 — `status` is always the canonical enum (never a raw carrier code, never `PENDING`/`UNKNOWN` — see edge-cases.md). |

Final wire-level contract (full schemas, versioning, error envelopes) is the API Design Agent's job; this spec fixes only the semantics called out above, which were genuine sign-off blockers.

## 6. Open Questions / Flagged Ambiguities

All items below were reviewed in this closure pass. None are cross-cutting (none change another service's publish/consume contract or resolve a BRD self-contradiction), so none required a new ADR — each is a single-service engineering default, decided directly below, or a legitimate non-blocking hand-off to a later pipeline stage.

1. **RESOLVED — Carrier webhook contract.** Was entirely unspecified (BRD §5.4 states only "carrier webhook → internal event"). Decision: see §5's new `POST /internal/webhooks/carriers/{carrierId}` row for the concrete minimal internal contract (carrier id, tracking id, raw status code, event timestamp, raw-payload passthrough) and its HMAC auth scheme. What remains genuinely carrier-dependent (each carrier's own native API/payload shape, the actual list of integrated carriers) is not a BRD gap this spec can close — that's inherent to depending on third parties — but it no longer blocks implementation, since the internal contract every per-carrier adapter must produce is now fixed.
2. **RESOLVED — ETA computation method.** Was unspecified (BRD §2.1 names ETA as core scope with no derivation method stated anywhere else). Decision: carrier-supplied ETA passed through when available, falling back to a static per-carrier/per-service-level SLA table; recomputed on every incoming status update (see §2, and edge-cases.md's "ETA Going Stale Mid-Transit", whose escalation this decision now closes). Treated as a single-service engineering default rather than escalated further: no BRD reading is contradicted by either fallback choice, and a distance/route or ML-based model would be substantial net-new scope the BRD never asks for.
3. **RESOLVED — Relationship to Shipping's `ShipmentDispatched`.** Was inferred, not stated. Decision: confirmed as the aggregate-creation trigger (see §2, Domain Invariants). This is a single-service reading of an event Tracking already consumes — it does not change Shipping's publish contract — so it's settled here rather than left for the DDD Agent to adjudicate; the DDD Agent still owns the resulting aggregate/entity design itself.
4. **RESOLVED — No retry/DLQ policy for carrier webhook ingestion.** Was unaddressed (the BRD's retry/DLQ topology, §8–§11, only covers the internal RabbitMQ bus; webhooks are third-party HTTP pushes, not bus messages). Decision: this service relies on three layered mechanisms, none of which require a broker-level retry: (a) most carrier webhook senders already retry on a non-`2xx` response, so returning `5xx` when this service fails to durably enqueue an inbound webhook lets the carrier's own redelivery do the work; (b) once a webhook is durably accepted, it is translated to an internal event and gets the same at-least-once/`tracking.dlq` machinery as any other internally-published event; (c) a scheduled polling fallback (edge-cases.md, "Carrier Webhook Failure or Silence") is the backstop for webhooks that never arrive at all — triggered once a shipment's last-updated timestamp exceeds a 6-hour staleness threshold (concrete default set here; see edge-cases.md for why 6 hours). This closes the gap without inventing a broker feature this integration point structurally cannot have.
5. **NON-BLOCKING / CARRIED FORWARD — Consistency/staleness numeric SLA.** Unlike Search, which at least names (without numbering) a "seconds, not milliseconds" bound at BRD §2.2, Delivery Tracking gets no equivalent callout despite "real-time" being its stated primary responsibility (§2.1). This is legitimately carried to the Architecture Agent, the same way a final numeric latency target is normally that stage's job — it depends on infrastructure choices (webhook fan-in capacity, polling cadence, Mongo write-projection latency) this spec has no basis to fix, and BRD §7 already establishes the platform's general pattern (eventual-consistency windows are "typically sub-second" but must be surfaced to the client, not hidden) as the baseline this stage should tighten into a number. **Not a blocker**: the qualitative behavior (surface a pending state rather than hide the gap) is already resolved at item 6 below and in §5.
6. **RESOLVED — Tracking query before the first status event has arrived.** Was left as a genuine unresolved escalation in edge-cases.md. Decision: `GET /tracking/{id}` returns `202 Accepted` with a distinguishable `{trackingId, status: "PENDING", message}` body, not a `404`, whenever no record has yet been materialized for that id — see §5 and edge-cases.md for the full Decision/Why/Trade-off. This mirrors BRD §7's own explicit CQRS precedent ("processing your order" state, surfaced rather than hidden) applied to this service's read side.

# Event Contract: kart-recommendation-service

Two transports, per `design-decisions.md`'s "Event Transport per Consumed Input" decision: `OrderDelivered`/`ProductCreated` on RabbitMQ (exchange `ecommerce.events`, routing key `service.entity.action`, per [kart-conventions.md](../../standards/kart-conventions.md)); the three clickstream events on Kafka (topic `kart.recommendation.clickstream-events`, per `kart-conventions.md`'s `kart.<service>.<entity>` naming, partitioned by `userId`). Every consumer queue/dead-letter destination is this service's own, never shared with another consumer of the same event (`event-standards.md`), matching every other confirmed consumer of `OrderDelivered`/`ProductCreated`'s own per-consumer-DLQ correction.

## Published Events

**None.** Recommendation is a pure consumer/query context (requirement-spec §2/§5; BRD §5.4 lists no outbound events; confirmed unchanged through `ddd-model.md`, which publishes no domain event from any of its four aggregates). This includes the three clickstream events below — they are formalized here as this service's own consumed-event contract entries, but their publisher is the Client Event Ingestion Gateway (infrastructure, `architecture.md`'s "Clickstream Publisher Resolution"), never this service.

## Retry-Tier Justification

- **`OrderDelivered`, 5x exponential, paged on-call:** mirrors `kart-order-service/event-contract.md`'s own published tier for this event exactly, the same posture `kart-review-service/event-contract.md` takes for its own identical consumption (matching the publisher's own stated tier rather than independently re-deriving a lower one) — Order elevated this event (along with its other lifecycle events) from BRD §10's original 3x/manual-replay tier because a stuck delivery signal blocks downstream correctness gates indefinitely (Review's verified-purchase gate; here, `ProductAffinity`'s purchase-signal completeness). Recommendation's own consequence of a lost delivery is a degraded-signal-completeness outcome (never an oversell/double-charge/financial one), but this service defers to the publisher's own criticality judgment for the retry/paging tier rather than asserting a lower one unilaterally, consistent with how it already treats `OrderDelivered`'s payload/publisher as Order's own decision to make.
- **`ProductCreated`, 3x, no paging:** matches every other confirmed consumer of this event platform-wide (`kart-product-service/event-contract.md`'s own table — Search, Analytics at the same tier). A lost delivery degrades new-product discoverability only, recoverable by the next `TrendingPool` recompute once real signal accumulates.
- **`ProductViewed`/`ProductClicked`/`SearchPerformed`, 3x (Kafka consumer-group retry), no paging:** no BRD-stated number exists for any clickstream input (requirement-spec §3's Retry/DLQ row named this as a genuine, previously-unfillable gap) — this stage's own engineering default, set to match the catalog-tier 3x convention rather than inventing an independent number, since a lost clickstream event has the same "degraded signal completeness, not a correctness gap" risk profile as `ProductCreated`'s loss: `UserBehaviorProfile`'s best-effort idempotency posture (`ddd-model.md`) already tolerates occasional loss/duplication as a low-stakes, Eventually-consistent behavioral signal. Kafka's own dead-letter mechanism is a dead-letter **topic** (`kart.recommendation.clickstream-events.dlq`), not a RabbitMQ queue — the same "every consumer gets its own DLQ, never shared" principle (`event-standards.md`) applied to Kafka's own idiom, since this is the only consumer group for this topic today (Analytics' own full fan-in, ADR-0004, reads the same topic independently and would define its own dead-letter topic for its own consumer group, not share this one).

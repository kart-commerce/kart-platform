## Consumed Events

**None.** Category consumes zero events. `architecture.md`'s Dependencies table has no async inbound edge for Category, and its own domain writes (create/rename/move/deprecate) are driven synchronously — either by the client-facing gateway's reads (`GET /categories`) or by Admin's own synchronous, RBAC-gated write-API calls (per **[ADR-0010](../../adr/0010-admin-service-scope-and-integration.md)**), never by reacting to another service's event. This is confirmed as intentional, not a gap: nothing in `requirement-spec.md` §2/§5 or `edge-cases.md` names an event this service needs to react to.

## Naming-Convention Compliance

`CategoryUpdated` = Entity `Category` (the one aggregate root this service owns — `database-design.md`) + past-tense verb `Updated` — compliant with the `<Entity><PastTenseVerb>` convention (`event-standards.md`). Routing key follows the `service.entity.action` convention (`kart-conventions.md`): `category.category.updated`. No collision found against any event name already registered in this repo's other `event-contract.md` files (`kart-offer-service`, `kart-admin-service`, `kart-cart-service`, `kart-analytics-service`).

## Payload Resolution: Growing Beyond `categoryId, name`

BRD §10, as amended by **[ADR-0008](../../adr/0008-event-catalog-completeness-round-2.md)**, fixes `CategoryUpdated`'s payload at the coarse `categoryId, name` — but `requirement-spec.md` §5/§6 explicitly carries the question of whether this payload needs to grow forward as *this stage's* job, and `database-design.md`'s `category_outbox_events.payload` column is deliberately typed `JSONB` for exactly this reason ("lets the Event Design Agent grow this ... without a schema migration"). This is resolved here, not left open further:

- **`parentId` and `path` are added now, not deferred.** This is not new invention — it fulfills a shape edge-cases.md's "Subtree re-parenting invalidates downstream category data" decision already locked in: "Publish one coarse event describing the moved subtree root/path and let consumers resolve the affected range themselves." A payload of only `categoryId, name` cannot describe a subtree root's *path*, so the decision edge-cases.md already made is not actually implementable without this field. `path` carries the same ordered ancestor-id array as `database-design.md`'s `ancestor_path` column (the ancestor chain from root to immediate parent), and `parentId` carries the immediate parent id — together they let a future consumer resolve "what changed under this path" without a recursive call back into Category's own read API.
- **`operation` is added** (`created | renamed | moved | deprecated`) so a consumer does not have to infer which taxonomy write triggered the event from field diffs alone — `requirement-spec.md` §2 names all four as triggering writes, but the BRD's original `categoryId, name` payload gives no way to distinguish them.
- **`occurredAt` is added** as the standard event-timestamp field, consistent with every other timestamped event payload already published on `ecommerce.events` (e.g. `OrderDelivered`'s `deliveredAt`, `WishlistPriceAlertTriggered`'s implicit ordering) — needed for Analytics' own event-time ordering, independent of broker delivery order.
- **This does not add a second consumer.** Analytics remains the sole confirmed consumer (ADR-0004, ADR-0008); growing the payload now — while the shape is already being fixed for the first time in this contract — avoids a second breaking-payload/version bump later if/when Product or Search confirm themselves as consumers per their own future requirement-spec revisions (ADR-0008's own stated scope boundary, `requirement-spec.md` §2/§6). `event-standards.md`'s versioning rule ("a breaking payload change requires a new version") makes getting this right now, while there is exactly one consumer to coordinate with, cheaper than a `v2` migration once a second consumer exists.
- **For a create/rename/deprecate**, `categoryId` is the affected node itself and `path`/`parentId` describe its own position, unchanged by the operation (except `parentId`, which is null-to-non-null only on initial creation of a root node, and is otherwise not the field that changes on rename/deprecate). **For a move**, `categoryId` is the moved subtree's root, `parentId`/`path` reflect its *new* position post-move — matching the "root/path" language edge-cases.md's decision already used.

## Retry-Tier Justification: Standard Catalog Tier, Not a Higher or Lower One

**Chosen: 3x retry, `category.category-updated.dlq`, no on-call paging on final failure** — the same retry count BRD §10 assigns via ADR-0008 ("same tier as sibling catalog events `ProductCreated`/`ProductPriceChanged`"), restated here against this service's own actual failure-mode risk per this stage's own responsibility, not copied forward unexamined:

- **Not the `Payment*`/`UserDataErased` tier (5x, paged on-call).** That tier exists for events that are the *sole* record of a state transition another service must act on to stay financially or compliance correct (a refund, a chargeback, a GDPR erasure). `CategoryUpdated` carries no such risk: Category's own PostgreSQL table (`database-design.md`) remains the sole, authoritative source of truth for the taxonomy regardless of whether this event is ever delivered — a lost/DLQ'd delivery costs Analytics a delayed or missing audit/reporting data point, never an oversell, a double-charge, or an incorrect customer-visible navigation state (Category's own reads are always current per ADR-0011's strong read-your-write target; only a downstream consumer's *own* copy could go stale, and none exists today). Escalating retry/paging here would treat a reporting-delivery risk as a money-moving risk, which it is not — the same reasoning `kart-offer-service/event-contract.md` already used to keep `CouponRedeemed` off the `Payment*` tier.
- **Not the fire-and-forget audit tier (1x, `AdminActionPerformed`/`NotificationSent`'s tier) either**, despite Category sharing Admin's "no synchronous downstream depends on this" profile. The distinguishing factor: `AdminActionPerformed`'s underlying fact is independently durable in Admin's own queryable `admin_actions` table with its own 5-year retention, so a DLQ'd delivery loses only a downstream copy of an already-durable record with a *dedicated* local audit mechanism. `CategoryUpdated`'s downstream value is different — Analytics' "Admin audit/compliance dashboard" and catalog-change reporting depend on this event as their only channel for *this* fact reaching Analytics at all (Category itself keeps no analytics/reporting projection of its own, per ADR-0011). A single retry attempt would be too thin a budget for a transient broker/network blip to cost a real, otherwise-unrecoverable-without-DLQ-replay reporting gap. 3x-then-park matches the platform's existing "standard business event" tier (`ProductCreated`, `ReviewSubmitted`), the same reasoning `kart-analytics-service/event-contract.md`'s own D5 section uses to place standard catalog/business events at 3x rather than either extreme.
- **Consumer assignment (Analytics only) is correct, not incomplete**, notwithstanding ADR-0004's platform-wide "Analytics is a consumer of every event by default" rule: no other service is named or safely inferable as a synchronous or asynchronous consumer of `CategoryUpdated` anywhere in the BRD, `requirement-spec.md`, or `architecture.md` today — see `architecture.md`'s "Deliberately absent" note on the Product/Search edge. If Product or Search later confirm themselves as consumers (per their own future requirement-spec revisions), that consumer's own retry/DLQ tier is that service's call to make against its own failure-mode risk, not a reason to change Category's own publish-side tier here.
- **DLQ name deliberately diverges from BRD §10/ADR-0008's simplified shared `catalog.dlq` label.** That label is shared across every sibling catalog event (`ProductCreated`, `ProductPriceChanged`, `CategoryUpdated`) published by two different services (Product, Category) — exactly the shared/global DLQ pattern `event-standards.md` forbids ("every consumer queue has its own DLQ — never a shared/global DLQ"), and the identical correction `kart-offer-service/event-contract.md` (`PriceQuoteIssued`/`PromotionActivated` overriding shared `coupon.dlq`), `kart-cart-service/event-contract.md` (`InventoryReservationFailed` overriding shared `inventory.dlq`), and `kart-admin-service/event-contract.md` (`AdminActionPerformed` overriding shared `admin.dlq`) already made for their own BRD-simplified DLQ names. `category.category-updated.dlq` is scoped to this event and this publisher only, independent of whatever `kart-product-service`'s own future event contract names for `ProductCreated`/`ProductPriceChanged`. The retry *count* (3x) is unchanged from BRD §10/ADR-0008 — only the shared queue label is corrected.

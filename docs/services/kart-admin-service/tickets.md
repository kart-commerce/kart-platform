## Epic: kart-admin-service v1

Back-office control-plane for four fixed operational categories — catalog management (Product + Category), coupon issuance (Offer's `Coupon`), user suspension (Identity's auth-identity), and inventory replenishment — plus a fifth `permission-management` meta-category governing the other four (BRD §2.1 item 20, §24.1; ADR-0010; `requirement-spec.md` §6 Decision item 1). Admin owns exactly two tables of its own (`admin_permission_grants`, `admin_actions`) and never a second copy of another service's domain data (Domain Invariant #3) — every business-category mutation is a synchronous proxy call to the owning service's own write API. Sixteen vertical slices below map 1:1 to `api-contract.yaml`'s sixteen paths; no path in the contract is left undecomposed and no ticket invents a route the contract doesn't already have.

## Flagged Gaps — Not Decomposed Into Tickets (would invent scope, not decompose it)

- **Bootstrap of the very first `permission-management` grant in a deployment is an ops/seed-script task, not an application-code ticket.** `requirement-spec.md` §6 Decision item 1 and `database-design.md`'s `granted_by = 'seed-script'` column comment both fix this as a one-time, out-of-band migration/seed script (avoiding an unbounded regress of "who grants the first grant") — ADM-1's own endpoint cannot be the mechanism that authorizes its own first caller. No ticket here; this is deployment/ops tooling outside this service's own vertical-slice decomposition, flagged for whichever later stage owns seed-data/migration scripts.
- **Final request/response shapes of the four owning services' own write APIs (Product, Category, Offer's coupon-create, Inventory's replenishment path) are each of those services' own API Design Agent's job**, not this service's — `api-contract.yaml`'s own header note says so explicitly, and ADM-4/ADM-5/ADM-7/ADM-8/ADM-11/ADM-15's request bodies are Admin's best-effort mirror of those shapes, flagged as assumptions to reconcile once each callee's own contract exists. No ticket added here to "define Product's/Category's/Offer's/Inventory's endpoint" — that scope belongs to those services' own ticket lists.
- **Wiring Admin's five new inbound edges into each owning service's own `architecture.md`** (`architecture.md`'s own "Follow-up flagged" note: `kart-offer-service/architecture.md` predates ADR-0010 and doesn't yet list the inbound-from-Admin edge; Product/Category/Identity/Inventory don't have an `architecture.md` yet at all) is each of those services' own documentation update, not a ticket in this list.
- **`admin_actions` partitioning/cold-storage tiering at scale** — `database-design.md`'s own "Partitioning/Sharding" section explicitly defers this ("flagged as a later cost detail... not decided here since no throughput evidence currently calls for it"). No ticket until that evidence exists.
- **A public endpoint for extending the five fixed categories, or for a sixth category** — `requirement-spec.md` §6 Decision item 1's own accepted trade-off is that a new admin action must be explicitly mapped to one of the five categories (or the table extended) before it is usable; no such extension is requested by any current design doc, so no ticket invents one.

## Notes for Sprint Planner Agent

- ADM-1 (`IssuePermissionGrant`) is the single foundation task: it lands both of Admin's own tables (`admin_permission_grants`, `admin_actions`) and the two shared mechanics every other ticket calls into — the live, uncached fine-grained grant check, and the Outbox-backed `AdminActionPerformed` publish. Every other ticket in this list depends on it directly; none depend on each other. Start it first, alone.
- ADM-2/ADM-3 (revoke/list grants) and the four business-category groups — Product (ADM-4/5/6), Category (ADM-7/8/9/10), Coupon (ADM-11/12), User (ADM-13/14), Inventory (ADM-15) — are all mutually independent once ADM-1 lands, and can be parallelized freely across engineers; ADM-16 (audit read) can start alongside them. No ticket in this set depends on any sibling business-category ticket, because the entity each one mutates (Product, Category, Coupon, user account, stock level) is owned and created by its respective owning service, not by any other Admin ticket — Domain Invariant #3 (`requirement-spec.md` §4) means there is no intra-Admin "create before update/deactivate" chain to sequence, unlike `kart-identity-service/tickets.md`'s IDN-10→IDN-11 or `kart-offer-service/tickets.md`'s OFF-5→OFF-6 pairs, where the same service owns both ends.
- ADM-13 (`LockUser`) and ADM-14 (`UnlockUser`) have no dependency on each other within this service's own ticket set for the same reason — the user's locked/unlocked state lives in Identity Service (that service's own `IDN-10`/`IDN-11` tickets), not in Admin. Sequencing recommendation only (not a hard dependency): build ADM-13 first so there is a locked user to exercise ADM-14 against in integration tests.
- No circular dependencies in this graph; every path is exactly 2 deep (ADM-1 → any of ADM-2 through ADM-16).
- Recommend building the four business-category groups only after `kart-product-service`, `kart-category-service`, `kart-offer-service`, `kart-identity-service`, and `kart-inventory-service` each have their own corresponding write endpoint stable and deployed — `docs/services/README.md`'s "Recommended Build Order" already places `kart-admin-service` (#17) after all five of those (#1, #2, #3, #5, #8) for exactly this reason. ADM-1/ADM-2/ADM-3/ADM-16 (Admin's own local-only permission/audit slices) have no such external dependency and can be built ahead of that order if capacity allows.

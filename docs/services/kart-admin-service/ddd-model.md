## Boundary Summary

`kart-admin-service` is a **thin orchestration/control-plane bounded context** (`architecture.md`'s Boundary Rationale), not a fifth owner of Product/Category/Coupon/user-identity/Inventory domain data (Domain Invariant #3, `requirement-spec.md` Â§4). It answers exactly one question: "which back-office action is *this* authenticated operator allowed to trigger, right now" â€” never "what is a Product/Category/Coupon/user account/stock level," which stays owned by its existing bounded context and is only ever mutated via that context's own write API (ADR-0010 Decision 2). Its bounded context is exactly two things:

1. A category-scoped, default-deny permission-grant ledger â€” "which of the five categories can this `Admin`-role holder actually exercise" (Domain Invariant #1) â€” layered on top of, and never replacing, the Identity-issued coarse `Admin` role claim.
2. An append-only, transactionally-atomic audit/outbox record of every back-office action actually performed, doubling as the sole source of the platform's `AdminActionPerformed` event (Domain Invariant #2, `requirement-spec.md` Â§6 Decision item 3).

Two aggregate roots below, each its own PostgreSQL table (`database-design.md`) and its own transaction boundary â€” confirmed by that document's already-built schema (two independent tables, no FK between `admin_permission_grants` and `admin_actions`) and by Cross-Aggregate Interaction below. Neither aggregate models a foreign entity's own domain state: `entityId` on `AdminAction` and `principalId` on `AdminPermissionGrant` are both opaque references, never a denormalized copy (Domain Invariant #3; database-design.md's own "no PII, no denormalized copy" note).

## Aggregate: AdminPermissionGrant

**Entity:** `AdminPermissionGrant` â€” identified by `GrantId` (value object; `grant_id` in `database-design.md`). Backed by `admin_permission_grants`.

**Value objects:**
- `GrantId` â€” generated UUID, the row's own identity, distinct from `PrincipalId`.
- `PrincipalId` â€” the Identity-issued principal id this grant is scoped to; a reference only, never modeled further (Domain Invariant #1 â€” Identity, not Admin, owns this identifier's own lifecycle).
- `PermissionCategory` â€” one of `catalog-management | coupon-issuance | user-suspension | inventory-replenishment | permission-management`, DB-enforced via `CHECK` (`database-design.md`). The first four are the closed, BRD/ADR-0010-fixed enumeration of back-office operation domains (`requirement-spec.md` Â§6 Decision item 2); `permission-management` is the fifth, meta-category value that governs granting/revoking the other four (Â§6 Decision item 1). This is a closed value set â€” extending it requires a new requirement-spec decision, not a code-level enum add (`requirement-spec.md` Â§6 Decision item 1's own accepted trade-off).

**Fields:** `grantId: GrantId`, `principalId: PrincipalId`, `category: PermissionCategory`, `grantedAt`, `grantedBy` (a `PrincipalId`, or the well-known literal `'seed-script'` for the one-time out-of-band bootstrap of the first `permission-management` grant), `revokedAt` (nullable), `revokedBy` (nullable), `version` (optimistic-concurrency token).

**Invariants:**
- At most one **live** (`revokedAt IS NULL`) grant per `(principalId, category)` â€” DB-enforced via `uq_admin_permission_grants_live` (`database-design.md`), and the single source of truth for "which categories can this principal currently exercise" (Domain Invariant #1). This is what every `/admin/*` handler reads fresh, uncached, on every request (`design-decisions.md`, "Caching Strategy for Fine-Grained Permission Grants") â€” never derived from, or cached alongside, the JWT-embedded coarse `Admin` claim.
- Revocation is the only mutation this aggregate ever undergoes after creation, and it is an `UPDATE` (`revokedAt`/`revokedBy` set, `version` incremented), never a `DELETE` â€” grant/revoke history is retained for the same audit reasons the Security NFR and `requirement-spec.md` Â§6 Decision item 3 require elsewhere in this service.
- A revoked grant takes effect on the *very next request* â€” there is no token-TTL-style exposure window for this layer, precisely because it is looked up live rather than embedded in a long-lived JWT (`edge-cases.md`, "Stale Admin Permission Outliving an Identity-Side Revocation"). This is the aggregate-level guarantee that edge case's Decision depends on.
- Mutating a grant that has already been concurrently revoked/re-issued by another operator is rejected via optimistic concurrency on `version` (`edge-cases.md`, "Concurrent Admins Editing the Same Back-Office Record"; `design-decisions.md`, "Concurrency Control for Back-Office Writes") â€” the losing writer must re-read and retry, never silently overwritten.
- Issuing or revoking a grant for `category = 'permission-management'` (i.e., mutating who else can manage grants) itself requires the acting principal to hold a live `permission-management` grant â€” enforced by the same live-grant check this aggregate's own read path always runs (Domain Invariant #1's formal `CanWrite` statement, `requirement-spec.md` Â§4), never a special-cased bypass. The sole exception is the one-time seed-script bootstrap (`grantedBy = 'seed-script'`), which exists precisely to avoid an unbounded regress of "who grants the first grant" (`requirement-spec.md` Â§6 Decision item 1).
- **CanRead/CanWrite/CanDelete ownership (BRD Â§24.1.2).** `CanWrite` (issue or revoke) requires a live `permission-management` grant for the acting principal (self-referential rule above). `CanRead` (`GET /admin/permission-grants`) is scoped by the RLS policy in `database-design.md`: a principal always sees its own rows, and additionally every row if it holds a live `permission-management` grant â€” never another principal's rows otherwise. `CanDelete` is never exposed â€” revocation is the only supported removal semantic, and it is an `UPDATE` (above).
- **Audit-actor invariant (BRD Â§24.3).** `grantedBy`/`grantedAt` (insert) and `revokedBy`/`revokedAt` (the one update) already satisfy this table's own `created_by`/`updated_by` equivalent in its own domain vocabulary â€” not a duplicated pair (`requirement-spec.md` Â§3's Security row cross-reference). Stamped by the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor, reading the acting principal's own id â€” never a value a request handler supplies directly, except the one well-known `'seed-script'` bootstrap literal.

**Domain events:** none published externally by this aggregate on its own â€” issuing or revoking a grant is itself one of the actions enumerated on `AdminAction` below (`action` values `grant.issue`/`grant.revoke`, per `database-design.md`'s own column comment), and it is that paired `AdminAction` row â€” not this aggregate directly â€” that drives the published `AdminActionPerformed` event. This mirrors the platform-wide "the write and its audit record are two aggregates, coordinated by the write path" pattern the Outbox pattern always implies, rather than making `AdminPermissionGrant` itself Outbox-aware.

## Aggregate: AdminAction

**Entity:** `AdminAction` â€” identified by `ActionId` (value object; `action_id` in `database-design.md`). Backed by `admin_actions`. Append-only: the only field ever mutated after insert is `publishedAt`/`publishedBy` (the Outbox poller's own hand-off), never any other field (`requirement-spec.md` Â§6 Decision item 3).

**Value objects:**
- `ActionId` â€” generated UUID, this row's own identity.
- `IdempotencyKey` â€” generated once per admin-action attempt, forwarded on every retry to the owning service's write API (`design-decisions.md`, "Idempotency Mechanism for Outbound Write Calls"); DB-enforced unique (`uq_admin_actions_idempotency_key`), so a retried attempt can never produce two local audit rows for one logical action.
- `PermissionCategory` â€” the same closed five-value set `AdminPermissionGrant` uses; every `AdminAction` maps to exactly one category (Domain Invariant #1).
- `ActionName` â€” the specific operation within the category (e.g. `product.create`, `product.deactivate`, `category.reorder`, `coupon.create`, `coupon.deactivate`, `user.lock`, `user.unlock`, `inventory.replenish`, `grant.issue`, `grant.revoke` â€” `database-design.md`'s own enumerated column comment). This is the `AdminActionPerformed` event's own `action` payload field.
- `EntityReference` â€” an opaque id of the mutated entity in the *owning* service (a Product id, Category node id, Coupon code, user id, Inventory SKU, or another `AdminPermissionGrant`'s `GrantId` for `permission-management` actions) â€” no foreign key, since the owning service, never Admin, is that entity's own source of truth (ADR-0010 Decision 2; Domain Invariant #3).

**Fields:** `actionId: ActionId`, `idempotencyKey: IdempotencyKey`, `adminId` (a `PrincipalId`), `category: PermissionCategory`, `action: ActionName`, `entityId: EntityReference`, `context` (nullable, richer audit detail beyond the three fields the published event actually carries), `performedAt`, `publishedAt` (nullable), `publishedBy` (nullable, set only alongside `publishedAt`).

**Invariants:**
- A row is written exactly once, in the same local PostgreSQL transaction as the outcome of the admin action it records, and **only after** the synchronous outbound call to the owning service (Product, Category, Offer, Identity, or Inventory) has already succeeded (Domain Invariant #2) â€” never before, and never speculatively for an action that did not actually complete downstream. This is what makes the row's mere existence a true fact ("this action happened"), not merely an attempt.
- Append-only at the application layer: no `UPDATE`/`DELETE` is ever exposed on `adminId`/`category`/`action`/`entityId`/`context`/`performedAt` once written; the sole exception is the Outbox poller setting `publishedAt`/`publishedBy` after a successful publish (`requirement-spec.md` Â§6 Decision item 3). This is Admin's authoritative, sole audit trail â€” no separate compliance log exists.
- Retained for 5 years (`requirement-spec.md` Â§6 Decision item 3, matching BRD Â§4.3's Order-write-storage precedent) â€” this aggregate is never purged on any shorter cycle.
- Publication of the corresponding `AdminActionPerformed` event is guaranteed-atomic with the row's own local commit via the Outbox pattern: a poller reads unpublished rows (`idx_admin_actions_unpublished`) and publishes them after commit, never before (`edge-cases.md`, "Admin Action Succeeds but `AdminActionPerformed` Is Lost"). A crash between the local commit and the poller's publish leaves the row durably present and re-scannable â€” the underlying fact is never lost even if the event delivery itself is retried or eventually dead-lettered (`event-contract.md`'s own "Retry-Tier Justification" section restates this explicitly).
- **CanRead/CanWrite/CanDelete ownership (BRD Â§24.1.2).** `CanWrite` (insert) happens exactly once per completed admin action, gated by the acting principal already holding a live `AdminPermissionGrant` for this row's own `category` (the write-time check that produced this very row â€” Domain Invariant #1's formal statement, `requirement-spec.md` Â§4). `CanRead` (`GET /admin/actions`) is deliberately **coarser** than the write-gating check: any caller holding the coarse Identity-issued `Admin` claim may read the full audit trail, with no fine-grained category check applied to this read-only view (`requirement-spec.md` Â§4 â€” restricting audit visibility by category would undermine the trail's own purpose). `CanDelete` is never exposed â€” this table is permanently append-only (above).
- **Audit-actor invariant (BRD Â§24.3).** `adminId`/`performedAt` (insert) already satisfy this table's own `created_by`/`created_at` equivalent. `publishedBy` â€” stamped with the well-known `system:admin-outbox-poller` literal whenever the poller sets `publishedAt` â€” closes the one genuine gap flagged in `requirement-spec.md` Â§3: this was previously the one mutation on either table with no actor column at all. Both stamps come from the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor, reading either the acting admin's own principal id or the well-known poller system id â€” never a value a request handler supplies directly.

**Domain events:**
- Published (external): `AdminActionPerformed` â€” payload `adminId`, `action`, `entityId` (`event-contract.md`), raised by the Outbox poller once this row's own local commit has landed, per the invariant above. This is this aggregate's sole published event, and this service's sole published event overall.
- Consumed: none â€” every `/admin/*` action, including the ones that produce this aggregate's own rows, is synchronously human/operator-initiated, never event-triggered (`requirement-spec.md` Â§2/Â§5, confirmed intentional, not a gap).

## Cross-Aggregate Interaction

`AdminPermissionGrant` and `AdminAction` are never written in the same local transaction, and there is no foreign key between `admin_permission_grants` and `admin_actions` (`database-design.md`) â€” confirming the transaction-boundary test that they are two aggregates, not one:

1. **Read-then-write, not read-then-transact:** every `/admin/*` handler first reads `AdminPermissionGrant` (a live grant for `(principalId, category)`) as an authorization *precondition* â€” a plain, uncached `SELECT`, never a lock held across the handler's own write. Only once that precondition passes does the handler perform its own downstream synchronous call and then, in a separate local transaction, insert the resulting `AdminAction` row. A grant revoked in between the read and the `AdminAction` insert is not retroactively invalidated for that one in-flight request (the same bounded, accepted window every uncached-read pattern has) â€” the *next* request re-reads fresh and is correctly blocked.
2. **`grant.issue`/`grant.revoke` actions are the one case where both aggregates are touched by a single logical operation**, and even here they are two separate writes, not one transaction: the `AdminPermissionGrant` row is inserted/updated first (the actual grant/revoke effect), and only once that write has committed does the handler insert the paired `AdminAction` row (`category = 'permission-management'`, `action = 'grant.issue'` or `'grant.revoke'`, `entityId` = the affected `GrantId`) recording that it happened â€” the same "outbound effect must succeed before the local audit record commits" ordering Domain Invariant #2 requires for every other category, with `AdminPermissionGrant`'s own write standing in for the "outbound call to an owning service" every other category makes to Product/Category/Offer/Identity/Inventory instead.
3. **No aggregate ever reads or joins across the other's table as part of its own invariant enforcement** â€” `AdminPermissionGrant`'s live-grant uniqueness (`uq_admin_permission_grants_live`) and `AdminAction`'s append-only/idempotency guarantees (`uq_admin_actions_idempotency_key`) are each fully self-contained within their own table.

## Modeling Decisions & Assumptions (resolved here, not escalated â€” engineering defaults, revisable)

1. **Two aggregates, not one (transaction-boundary test applied, refining ADR-0010's own shorthand).** ADR-0010's Consequences section describes "the admin action / permission grant" as if a single concept; `database-design.md` already built it as two independent tables with no FK and no joint transaction. This document formalizes that as two aggregate roots â€” `AdminPermissionGrant` and `AdminAction` â€” because merging them would force a single row to represent both "what a principal is currently allowed to do" (a slowly-changing, revocable grant) and "what was actually done and when" (an ever-growing, strictly-append-only log), which have materially different write patterns, retention rules (indefinite-until-revoked vs. fixed 5-year retention), and read-access rules (self-or-`permission-management`-scoped vs. any-coarse-`Admin`-claim-holder). This is a formalization of the existing, already-built schema's own shape, not a redesign of it â€” no migration or code change is implied.
2. **`PermissionCategory` is a shared value object, not a duplicated enum.** Both aggregates constrain their own `category` column to the identical five-value `CHECK` set (`database-design.md`); modeling it once, here, as a single value object avoids two independently-drifting definitions of "what a category is" across the two aggregates that both need it.
3. **No internal-only domain events proposed.** Unlike `kart-analytics-service` (`EventIngested`, etc.) or `kart-delivery-tracking-service` (`CarrierStatusIngested`), Admin's own write path has no async internal processing step for either aggregate to signal â€” every mutation is a direct, synchronous request/response handler action, and the one genuinely async step (the Outbox poller publishing `AdminActionPerformed`) is already a first-class, externally-published event, not an internal one needing its own name.
4. **Back-office category enumeration (Â§6 Decision item 2) and fine-grained permission model (Â§6 Decision item 1) are cited, not re-decided.** Both were already closed in `requirement-spec.md` Â§6 and ADR-0010 before this stage ran; this document's only job regarding them was to give them a concrete DDD shape (the `PermissionCategory` value object; the `AdminPermissionGrant`/`AdminAction` aggregate split), consistent with `database-design.md`'s and `event-contract.md`'s own already-built artifacts, not to reopen either decision.

# Edge Cases: kart-admin-service

## Edge Case: Privilege Escalation via Under-Specified Fine-Grained Permissions Within the Admin Role

- **What happens:** An actor already holding the Identity-issued `Admin` role claim performs a specific back-office action (e.g., issuing a coupon, suspending a user) that the platform did not intend that particular admin to be able to perform, because Admin Service's own action-level check treats every `Admin` claim holder as equally permitted to do everything.
- **Why it happens:** The coarse RBAC model is no longer open — BRD §24.1 fixes it directly (single `Admin` role, single issuer at Identity, two-tier enforcement) and Domain Invariant #1 states Admin Service consumes that claim rather than owning or maintaining any independent role/permission table. What Domain Invariant #1 and Open Question #1 leave genuinely unresolved is the layer Admin *does* control: the fine-grained, per-action permission mapping within the Admin role itself — whether the role is monolithic (any `Admin` claim holder may perform any `/admin/*` action) or internally scoped into per-action-category grants (catalog management vs. coupon issuance vs. user suspension, §24.1's own three example grants). An under-specified fine-grained layer is the direct condition that lets a monolithic-by-default implementation grant every admin every action, regardless of intended scope.
- **Solutions available (3):** Fine-grained, default-deny permission model scoped per action category, layered on top of (never replacing) the Identity-issued `Admin` claim — every `/admin/*` action requires an explicit fine-grained grant beyond the coarse claim; absence of a grant blocks the action · Monolithic Admin role with no internal scoping — any `Admin` claim holder can perform any `/admin/*` action, treating §24.1's three example grants as illustrative only, not enforced boundaries · Coarse category tiers only (e.g., a small number of named sub-roles like "catalog-admin"/"user-admin") without full per-action granularity
- **Decision:**
  - Chosen: Fine-grained, default-deny permission model scoped per action category, layered on top of the Identity-issued `Admin` claim
  - Why: RBAC is Admin Service's stated reason to exist (BRD §2.1 item 20), and both the FR (§2, third bullet) and Domain Invariant #1 require Admin to perform its own action-level check "before it executes" — a default-deny posture at that layer fails closed when a new admin action is added and its permission grant is forgotten, whereas a monolithic role fails open in the same scenario. This decision is squarely within Admin's own design space per Domain Invariant #1 (the layer it controls), not a re-litigation of the coarse RBAC model, which BRD §24.1 already fixes at Identity.
  - Trade-off accepted: Every new admin action requires an explicit fine-grained permission to be defined and assigned before it is usable. This decision picks the *shape* of the fine-grained check (default-deny, category-scoped) only — the concrete category boundaries (which actions share a category, how a category-scoped grant is represented and revoked) remain unresolved until Open Question #1 is answered; that enumeration is not picked here.

## Edge Case: Admin Action Succeeds but `AdminActionPerformed` Is Lost

- **What happens:** A back-office write commits, but the corresponding `AdminActionPerformed` event never publishes (crash between the DB write and the publish step, or a partial-write failure on a multi-record admin operation), leaving the action with no audit trail.
- **Why it happens:** Domain Invariant #2 states the action and its audit event should not silently diverge, but the requirement-spec does not confirm Admin uses the Outbox pattern (BRD §11) that exists platform-wide to solve exactly this dual-write problem — no numbered Open Question in the current requirement-spec covers this specific gap (the old #3 this previously cited was the now-resolved `AdminActionPerformed` Event-Catalog gap, a different issue; this Outbox-atomicity question remains genuinely open but uncatalogued) — without the pattern, a crash between the two writes is unrecoverable by construction.
- **Solutions available (3):** Outbox pattern (write the event into the same PostgreSQL transaction as the admin action; a poller publishes it after commit) · Synchronous publish-then-commit (publish first, commit the action only on broker ack) · Best-effort fire-and-forget publish with no transactional guarantee
- **Decision:**
  - Chosen: Outbox pattern, matching the platform-wide pattern (BRD §11)
  - Why: Admin actions are the platform's highest-privilege operations (Non-Functional Requirement: Security); an audit event that can silently vanish undermines the entire reason RBAC and audit exist, and the platform already has a standard mechanism for this exact failure mode rather than needing a one-off
  - Trade-off accepted: Introduces poller latency (event visible to consumers slightly after the action commits, not synchronously) versus the simpler but unsafe fire-and-forget option

## Edge Case: Concurrent Admins Editing the Same Back-Office Record

- **What happens:** Two admins (or one admin and an automated back-office process) read the same record, both act on it based on that stale read, and the second write silently overwrites the first's change.
- **Why it happens:** The Consistency NFR requires strong consistency on Admin's PostgreSQL write path because a stale read is a "security or operational-correctness defect, not a UX one" — but nothing in the requirement-spec constrains concurrent writers to the same record, and "back-office operations" (Open Question 2) are exactly the kind of infrequent, high-stakes, multi-operator writes where this race is plausible (e.g., two admins independently resolving the same flagged account).
- **Solutions available (3):** Optimistic concurrency control (version/row-timestamp check; second writer's update is rejected and must re-read) · Pessimistic row-level locking for the duration of an admin edit · No concurrency control (last-write-wins)
- **Decision:**
  - Chosen: Optimistic concurrency control (version column, conditional update)
  - Why: Admin actions are infrequent and human-paced relative to customer-facing traffic, so lock contention is not a throughput concern; a rejected stale write forces the second admin to see the first admin's change before proceeding, which matters more here than in a high-throughput path
  - Trade-off accepted: The losing admin's action is rejected and must be manually retried against fresh data, rather than being silently merged or serialized automatically

## Edge Case: Over-Broad `/admin/*` Surface as a Single Blast-Radius Point

- **What happens:** A single compromised admin credential, or a single vulnerability in the `/admin/*` surface, exposes actions across multiple unrelated back-office domains at once (e.g., user accounts, orders, inventory) rather than being contained to one.
- **Why it happens:** "Back-office operations" is unenumerated (Open Question 2) and the BRD gives Admin one flat API surface (`/admin/*`) rather than domain-scoped surfaces — combined with the unspecified fine-grained, per-action permission model within the Admin role (Open Question 1; the coarse role/issuer model itself is BRD-fixed at §24.1 and not in question here), a broad or coarse-grained surface makes any single compromise high-impact by construction, which is exactly the risk the Security NFR flags as Admin's defining concern.
- **Solutions available (3):** Domain-scoped sub-surfaces with independent permission grants per domain (e.g., a role scoped only to user-account actions cannot touch inventory actions) · Single flat `/admin/*` surface with fine-grained per-action permissions but no domain isolation · Separate deployable admin surfaces per domain (no shared blast radius at the service level)
- **Decision:**
  - Chosen: Single service with domain-scoped sub-surfaces and independent permission grants per domain
  - Why: Splitting into separate services per domain contradicts the BRD's own service boundary (Admin is one row, one service, at §2.1 item 20); domain-scoped permissions within one service contains blast radius without inventing a service split the BRD never states
  - Trade-off accepted: A vulnerability in the shared service process itself (as opposed to a permission-model gap) still exposes every domain, since isolation here is at the authorization layer, not the deployment boundary

## Edge Case: Stale Admin Permission Outliving an Identity-Side Revocation

- **What happens:** An admin's `Admin` role is revoked or downgraded at Identity (or, once Open Question #1 resolves, a specific fine-grained permission within the Admin role is revoked there), but a JWT already issued with the old scoped claim continues to be honored by Admin Service's own action-level check until either Identity's revocation-list mechanism catches it or the token naturally expires.
- **Why it happens:** Domain Invariant #1 is explicit that Admin Service does not own or maintain an independent role/permission table — issuance *and* revocation happen exclusively at Identity (BRD §24.1: "Identity Service is the single issuer of roles"). `kart-identity-service/edge-cases.md`'s "RBAC Role Change Outlives an Already-Minted JWT" edge case already establishes the mechanism for exactly this class of problem (a role change outliving a minted token): extending the Redis-backed revocation-list Gateway already consults for logout to also cover role-change events. What the corrected requirement-spec leaves open is narrower — only whether Admin's own fine-grained, per-action check (FR §2, third bullet; Domain Invariant #1) needs to independently consult that same revocation list, or whether the Gateway's existing per-request coarse validation is sufficient before a request ever reaches Admin's fine-grained layer. That can't be fully settled until Open Question #1 defines what a fine-grained permission grant (and its revocation) even looks like.
- **Solutions available (2):** Admin's fine-grained check trusts the Gateway's per-request coarse validation (including Identity's revocation-list check) as sufficient and performs no independent revocation lookup of its own — consistent with Domain Invariant #1's prohibition on a second, Admin-owned authorization store · Admin's fine-grained check additionally consults Identity's existing revocation list directly (not a new Admin-owned list) for defense-in-depth on sensitive actions, in case the fine-grained permission model that eventually answers Open Question #1 introduces a narrower revocation grain than the coarse role
- **Decision:**
  - Chosen: Admin's fine-grained check trusts the Gateway's per-request revalidation (including Identity's revocation-list check) and performs no independent lookup against any Admin-side state.
  - Why: Domain Invariant #1 explicitly rules out Admin maintaining or consulting a second, Admin-local authorization store as a source of truth. Building a duplicate revocation check inside Admin Service would recreate exactly the "second source of truth for who can do what" the corrected requirement-spec says the single-issuer model exists to prevent; `kart-identity-service/edge-cases.md` already documents and accepts the resulting bounded exposure window (up to the access-token TTL, narrowed by the revocation-list check) for this class of problem, and Admin inherits that same window rather than needing a separate one.
  - Trade-off accepted: If Open Question #1 eventually resolves to a fine-grained permission grain narrower than the coarse `Admin` role (e.g., one action-category permission revoked without revoking the whole role) and Identity's revocation-list mechanism is not extended to that same grain, Admin's fine-grained check would not independently detect that narrower staleness on its own. This is deferred to when Open Question #1 is answered and the Architecture Agent designs the concrete fine-grained model, rather than solved now by inventing an Admin-side store.

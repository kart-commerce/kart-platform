# Edge Cases: kart-admin-service

## Edge Case: Privilege Escalation via Under-Specified Fine-Grained Permissions Within the Admin Role

- **What happens:** An actor already holding the Identity-issued `Admin` role claim performs a specific back-office action (e.g., issuing a coupon, suspending a user) that the platform did not intend that particular admin to be able to perform, because Admin Service's own action-level check treats every `Admin` claim holder as equally permitted to do everything.
- **Why it happens:** The coarse RBAC model is no longer open — BRD §24.1 fixes it directly (single `Admin` role, single issuer at Identity, two-tier enforcement) and Domain Invariant #1 states Admin Service consumes that claim rather than owning or maintaining any independent *role* table. The layer Admin *does* control — the fine-grained, per-action permission mapping within the Admin role itself — was previously under-specified; left that way, a monolithic-by-default implementation would grant every admin every action, regardless of intended scope.
- **Solutions available (3):** Fine-grained, default-deny permission model scoped per action category, layered on top of (never replacing) the Identity-issued `Admin` claim — every `/admin/*` action requires an explicit fine-grained grant beyond the coarse claim; absence of a grant blocks the action · Monolithic Admin role with no internal scoping — any `Admin` claim holder can perform any `/admin/*` action, treating the four back-office categories (requirement-spec.md §6 Decision item 2) as illustrative only, not enforced boundaries · Coarse category tiers only (e.g., a small number of named sub-roles like "catalog-admin"/"user-admin") without full per-action granularity
- **Decision:**
  - Chosen: Fine-grained, default-deny permission model scoped per action category, layered on top of the Identity-issued `Admin` claim — **now fully specified in requirement-spec.md §6 Decision item 1:** Admin owns an `admin_permission_grants` table (`principalId`, `category`, `grantedAt`, `grantedBy`, `revokedAt`), scoped to the four categories fixed by ADR-0010 (`docs/adr/0010-admin-service-scope-and-integration.md`) — `catalog-management`, `coupon-issuance`, `user-suspension`, `inventory-replenishment` — plus a fifth meta-category, `permission-management`, for granting/revoking the other four (its own first grant seeded out-of-band, not self-service).
  - Why: RBAC is Admin Service's stated reason to exist (BRD §2.1 item 20), and both the FR (§2, third bullet) and Domain Invariant #1 require Admin to perform its own action-level check "before it executes" — a default-deny posture at that layer fails closed when a new admin action is added and its permission grant is forgotten, whereas a monolithic role fails open in the same scenario. This decision is squarely within Admin's own design space per Domain Invariant #1 (the layer it controls), not a re-litigation of the coarse RBAC model, which BRD §24.1 already fixes at Identity.
  - Trade-off accepted: Every new admin action requires an explicit fine-grained permission to be defined and assigned before it is usable. The category boundaries are now fixed (the four operational categories plus `permission-management`); the concrete grant-management CRUD surface (routes, payload shape) remains the API Design Agent's job, not this pass's — see requirement-spec.md §6 "Non-blocking, carried forward."

## Edge Case: Admin Action Succeeds but `AdminActionPerformed` Is Lost

- **What happens:** A back-office write commits, but the corresponding `AdminActionPerformed` event never publishes (crash between the DB write and the publish step, or a partial-write failure on a multi-record admin operation), leaving the action with no audit trail.
- **Why it happens:** Domain Invariant #2 states the action and its audit event should not silently diverge; without the Outbox pattern (BRD §11), which exists platform-wide to solve exactly this dual-write problem, a crash between the DB write and the publish step is unrecoverable by construction. This edge case's Decision below is the source resolution — restated as closed in requirement-spec.md §6 Decision item 7 and Domain Invariant #2.
- **Solutions available (3):** Outbox pattern (write the event into the same PostgreSQL transaction as the admin action; a poller publishes it after commit) · Synchronous publish-then-commit (publish first, commit the action only on broker ack) · Best-effort fire-and-forget publish with no transactional guarantee
- **Decision:**
  - Chosen: Outbox pattern, matching the platform-wide pattern (BRD §11)
  - Why: Admin actions are the platform's highest-privilege operations (Non-Functional Requirement: Security); an audit event that can silently vanish undermines the entire reason RBAC and audit exist, and the platform already has a standard mechanism for this exact failure mode rather than needing a one-off
  - Trade-off accepted: Introduces poller latency (event visible to consumers slightly after the action commits, not synchronously) versus the simpler but unsafe fire-and-forget option

## Edge Case: Concurrent Admins Editing the Same Back-Office Record

- **What happens:** Two admins (or one admin and an automated back-office process) read the same record, both act on it based on that stale read, and the second write silently overwrites the first's change.
- **Why it happens:** The Consistency NFR requires strong consistency on Admin's PostgreSQL write path because a stale read is a "security or operational-correctness defect, not a UX one" — but nothing in the requirement-spec constrains concurrent writers to the same record, and "back-office operations" (requirement-spec.md §6 Decision item 2's four categories — catalog management, coupon issuance, user suspension, inventory replenishment) are exactly the kind of infrequent, high-stakes, multi-operator writes where this race is plausible (e.g., two admins independently resolving the same flagged account).
- **Solutions available (3):** Optimistic concurrency control (version/row-timestamp check; second writer's update is rejected and must re-read) · Pessimistic row-level locking for the duration of an admin edit · No concurrency control (last-write-wins)
- **Decision:**
  - Chosen: Optimistic concurrency control (version column, conditional update)
  - Why: Admin actions are infrequent and human-paced relative to customer-facing traffic, so lock contention is not a throughput concern; a rejected stale write forces the second admin to see the first admin's change before proceeding, which matters more here than in a high-throughput path
  - Trade-off accepted: The losing admin's action is rejected and must be manually retried against fresh data, rather than being silently merged or serialized automatically

## Edge Case: Over-Broad `/admin/*` Surface as a Single Blast-Radius Point

- **What happens:** A single compromised admin credential, or a single vulnerability in the `/admin/*` surface, exposes actions across multiple unrelated back-office domains at once (e.g., user accounts, orders, inventory) rather than being contained to one.
- **Why it happens:** The BRD gives Admin one flat API surface (`/admin/*`) rather than domain-scoped surfaces, and "back-office operations" spans four unrelated domains (requirement-spec.md §6 Decision item 2: catalog management, coupon issuance, user suspension, inventory replenishment) — without a permission layer mapped to those domains, a broad or coarse-grained surface makes any single compromise high-impact by construction, which is exactly the risk the Security NFR flags as Admin's defining concern.
- **Solutions available (3):** Domain-scoped sub-surfaces with independent permission grants per domain (e.g., a grant scoped only to user-suspension actions cannot touch inventory-replenishment actions) · Single flat `/admin/*` surface with fine-grained per-action permissions but no domain isolation · Separate deployable admin surfaces per domain (no shared blast radius at the service level)
- **Decision:**
  - Chosen: Single service with domain-scoped sub-surfaces and independent permission grants per domain
  - Why: Splitting into separate services per domain contradicts the BRD's own service boundary (Admin is one row, one service, at §2.1 item 20); domain-scoped permissions within one service contains blast radius without inventing a service split the BRD never states. This is the same model requirement-spec.md §6 Decision item 1 fixes: the four `admin_permission_grants` categories (plus `permission-management`) *are* this edge case's "domain-scoped sub-surface" — one category per domain, each independently grantable/revocable, sharing the one flat `/admin/*` route surface without sharing authorization scope
  - Trade-off accepted: A vulnerability in the shared service process itself (as opposed to a permission-model gap) still exposes every domain, since isolation here is at the authorization layer, not the deployment boundary

## Edge Case: Stale Admin Permission Outliving an Identity-Side Revocation

- **What happens:** An admin's coarse `Admin` role is revoked or downgraded at Identity, but a JWT already issued with the old scoped claim continues to be honored by Admin Service's own action-level check until either Identity's revocation-list mechanism catches it or the token naturally expires.
- **Why it happens:** Domain Invariant #1 is explicit that Admin Service does not own or maintain an independent *role* table — coarse issuance *and* revocation happen exclusively at Identity (BRD §24.1: "Identity Service is the single issuer of roles"). `kart-identity-service/edge-cases.md`'s "RBAC Role Change Outlives an Already-Minted JWT" edge case already establishes the mechanism for this class of problem (a role change outliving a minted token): extending the Redis-backed revocation-list Gateway already consults for logout to also cover role-change events. This is distinct from — and must not be confused with — Admin's own fine-grained `admin_permission_grants` table (requirement-spec.md §6 Decision item 1), which Admin already owns and consults on every request regardless of this edge case's outcome.
- **Solutions available (2):** Admin's fine-grained check consults its own `admin_permission_grants` table fresh on every request (already required by Decision item 1) and additionally trusts the Gateway's per-request coarse validation (including Identity's revocation-list check) for the *coarse role* layer, performing no separate, duplicate revocation-list lookup of its own for that layer · Admin's fine-grained check additionally re-implements a direct call to Identity's revocation list itself (on top of its own grant-table lookup) for defense-in-depth on sensitive actions
- **Decision:**
  - Chosen: Admin's fine-grained check always consults its own `admin_permission_grants` table live, per request (this is Admin-owned state, required by Decision item 1, and is not the "second source of truth" Domain Invariant #1 prohibits — it is the fine-grained layer Invariant #1 explicitly reserves to Admin). For the *coarse* `Admin` role claim itself, Admin's check trusts the Gateway's per-request revalidation (including Identity's revocation-list check) and does not separately re-implement Identity's coarse revocation-list lookup inside Admin Service.
  - Why: This resolves the staleness problem for the layer Admin actually controls — because a fine-grained grant is looked up fresh from Admin's own table on every request rather than embedded in a long-lived JWT, revoking it (via the `permission-management` grant action) takes effect on the very next request, with no token-TTL exposure window at all. For the coarse role layer, which Admin does not own, re-implementing Identity's revocation-list check inside Admin Service would recreate exactly the "second source of truth for who can do what" the single-issuer model exists to prevent; `kart-identity-service/edge-cases.md` already documents and accepts the resulting bounded exposure window (up to the access-token TTL, narrowed by the revocation-list check), and Admin inherits that same window for the coarse layer only.
  - Trade-off accepted: A revoked coarse `Admin` role claim is still honored by Admin Service for up to the access-token TTL (narrowed by Identity's revocation-list check), the same bounded window every other RBAC-gated service on the platform accepts — Admin does not get a tighter guarantee here than the rest of the platform. The fine-grained grant layer has no equivalent exposure window, since it is checked live rather than embedded in the token.

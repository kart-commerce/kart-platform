# Requirement Spec: kart-admin-service

## 1. Scope

Covers the single BRD service **Admin Service** (BRD §2.1 item 20): "Back-office operations, RBAC." No merge; no ADR applies.

The BRD's treatment of Admin is the most condensed of any service: it gets one row in the core-modules table (§2.1), one row in the "remaining services" table (§5.4), and no dedicated deep-dive section (unlike Order/Inventory/Payment at §5.1–5.3). It is not part of the Order Saga (§5.5's boundary diagram does not route Admin at all). Two gaps flagged as total in the prior draft of this spec are now resolved directly by the BRD and by ADR, and no longer read as "unspecified":

1. The **coarse RBAC model** — role vocabulary, single issuer, two-tier (Gateway + service) enforcement — is fully specified at BRD §24.1 ("Cross-Cutting RBAC Model") and §24.2 ("SSO / Identity Federation"), not merely implied by the one-word "RBAC" label in §2.1. This redraft's root cause was reading Domain Invariant #1 and Open Question #4 (§6, prior draft) as if §24.1/§24.2 didn't exist; they do, and they answer both directly. See §4 and §6 below.
2. `AdminActionPerformed`, previously absent from the Event Catalog (§10) despite §5.4 naming it as published, now has a full catalog row — publisher, consumer, payload fields, retry tier, DLQ — added by **ADR-0007** (Event Catalog Completeness Pass). See §3, §5, and §6.

What remains genuinely unspecified is narrower than the prior draft's framing: the BRD gives Admin's coarse role and its top-level enforcement pattern, but not the fine-grained, per-action permission grants *within* the Admin role, nor a concrete enumeration of back-office operations. Those gaps are real and are carried forward in §6.

## 2. Functional Requirements

- Expose a back-office API surface gated by role-based access control (`/admin/*`, RBAC-gated, BRD §5.4). The gate gets its identity from the `Admin` role claim minted and issued exclusively by Identity Service (BRD §24.1's role table: "Admin | Full back-office operations | Catalog management, coupon issuance, user suspension — all `/admin/*` routes") — Admin Service does not define or issue this claim itself.
- Perform "back-office operations" (BRD §2.1 item 20) — the BRD names this as the service's core responsibility and §24.1's Admin row gives three illustrative example grants (catalog management, coupon issuance, user suspension), but no BRD section enumerates the full, concrete operation list (e.g., locking a user account, overriding an order, issuing a manual refund, adjusting inventory, moderating content). No complete operation-level functional requirement can be derived beyond those three named examples plus the general RBAC-gated surface; see Open Questions.
- Enforce fine-grained, action-level access control on every admin action before it executes, on top of the coarse `Admin` role claim already validated at the Gateway. BRD §24.1 states enforcement happens twice: "coarse-grained at the API Gateway (§18, reject before the request reaches a service) and fine-grained again at the owning service" — using "Support Agent can refund, but only up to $X" (a Payment Service example) as the pattern for what "fine-grained, service-owned" enforcement means. Admin Service is the owning service for its own `/admin/*` actions in exactly this sense: the Gateway can confirm the caller holds the `Admin` role, but only Admin Service itself can know, e.g., whether *this* admin is permitted to suspend *this specific* account. The BRD confirms this second check must exist and its general shape, but not the specific per-action permission grants within the Admin role — see Open Questions.
- Publish `AdminActionPerformed` after an admin action (BRD §5.4). This event now has a full row in the Event Catalog (§10, added by **ADR-0007**): publisher Admin, consumer Analytics (audit trail), payload `adminId`, `action`, `entityId`, retry 1x (fire-and-forget audit tier), DLQ `admin.dlq`. What the BRD still does not state is whether publication is guaranteed-atomic with the triggering write via the Outbox pattern (§11) — used elsewhere for exactly this dual-write problem, but not stated as applying to Admin specifically; see Open Questions.
- No consumed events are stated for this service — BRD §5.4 lists "—" under Admin's "Key Events Consumed" column. A back-office service reacting to nothing elsewhere in the platform (no `OrderCreated`, no `PaymentFailed`, no support-ticket-style trigger) is a notable BRD gap given the "back-office operations" label; see Open Questions.

## 4. Domain Invariants

- Admin Service does not own or maintain an independent role/permission table as a source of truth for "who can do what" — it consumes the `Admin` role claim issued exclusively by Identity Service and performs its own fine-grained, action-level enforcement on top of that claim before executing any admin action. This is stated directly at BRD §24.1: "RBAC is a platform-wide concern, not an Admin Service–local feature — every one of the 20 services trusts the same role claims rather than maintaining its own permission table. Identity Service is the single issuer of roles; every other service is a **consumer** of the claim." The two-tier enforcement Admin performs (Gateway coarse-check, then Admin's own fine-grained check) is the identical pattern §24.1 describes for Payment enforcing "Support Agent can refund, but only up to $X." The coarse model — four platform-wide roles (Customer, Support Agent, Admin, Partner API), one issuer, two-tier enforcement — is BRD-specified, not open; this replaces the prior draft's invariant, which incorrectly treated the model's shape (default-allow/deny, role- vs. permission-based, hierarchy) as wholly unstated. What remains genuinely unspecified is the fine-grained mapping of admin actions to permissions *within* the Admin role itself — e.g., whether every `/admin/*` action requires the same blanket `Admin` grant, or whether finer sub-permissions exist scoped to individual action categories (catalog management vs. user suspension vs. coupon issuance are §24.1's own three example grants) — see Open Questions.
- A state-changing admin action and its corresponding `AdminActionPerformed` record should not silently diverge (BRD §5.4 states the event is published, now with a full Event Catalog row per ADR-0007; BRD §11's Outbox pattern exists platform-wide precisely to prevent a business write and its event from disagreeing after a crash) — the BRD does not state that Admin uses the Outbox pattern, only that the general dual-write problem and its standard solution exist elsewhere on the platform; see Open Questions.
- Admin authorization ultimately traces back to a validated, signed identity minted exclusively by Identity Service — Admin Service never mints its own admin-scoped credentials or maintains a parallel session/role store. This is resolved directly by BRD §24.1 (single-issuer model: "Identity Service resolves a user/service-principal to a role set at token-mint time and embeds it as a scoped claim in the JWT") and §24.2, which states Admin/back-office SSO federation is terminated entirely at Identity Service against the org's enterprise IdP (SAML 2.0/OIDC) and that "no other service ever validates a SAML assertion or holds an external IdP client secret." This replaces the prior draft's invariant, which incorrectly treated this relationship as wholly unstated and unintegrated (it is stated, at §24.1/§24.2, even though no other BRD section names an explicit Identity↔Admin API call).

## 6. Open Questions / Flagged Ambiguities

**Resolved since the prior draft.** The prior draft carried two items (#4 in full, and the coarse-model portion of #1) as open questions that BRD §24.1, §24.2, and ADR-0007 already answer directly:

- *Relationship to Identity Service for admin authentication* (prior #4: "unconfirmed") is resolved by BRD §24.1 ("Identity Service is the single issuer of roles... every other service is a consumer of the claim") and §24.2 ("Federation is terminated entirely at Identity Service... No other service ever validates a SAML assertion or holds an external IdP client secret"). Admin trusts Identity-issued JWTs and never mints its own credentials or maintains a parallel role store. See §4's Domain Invariants and §2's Functional Requirements.
- *Coarse role/permission model* (the "default-allow or default-deny, role-based or permission-based, hierarchical or not" portion of prior #1) is resolved by BRD §24.1: four platform-wide roles (Customer, Support Agent, Admin, Partner API), single issuer, two-tier (Gateway + owning-service) enforcement. See §4's Domain Invariants.
- *`AdminActionPerformed` missing from the Event Catalog* (prior #3) is resolved by **ADR-0007** (Event Catalog Completeness Pass), which added a full row — publisher, consumer, payload, retry tier, DLQ — to §10. See §3's Reliability row and §5's API Surface table.

These are removed from the list below rather than carried forward as open. Remaining genuinely open items (renumbered):

1. **Fine-grained, per-action permission model within the Admin role is unspecified.** BRD §24.1 confirms the coarse model (single `Admin` role, single issuer, two-tier enforcement) and names three example grants (catalog management, coupon issuance, user suspension), but no BRD section states whether these three — or any other back-office action — require distinct sub-permissions, whether the Admin role is monolithic (any `Admin` claim holder can perform any `/admin/*` action) or internally scoped, or how a specific permission scoped to one action would be granted or revoked. This is narrower than the prior draft's "role/permission model unspecified" framing (the coarse model is now BRD-confirmed) but is still blocking for a concrete fine-grained enforcement design.
2. **"Back-office operations" is never fully enumerated.** BRD §24.1 names three example grants (catalog management, coupon issuance, user suspension), which is more than the prior draft had to work with, but no BRD section gives a complete operation list (account lock/unlock, order override, manual refund, inventory adjustment, content moderation override, etc.). Item 14 (Review Service) separately owns "moderation" for reviews, so at least that one candidate operation is confirmed to belong elsewhere — but nothing confirms the full, closed set of what Admin Service actually does beyond exposing an RBAC-gated surface and the three named examples.
3. **Audit completeness beyond `AdminActionPerformed` is unstated.** Given that back-office actions are the platform's highest-privilege operations, the BRD gives no requirement for an audit trail beyond the one event type — no mention of immutability, retention, or a compliance-grade log distinct from the general message bus. Whether `AdminActionPerformed` is meant to serve as that audit trail, or a separate one is assumed and simply unstated, is not resolved here.
4. **Availability and latency tiers unassigned.** BRD §3 splits Availability into "99.99% (order path)" and "99.9% (secondary)" and gives P95/P99 latency budgets framed around customer-facing read/write paths; Admin is absent from the §5.5 service boundary diagram entirely, which argues for treating it as secondary/internal, but the BRD never says so explicitly.
5. **Integration pattern with other services' data is unstated.** Admin has its own PostgreSQL database (§5.4), yet "back-office operations" plausibly implies acting on data owned by other services (e.g., a User account, an Order, Inventory stock). The BRD does not state whether Admin calls other services synchronously, only records actions in its own DB while other services independently react to `AdminActionPerformed`, or something else.
6. **Overlap with "admin tooling" mentioned in the messaging design (§8.1) is ambiguous.** The RabbitMQ DLQ row states DLQs are "inspected via admin tooling," which reads as generic ops/observability tooling rather than this Admin Service's `/admin/*` business surface — but the BRD never disambiguates the two, and a reader could conflate them. Not assumed to be the same thing here.
7. **Whether `AdminActionPerformed` publication is atomic with its triggering write (Outbox pattern) is unstated.** BRD §11 defines the Outbox pattern platform-wide precisely to prevent a business write and its event from silently diverging after a crash, but no BRD section states that Admin Service actually uses it for `AdminActionPerformed` — this is a real, tracked gap (Domain Invariant #2 depends on it) that the prior draft's Open Questions list never gave its own numbered item, despite Domain Invariant #2 pointing to "see Open Questions" for it. See `edge-cases.md`'s "Admin Action Succeeds but `AdminActionPerformed` Is Lost" for the failure mode and a proposed default.

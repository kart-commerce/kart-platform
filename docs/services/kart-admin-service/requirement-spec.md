# Requirement Spec: kart-admin-service

## 1. Scope

Covers the single BRD service **Admin Service** (BRD §2.1 item 20): "Back-office operations, RBAC." No merge; no ADR applies.

The BRD's treatment of Admin is the most condensed of any service: it gets one row in the core-modules table (§2.1), one row in the "remaining services" table (§5.4), and no dedicated deep-dive section (unlike Order/Inventory/Payment at §5.1–5.3). It is not part of the Order Saga (§5.5's boundary diagram does not route Admin at all). Two gaps flagged as total in the prior draft of this spec are now resolved directly by the BRD and by ADR, and no longer read as "unspecified":

1. The **coarse RBAC model** — role vocabulary, single issuer, two-tier (Gateway + service) enforcement — is fully specified at BRD §24.1 ("Cross-Cutting RBAC Model") and §24.2 ("SSO / Identity Federation"), not merely implied by the one-word "RBAC" label in §2.1. This redraft's root cause was reading Domain Invariant #1 and Open Question #4 (§6, prior draft) as if §24.1/§24.2 didn't exist; they do, and they answer both directly. See §4 and §6 below.
2. `AdminActionPerformed`, previously absent from the Event Catalog (§10) despite §5.4 naming it as published, now has a full catalog row — publisher, consumer, payload fields, retry tier, DLQ — added by **ADR-0007** (Event Catalog Completeness Pass). See §3, §5, and §6.

What the prior draft flagged as narrower-but-still-open — the fine-grained, per-action permission grants *within* the Admin role, and a concrete enumeration of back-office operations — is now closed this pass. The operation enumeration and Admin's integration pattern with other services' data were genuinely cross-cutting (they independently blocked open questions in `kart-category-service`, `kart-product-service`, and `kart-inventory-service`'s own requirement-specs) and are resolved via a new ADR, ADR-0010 (`docs/adr/0010-admin-service-scope-and-integration.md`). The fine-grained permission model itself is a single-service engineering default, decided directly in §6 Decision D1. See §6 for the full closure log — zero blocking items remain.

## 2. Functional Requirements

- Expose a back-office API surface gated by role-based access control (`/admin/*`, RBAC-gated, BRD §5.4). The gate gets its identity from the `Admin` role claim minted and issued exclusively by Identity Service (BRD §24.1's role table: "Admin | Full back-office operations | Catalog management, coupon issuance, user suspension — all `/admin/*` routes") — Admin Service does not define or issue this claim itself.
- Perform "back-office operations" (BRD §2.1 item 20) — **now a closed enumeration** per ADR-0010 (`docs/adr/0010-admin-service-scope-and-integration.md`): four categories — catalog management (create/update/deactivate a `Product`; create/update/reorder/move a `Category` node), coupon issuance (create/deactivate a `Coupon`), user suspension (lock/unlock a user's ability to authenticate), inventory replenishment (manually trigger a restock adjustment, publishing `InventoryReplenished`). Review moderation and Payment refunds are explicitly excluded — BRD §2.1 item 14 assigns moderation to Review Service, and BRD §24.1's Support Agent example grant (capped refunds) runs through Payment Service under the `Support Agent` role, not `Admin`. See §6 Decision item 2.
- Enforce fine-grained, action-level access control on every admin action before it executes, on top of the coarse `Admin` role claim already validated at the Gateway. BRD §24.1 states enforcement happens twice: "coarse-grained at the API Gateway (§18, reject before the request reaches a service) and fine-grained again at the owning service" — using "Support Agent can refund, but only up to $X" (a Payment Service example) as the pattern for what "fine-grained, service-owned" enforcement means. Admin Service is the owning service for its own `/admin/*` actions in exactly this sense: the Gateway can confirm the caller holds the `Admin` role, but only Admin Service itself can know, e.g., whether *this* admin is permitted to suspend *this specific* account. **Now resolved (single-service engineering default, §6 Decision item 1):** a default-deny, category-scoped permission-grant model, owned by Admin Service itself and layered on top of (never replacing) the Identity-issued `Admin` claim, scoped to the same four categories named above.
- Publish `AdminActionPerformed` after an admin action (BRD §5.4). This event now has a full row in the Event Catalog (§10, added by **ADR-0007**): publisher Admin, consumer Analytics (audit trail), payload `adminId`, `action`, `entityId`, retry 1x (fire-and-forget audit tier), DLQ `admin.dlq`. **Atomicity now resolved:** publication is guaranteed-atomic with the triggering write via the Outbox pattern (BRD §11) — decided in `edge-cases.md`'s "Admin Action Succeeds but `AdminActionPerformed` Is Lost," restated as closed in §6 Decision item 7.
- No consumed events are stated for this service — BRD §5.4 lists "—" under Admin's "Key Events Consumed" column. **Confirmed as intentional, not a gap:** Admin is an operator-driven console — every `/admin/*` action is initiated synchronously by a human back-office user, not triggered by a platform event. The one plausible event-reactive candidate (inventory replenishment) is resolved as an *automated* trigger owned by Inventory Service itself (`kart-inventory-service/requirement-spec.md` §6 Decision 5 — threshold-based reorder signal), not something Admin Service consumes an event to perform; Admin's own replenishment path is the manual, human-initiated one from the same enumeration above. No Open Question remains here.

## 4. Domain Invariants

- Admin Service does not own or maintain an independent *role* table as a source of truth for "who can do what" at the coarse level — it consumes the `Admin` role claim issued exclusively by Identity Service. This is stated directly at BRD §24.1: "RBAC is a platform-wide concern, not an Admin Service–local feature — every one of the 18 deployable services trusts the same role claims rather than maintaining its own permission table. Identity Service is the single issuer of roles; every other service is a **consumer** of the claim." The coarse model — four platform-wide roles (Customer, Support Agent, Admin, Partner API), one issuer, two-tier enforcement — is BRD-specified. **What Admin *does* own, and is required to by BRD §24.1's own "fine-grained again at the owning service" clause, is a second, narrower layer: a category-scoped permission-grant table, resolved in §6 Decision item 1.** This is not a contradiction of "never a second source of truth" — Identity remains the sole source of truth for *who holds the `Admin` role at all*; Admin's own grant table is the source of truth only for *which of the four back-office categories (§6 Decision item 2) a given `Admin`-role holder may exercise*, the exact "owning-service, fine-grained" layer §24.1 describes for Payment's capped-refund check. Revoking a fine-grained grant takes effect on Admin's very next request (Admin looks its own table up fresh every time, unlike the JWT-embedded coarse claim) — see `edge-cases.md`'s "Stale Admin Permission Outliving an Identity-Side Revocation" for how this interacts with coarse-role staleness.
- A state-changing admin action and its corresponding `AdminActionPerformed` record must not silently diverge. **Resolved (§6 Decision item 7):** Admin Service writes the `AdminActionPerformed` row into the same PostgreSQL transaction as the triggering business write (BRD §11's Outbox pattern), with a poller publishing it after commit — see `edge-cases.md`'s "Admin Action Succeeds but `AdminActionPerformed` Is Lost."
- Admin authorization ultimately traces back to a validated, signed identity minted exclusively by Identity Service — Admin Service never mints its own admin-scoped credentials or maintains a parallel session/role store. This is resolved directly by BRD §24.1 (single-issuer model: "Identity Service resolves a user/service-principal to a role set at token-mint time and embeds it as a scoped claim in the JWT") and §24.2, which states Admin/back-office SSO federation is terminated entirely at Identity Service against the org's enterprise IdP (SAML 2.0/OIDC) and that "no other service ever validates a SAML assertion or holds an external IdP client secret."
- Admin Service never becomes a second owner of another service's domain data. **Resolved (§6 Decision item 5, ADR-0010 (`docs/adr/0010-admin-service-scope-and-integration.md`)):** for every operation in the closed enumeration (§6 Decision item 2), Admin invokes the owning service's own write API synchronously (internal, client-credentials service-to-service call); Admin's own PostgreSQL stores only its permission grants and its `AdminActionPerformed` outbox rows, never a duplicate copy of Product/Category/Coupon/user-identity/Inventory data.

**Resolved this pass (full closure log, renumbered to match the prior draft's own numbering):**

1. **Fine-grained, per-action permission model within the Admin role — RESOLVED (single-service engineering default, Decision D1).**
   - *Decision:* Admin Service owns a category-scoped, default-deny permission-grant model, layered on top of (never replacing) the Identity-issued `Admin` claim. Admin owns one Admin-local table, `admin_permission_grants` (`principalId`, `category`, `grantedAt`, `grantedBy`, `revokedAt` nullable), scoped to the four categories fixed in item 2 below: `catalog-management`, `coupon-issuance`, `user-suspension`, `inventory-replenishment`. Every `/admin/*` handler maps to exactly one category; before executing, Admin checks for a live (non-revoked) grant row for `(principalId, category)` in addition to (never instead of) the Gateway-validated coarse claim — absence of a grant blocks the action (`403`) even with a valid `Admin` claim present. Grant issuance/revocation is itself an `/admin/*` action, gated by a fifth meta-category, `permission-management`, which is not self-service in v1: the first `permission-management` grant for a deployment is seeded out-of-band by a one-time ops migration/seed script (avoiding an unbounded regress of "who grants the first grant"); once at least one principal holds it, further grants of any category are issued through the same `/admin/*` surface like any other admin action.
   - *Why:* RBAC is Admin Service's stated reason to exist (BRD §2.1 item 20); a default-deny posture fails closed when a new action is added and its grant is forgotten, matching `edge-cases.md`'s "Privilege Escalation" decision now made concrete; scoping to the same four categories item 2 uses means one model answers both "what can Admin do" and "who can do it."
   - *Trade-off accepted:* a new admin action added later must be explicitly mapped to one of the four categories (or the table extended with a fifth) before it is usable — an intentional fail-closed cost. The concrete grant-management CRUD surface (routes, payload shape) is not fixed here — see "Non-blocking, carried forward" below.
2. **"Back-office operations" enumeration — RESOLVED (genuinely cross-cutting; new PENDING ADR).** See ADR-0010 (`docs/adr/0010-admin-service-scope-and-integration.md`), Decision 1: a closed, four-category enumeration for v1 — catalog management (Product/Category mutation), coupon issuance (Offer's `Coupon` aggregate, per ADR-0001), user suspension (Identity's auth-identity domain, per ADR-0006), inventory replenishment (manual restock trigger, publishing `InventoryReplenished`). Review moderation (BRD §2.1 item 14, Review Service's own domain) and Payment refunds (BRD §24.1's Support Agent grant, via Payment Service) are explicitly excluded. This was cross-cutting, not a single-service call, because it independently blocked `kart-category-service` Open Question #6, `kart-product-service` Open Question #4, and `kart-inventory-service` Open Question #5 — each needed Admin's side of this answer to close its own write-path question.
3. **Audit completeness beyond `AdminActionPerformed` — RESOLVED (single-service engineering default, Decision D3).** `AdminActionPerformed` (published via the Outbox pattern, item 7 below) is Admin's authoritative, sole audit trail — no separate compliance log is introduced, consistent with `kart-analytics-service/requirement-spec.md`'s own treatment of this event as its "Admin audit/compliance dashboard" source. Two properties fixed as defaults: **immutability** — the outbox/admin-action table is append-only at the application layer (no `UPDATE`/`DELETE` exposed once a row is written; only the poller's `published_at` column mutates, per BRD §11's standard mechanics); **retention** — 5 years, matching BRD §4.3's own stated precedent for Order's write storage (no platform-wide audit-retention number exists in the BRD, so this borrows the one concrete figure the BRD already commits to for a comparably compliance-sensitive table rather than inventing an unprecedented one). *Why:* back-office actions are the platform's highest-privilege operations (Security NFR); an editable or arbitrarily-expiring audit trail undermines the reason it exists. *Trade-off accepted:* 5 years of rows must be retained (tiered to cold storage as a later cost detail), the same cost Order already accepts for the same reason.
4. **Availability and latency tiers — RESOLVED (single-service engineering default, Decision D4).** Availability: secondary tier, 99.9% (BRD §3) — Admin is confirmed absent from the Order Saga and the §5.5 boundary diagram, the BRD's own dividing line between the two tiers. Latency: no dedicated customer-facing SLA (no path in §5.5 to budget against), but writes are held to the BRD's general write-path ceiling as an internal operational ceiling: P95 < 300ms including the Outbox insert (BRD §3), monitored via the same RED metrics (BRD §23) as every service but not paged on-call the way order-path latency is. *Why:* reuses two numbers the BRD already commits to rather than inventing new ones for a service the BRD never puts a bespoke number against.
5. **Integration pattern with other services' data — RESOLVED (genuinely cross-cutting; new PENDING ADR).** See ADR-0010 (`docs/adr/0010-admin-service-scope-and-integration.md`) Decision 2: for every operation in item 2's enumeration, Admin invokes the owning service's own write API synchronously (internal, client-credentials service-to-service call) — Product's, Category's, Offer's, Identity's, or Inventory's write API, respectively. Admin's own PostgreSQL stores only its own permission grants (item 1) and its `AdminActionPerformed` outbox rows — never a duplicate copy of another service's domain data. Cross-cutting for the same reason as item 2 (the same three sibling specs depend on this answer).
6. **Overlap with "admin tooling" in the messaging design (§8.1) — RESOLVED (single-service clarification, Decision D6).** Two distinct concerns share the word "admin": BRD §8.1's "DLQ ... inspected via admin tooling" is generic ops/SRE tooling (e.g., a RabbitMQ management console or internal ops dashboard) used to triage dead-letter messages across every service's queue — it is not this Admin Service's customer-domain `/admin/*` business surface (item 2's four categories), and DLQ inspection is not modeled as one of Admin Service's own back-office operations. No cross-cutting impact — this is a naming disambiguation, not a decision another service depends on.
7. **`AdminActionPerformed` publication atomicity (Outbox) — RESOLVED (single-service engineering default, Decision D7; already decided in `edge-cases.md`, restated here for closure).** Admin writes the `AdminActionPerformed` row into the same PostgreSQL transaction as the triggering admin action (BRD §11's Outbox pattern); a poller publishes it after commit. See `edge-cases.md`'s "Admin Action Succeeds but `AdminActionPerformed` Is Lost" for the full Why/Trade-off. Domain Invariant #2 (§4) is updated accordingly.

# Architecture: kart-user-service

## Boundary Rationale

`kart-user-service` is the bounded context that answers "what does this user's profile look like" — addresses, preferences, and a denormalized display copy of a couple of Identity-owned fields — never "who is this and are they allowed to do X," which stays Identity's alone (BRD §2.1 item 2 vs. item 1; requirement-spec.md §1). The boundary between the two is not this service's own call to make and is not re-litigated here: [ADR-0006](../../adr/0006-identity-user-profile-sync-event.md) already settled it platform-wide — Identity owns login-email/display-name as source of truth and publishes `UserAccountUpdated` on change; User owns the broader profile (addresses, preferences) and keeps a denormalized, eventually-consistent copy of those two fields, reconciled via that event. `kart-identity-service/architecture.md`'s own Boundary Rationale states the identical split from its side ("Identity never models a full user profile, and User never re-derives or overrides Identity's copy of login email/display name") — the two docs agree, no divergence to flag.

User Service is a CQRS-split service internally (PostgreSQL write model, MongoDB read projection via the platform's standard Outbox pipeline — requirement-spec.md §2, §3, §7), but that split is an internal implementation detail, not a boundary concern: both stores are owned entirely by this one service, and no other service ever writes to either directly.

User Service is absent from the Order Saga (BRD §12) and from BRD §5.5's boundary diagram — requirement-spec.md §3/§6 item 7 already committed it to the platform's secondary availability tier (99.9%) on that basis. Nothing else on the platform synchronously blocks on User Service being up, and (see Dependencies below) User Service itself holds no synchronous outbound dependency on any other service — the least-coupled boundary shape a non-Saga, profile-holding service can have.

The GDPR/erasure policy (ADR-0016, `docs/adr/0016-user-gdpr-erasure-policy.md`) is likewise cited, not re-decided, here: PII is tombstoned in the write model within 30 days of a verified request, retained order/audit history elsewhere on the platform is anonymized rather than deleted, and a new `UserDataErased` event fans the erasure out to the six named downstream consumers. This document's job is only to place that already-decided policy into the dependency graph (see Dependencies) and to resolve the one piece it genuinely left open — the erasure-request intake mechanism (see the dedicated section below).

## Resolved Integration-Contract Question: Erasure-Request Intake Mechanism (design-decisions.md, "Not Decided Here," item 5)

`design-decisions.md` correctly deferred this rather than inventing a caller/topology at the design-pattern stage ("not enough grounding here to decide without inventing the caller/topology — left to the Architecture Agent"). Resolved here, to the extent this stage legitimately can:

- **Mechanism decided: a synchronous internal REST call, not an event.** A verified erasure request needs a request-time confirmation that User Service has accepted the tombstone obligation — an acknowledged compliance commitment, not a best-effort fire-and-forget — the same reasoning `kart-admin-service/architecture.md`'s "Sync vs. Async Resolution" already applies to its own audited back-office actions: a write with a compliance/audit consequence should be confirmed synchronously before the caller treats the request as accepted, not learned about later through an event that might land in a DLQ. An async "erasure requested" event was considered and rejected on this basis.
- **Caller: not yet a modeled platform service — flagged for a human decision, not invented.** The platform's one existing precedent for a back-office actor calling a user-scoped internal endpoint is Admin Service's `POST /internal/users/{userId}/lock`/`unlock` (ADR-0010), but Admin Service's scope is fixed to exactly four categories (catalog, coupon, user-suspension, inventory-replenishment, per `kart-admin-service/architecture.md`) and does not include a compliance/erasure category — adding a fifth would be new scope requiring its own amendment to ADR-0010, not something this document can decide unilaterally for a service it isn't profiling. Identity Service is the other plausible candidate (it owns the account/credential lifecycle), but neither its `requirement-spec.md` nor its own `architecture.md` names such a responsibility today. **This is a genuine open item**, consistent with this agent's own escalation rule (do not silently pick a side on a cross-document gap) — a human should decide which existing service, if any, becomes the caller, or whether a dedicated compliance/support tooling surface outside the current 18-service graph is the intended trigger instead.
- **What is decided regardless of who the caller turns out to be:** the endpoint is internal-only (never reachable via the public API Gateway), client-credentials-gated, and its sole effect is to start the already-decided ADR-0016 tombstone write / 30-day SLA / `UserDataErased` fan-out — no new domain behavior beyond what ADR-0016 already specifies.

## Distributed-Monolith Risk

**None identified at the synchronous-call-graph level.** User Service has zero synchronous outbound dependencies on any other service; its only synchronous edges are both inbound (the client-facing `/users/{id}` surface and the new internal erasure-request intake call), and neither chains into a further synchronous call — a request into User Service never fans out synchronously to Identity, Order, or any other peer. This matches the "foundational-but-decoupled" shape requirement-spec.md's secondary-tier availability posture already implies, not the "chatty, should-be-async" pattern this stage exists to catch.

Two risks worth restating here — both already accepted elsewhere, not newly introduced by this document, but now visible as real edges in a dependency-graph-complete picture rather than a single-service one:

- **The `UserAccountUpdated` ordering gap remains genuinely open**, not resolved by this document. Until the Event Design Agent confirms (or rejects) the proposed `updatedAt`/sequence field, a redelivered or out-of-order `UserAccountUpdated` can let a stale email/display-name value overwrite a newer one in User's denormalized copy — a known, named gap (already flagged from Identity's side), not papered over here by unilaterally assuming the field exists.
- **Downstream `UserDataErased` consumption is fan-out, not fan-in, and its completeness depends on six other services' own future docs passes**, not on anything User Service can enforce from its own boundary. If Order/Notification/Analytics/Review/Recommendation/Wishlist never add their own consumer, ADR-0016's guarantee only holds for User Service's own copy of a user's PII, not the platform's — an accepted risk already named in ADR-0016's own Consequences section, restated here only because it is now a real fan-out edge in this service's own dependency graph rather than an abstract future concern.

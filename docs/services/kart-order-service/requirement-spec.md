# Requirement Spec: kart-order-service

## 1. Scope

Covers BRD service **#8, Order Service** (BRD §2.1 row 8: "Order lifecycle, order state machine"). No merge — this is a single BRD service, unlike `kart-offer-service`; no ADR is required to justify scope.

Order is one of the BRD's three named deep-dive services (§5.1, "representative deep-dive — the Saga orchestrator") and is called out explicitly in §2.2's forced-decision list: *"Order Service is the Saga orchestrator touching Inventory, Payment, and Shipping → forces distributed transaction design."* Its boundary rationale (§5.1) is architectural, not just functional: **Order is the only service allowed to drive cross-service business transactions**, specifically to prevent a "distributed monolith" where every service calls every other service directly. Every requirement below is scoped by that single fact — this service's job is orchestration and state ownership, not owning inventory, payment, or shipping logic itself.

## 2. Functional Requirements

### Order Lifecycle & State Machine
- Own the order state machine end-to-end: `Created → Reserved → Paid → Shipped → Delivered → Cancelled/Refunded` (BRD §5.1).
- `POST /orders` — create an order; per the CQRS sequence (BRD §7) this returns `202 Accepted` after the write-side transaction commits, not after the full Saga completes.
- **Idempotency on `POST /orders` — resolved.** Requires a client-supplied `Idempotency-Key` header, deduplicated via a unique constraint (mirroring Payment's `idempotency_keys` table, BRD §5.3/§6.1). A retried request carrying a previously-seen key returns the original order's `202`/representation rather than creating a second order. See Open Questions resolution #3 below and edge-cases.md's "Duplicate Order Submission" for the full analysis; this is now the final decision, not a proposed default — the exact request/response wire shape is left to the API Design Agent (non-blocking carry-forward).
- `GET /orders/{id}` — read an order; served from the MongoDB projected read model (BRD §7), not directly from PostgreSQL, per the platform's CQRS split (BRD §6.1: Order is write-heavy, PostgreSQL is source of truth for writes only).
- `POST /orders/{id}/cancel` — client-initiated cancellation (BRD §5.1). **Legal only pre-`Shipped`** (`Created`/`Reserved`/`Paid`) — resolved in Open Questions #2 below; once `Shipped`, this endpoint rejects (e.g. `409 Conflict`) and the customer must use the separate returns/refund flow (Payment's `/payments/{id}/refund`, BRD §5.3, §12). Routes through the same Saga/compensation state machine as any other failure trigger, per edge-cases.md's "Client Cancel Request Racing an In-Flight Saga" decision — single writer, no separate cancel code path.
- **`Shipped → Delivered` transition is event-driven, not client- or Saga-initiated (ADR-0005).** Order consumes `DeliveryStatusUpdated`, published by Delivery Tracking Service, and on its terminal "delivered" status value publishes `OrderDelivered`, moving the order into the `Delivered` state (BRD §5.1, §10 — both now list this consume/publish pair, citing ADR-0005). See "Terminal delivery event" under Saga Orchestration below for the full resolution and its provenance.
- **`Paid → FulfillmentException` — new non-terminal hold state (ADR-0015).** Order consumes Shipping's `ShipmentCreationFailed` (published once every configured carrier option is exhausted without producing a valid label) and, on receipt, transitions the order into a new, distinguishable status: `FulfillmentException`. This is a **post-confirmation** failure — it only fires after `OrderConfirmed` has already been published (ADR-0002) and Payment has already captured funds — so it is structurally different from the pre-confirmation compensation flow in BRD §12.2, which only ever unwinds an unconfirmed reservation/charge. See "Post-Confirmation Fulfillment Exception Handling" under Saga Orchestration below for the full reaction and its resolution rationale (this was flagged as Order's own open design call in ADR-0015, not decided there).
- **`ChargebackReceived` reaction — new (ADR-0012).** Order consumes Payment's `ChargebackReceived`, published when a card network/bank reverses a charge externally, after the fact. Unlike every other consumed event, this can arrive at **any** point after `PaymentCompleted` — including after `Delivered` — since a chargeback is a customer/bank-initiated dispute with no relationship to Order's own saga timing. See "Chargeback Reaction" under Saga Orchestration below.

### Saga Orchestration
- Drive the distributed, compensable transaction across three downstream services (BRD §5.1 Dependencies): Inventory (synchronous reserve call + asynchronous confirm), Payment (asynchronous charge), Shipping (asynchronous shipment creation — see integration-direction note immediately below).
- **Order↔Shipping integration direction — resolved BRD contradiction (ADR-0002).** BRD §12.1's Saga sequence diagram renders `Order->>Shipping: Create shipment` / `Shipping-->>Order: ShipmentDispatched` using the same request/reply arrow notation as a synchronous call, and places `Order->>Order: Mark OrderConfirmed` *after* that exchange — read literally, this implies Order blocks on Shipping before confirming. That reading directly contradicted §5.4 and §10, both of which describe Shipping as a pure async consumer of `OrderConfirmed` with no synchronous inbound endpoint, publishing `ShipmentDispatched` back independently. `docs/services/README.md`'s gap table tracked this as an open cross-service contradiction, and this service's own docs previously did not flag it as unresolved — one of the two findings that disqualified the prior draft of this spec. **ADR-0002 resolves it**: the integration is async, end to end. `OrderConfirmed` is published as soon as `PaymentCompleted` is received — that is the actual confirmation trigger, not shipment creation — and `ShipmentDispatched` flows back to Order (and to Notification, Delivery Tracking) purely as an informational status update, never as a gate on confirmation. `kart-requirements.md` §12.1 now carries a clarifying prose note under the diagram adopting this reading (the same request/reply notation is used for the unambiguously-async Payment step, so the notation itself never carried sync/async meaning — only causal order). This spec adopts ADR-0002's resolution rather than the diagram's literal notation.
- Success flow (BRD §12.1, read per ADR-0002's resolution above — **not** the diagram's literal step order): reserve stock → charge payment → mark `OrderConfirmed` (published as soon as `PaymentCompleted` is received, not gated on shipment creation) → Shipping asynchronously consumes `OrderConfirmed` and creates the shipment, publishing `ShipmentDispatched` back to Order as an informational status update.
- Failure/compensation flow (BRD §12.2, the only compensation path the BRD documents): if `PaymentFailed` arrives after inventory was reserved, release the inventory reservation as a compensating action, then mark `OrderCancelled`.
- Refunds are explicitly **not** modeled as a continuation of the order Saga: BRD §12 notes refund-after-dispute is "tracked as its own saga instance rather than reusing the order saga state machine, since refunds can be partial and asynchronous relative to the original order lifecycle." Order does not own refund orchestration.
- **Payment-succeeds/Shipping-fails compensation ordering — resolved, scope narrowed by this pass' ADR-0015 amendment.** When a step later than Payment fails or times out after `PaymentCompleted` was already received, Order compensates in reverse execution order: release the Inventory reservation first, then issue the Payment refund. See Open Questions resolution #1 below and edge-cases.md's "Payment-Success/Shipping-Failure Compensation Ordering" for the full analysis; this is now the final decision, not a proposed default. **This mechanism (reverse-order auto-compensation) no longer applies to the Shipping step specifically** — ADR-0015's `FulfillmentException` supersedes it there, for both an explicit `ShipmentCreationFailed` signal and a silent Shipping timeout caught by the reconciliation sweep (see Open Questions resolution #6's amendment below): both route to the same manual-resolution-only hold state, never to automatic refund-and-cancel, since an unresponsive Shipping step is no more safely auto-reversible than an explicitly-failed one. The reverse-order mechanism itself remains the one Order uses when `FulfillmentException` is manually resolved to `Cancelled` (a human, not a timeout, triggers it there) and is the pattern this spec's Chargeback Reaction reuses for its own conditional Inventory release.
- **Per-step Saga timeout budget — resolved.** Inventory synchronous reserve call: **2s**. Payment asynchronous completion (`PaymentCompleted`/`PaymentFailed`): **30s**. Shipping asynchronous dispatch (`ShipmentDispatched`): **60s**. See Open Questions resolution #6 below for the full reasoning and how this also bounds the Orphaned-Saga reconciliation threshold and the `DeliveryStatusUpdated`-before-`Shipped` hold/retry window in edge-cases.md.
- **Partial fulfillment / split shipment — out of scope for v1.** Exactly one shipment per order; no per-item shipment state or backordering-a-subset support. See Open Questions resolution #4 below.
- **Terminal delivery event — resolved BRD self-contradiction (ADR-0005).** BRD §5.4 originally named two different terminal-order events with no publisher anywhere in the BRD: Recommendation's row consumed `OrderCompleted`, Review's row consumed `OrderDelivered`, and Order's own Publishes list (§5.1, §10, pre-ADR) contained neither. `docs/services/README.md`'s gap table tracked this contradiction; this service's own prior draft silently omitted `OrderDelivered` entirely rather than flagging it — the second of the two findings that disqualified that draft. **ADR-0005 unifies both names into one event, `OrderDelivered`, published by Order** — chosen over `OrderCompleted` because it matches Order's own state-machine vocabulary exactly (`Delivered` is already a named state; `Completed` is not). Order publishes `OrderDelivered` when its state machine transitions to `Delivered`, triggered by consuming Delivery Tracking Service's `DeliveryStatusUpdated` event and detecting its terminal "delivered" status value (BRD §5.1, §10 now list this consume/publish pair directly, both citing ADR-0005). This adds a **fourth dependency Order did not previously have**: Delivery Tracking Service, alongside Inventory/Payment/Shipping — distinct from those three in that it is not part of the compensable forward transaction; there is no compensation path for a delivery-status event, since it only fires after the order has already reached `Shipped` successfully. **State-machine implication**: `Shipped → Delivered` is the one lifecycle transition Order does not actively drive itself — there is no client-facing API and no Order-Saga-internal trigger for it. For this transition only, Order is a passive listener for `DeliveryStatusUpdated`'s terminal value, and that consumption is the *sole* trigger for both the `Delivered` transition and the `OrderDelivered` publish.
- **Post-Confirmation Fulfillment Exception Handling — resolved (ADR-0015 names the expectation; the reaction mechanism is this service's own call, made here).** ADR-0015 introduces `ShipmentCreationFailed` (Publisher: Shipping; payload `orderId, reason`; 3x retry, `shipping.dlq`) and explicitly states that Order's reaction "is Order Service's own subsequent design decision, not one this ADR makes on Order's behalf." Resolved: on consuming `ShipmentCreationFailed`, Order transitions the order to a new **`FulfillmentException`** status — a non-terminal, distinguishable hold state, reachable only from `Paid` (i.e., after `OrderConfirmed`/`PaymentCompleted`, before the order successfully reaches `Shipped`). Order does **not** run its existing pre-confirmation compensation flow (§12.2) automatically, because that flow's actions (release Inventory, mark `OrderCancelled`) assume a reservation/charge that never completed — here, payment has already been captured. Resolution out of `FulfillmentException` is a manual/operational action (via Admin Service or ops tooling, outside this service's own v1 client-facing API surface), which drives the order to exactly one of two legal exits: back to `Paid` (a corrected address/carrier retry succeeds and Shipping is asked to re-attempt — modeled as re-entering the same pre-`Shipped` waiting state, not a new state) or to `Cancelled` **with a conditional Inventory-release attempt (idempotent no-op if no longer held) followed by an explicit Payment refund call** (distinct from the reverse-order pre-confirmation compensation in Open Questions resolution #1, since here Payment has already captured funds — the refund call is the same direct `POST /payments/{id}/refund` mechanism used for Chargeback below, and the conditional release is the same idempotent pattern Chargeback Reaction uses, not two new mechanisms). Why a manual gate rather than full automation: an automatic refund-and-cancel on the first carrier failure would silently abandon fulfillment attempts a human ops reviewer might resolve (e.g., a corrected address), and BRD gives no basis for a specific auto-retry policy — this mirrors the platform's existing "Orphaned/Stuck Saga Detection" reconciliation-job pattern (edge-cases.md) in spirit (surface, don't silently resolve) but requires an explicit human trigger given the money already captured. **The reconciliation sweep's Shipping-step timeout (Open Questions resolution #6) also enters `FulfillmentException`, not automatic reverse-order compensation** — an order stuck awaiting `ShipmentDispatched` past the sweep's threshold with no `ShipmentCreationFailed` ever received is treated identically to one that did receive it, since a silently unresponsive Shipping step is no safer to auto-reverse than an explicitly-failed one; this is the scope-narrowing this pass applies to resolution #1 (see Saga Orchestration's compensation-ordering bullet above). Resolved: the manual resolution surface is a dedicated Order-owned endpoint, `POST /orders/{id}/resolve-fulfillment-exception`, callable only by Admin Service's service principal (architecture.md) — the API-surface question is no longer open.
- **Chargeback Reaction — resolved (ADR-0012 names the expectation; the reaction mechanism is this service's own call, made here).** ADR-0012 introduces `ChargebackReceived` (Publisher: Payment; payload `orderId, paymentIntentId, chargebackId, amount, reason`; 5x retry, `payment.dlq`, paged on-call) and states Order's required reaction is to "hold or cancel the order using its own existing compensation logic — release Inventory if still held, mark the order into a terminal `Cancelled`/`Refunded`-adjacent state — but does **not** call Payment for a refund." Resolved: on consuming `ChargebackReceived`, Order (a) releases the Inventory reservation only if one is still held (a state-guarded, idempotent no-op in the overwhelmingly common case, since a chargeback typically arrives well after Inventory's 15-minute reservation TTL has already converted/expired — inferred from `kart-inventory-service`'s own approved `ddd-model.md`), and (b) transitions the order directly to **`Refunded`**, regardless of its current lifecycle state at the moment the chargeback arrives. This is a deliberate, narrow carve-out of the "Legal state transitions only" invariant's existing rule that `Refunded` is reachable only from `Delivered` (see Domain Invariants §4 below) — a chargeback is an externally-forced reversal that PaymentIntentStatus's own state machine (`kart-payment-service/ddd-model.md`) only allows from `Completed`, meaning it can legitimately arrive while Order is in **any** state from `Paid` onward — `Paid`, `Shipped`, `Delivered`, **or `FulfillmentException`** (an order held on a fulfillment exception has certainly had `PaymentCompleted` already, so a chargeback against it is exactly as legitimate as against any other post-`Paid` state; omitting it here would mean a genuine bank-initiated chargeback against a held order is silently dropped, which is the one failure mode this event's own existence is meant to prevent) — not only post-delivery. Order never calls Payment's refund endpoint in reaction to a chargeback — the bank has already reversed the charge externally, and a second Payment-initiated refund would double-reverse the same funds, directly violating the "Payment must never double-charge" domain rule's inverse (BRD §2.2).

### Persistence & Messaging
- Write side: PostgreSQL tables `orders`, `order_items`, `order_events` (BRD §5.1), range-partitioned by month on `orders`/`order_events` (BRD §6.1) for bounded index size and archival.
- Outbox Pattern (BRD §11, §7): the order state transition and its outbox row must be written in the same DB transaction; a poller publishes from the outbox to RabbitMQ, marking rows published only after broker ack.
- **`order_events` doubles as the Outbox table — resolved (Open Questions resolution #5 below).** There is no separate, unlisted `outbox` table for Order: `order_events` serves both roles (append-only state-transition audit log, and the transactional relay BRD §11 describes generically), carrying the Outbox-specific columns (`published_at`, event-type/routing metadata) alongside its audit fields, with `idx_outbox_unpublished`-equivalent index created against it.
- Publish: `OrderCreated`, `OrderConfirmed`, `OrderCancelled`, `OrderCompensationTriggered`, `OrderDelivered` (BRD §5.1; `OrderDelivered` added per ADR-0005 — see "Terminal delivery event" above).
- Consume: `InventoryReserved`, `InventoryReservationFailed`, `PaymentCompleted`, `PaymentFailed`, `ShipmentDispatched`, `DeliveryStatusUpdated` (terminal "delivered" status value only) (BRD §5.1; `DeliveryStatusUpdated` added per ADR-0005 — see "Terminal delivery event" above), `ShipmentCreationFailed` (Shipping, ADR-0015 — see "Post-Confirmation Fulfillment Exception Handling" above), `ChargebackReceived` (Payment, ADR-0012 — see "Chargeback Reaction" above).

**Cross-cutting Security obligations added by BRD §24.1.2, §24.1.4, §24.1.5, §24.3 (cross-reference only — the detailed "why" lives in the BRD itself and in this service's own ddd-model.md/database-design.md, not re-derived here):**

- **§24.1.2 (CanRead/CanWrite/CanDelete, service-owned):** Order is the resource owner of the `Order` aggregate (`orders`, `order_items`, `order_events`) and defines its own fine-grained permission mechanism for it — the BRD's own §24.1.2 worked table already names Order directly: **CanWrite gated by the order's own current state-machine position (an inline state-machine guard clause), CanDelete never exposed** (orders are never hard-deleted, only advanced through the state machine to a terminal `Cancelled`/`Refunded` status, per §4's Legal Transitions invariant). `CanRead` is a plain ownership check (`orders.user_id == token.sub`) for the owning Customer, widened to Support Agent/Admin per their coarse role for order-support workflows. `CanWrite` beyond the owning Customer's narrow `POST /orders/{id}/cancel` (itself gated by the pre-`Shipped` state guard, §4) is otherwise restricted to Order's own Saga-step consumers and to Admin Service's client-credentials principal on `POST /orders/{id}/resolve-fulfillment-exception` (§5, architecture.md) — never a second, independent ownership path. The concrete mechanism is recorded as a Domain Invariant in ddd-model.md.
- **§24.1.4 (row-level security):** Order's write model is entirely PostgreSQL (`orders`, `order_items`, `order_events` — database-design.md), so native RLS (`ENABLE ROW LEVEL SECURITY` / `CREATE POLICY`, keyed on a session-scoped `current_setting('app.current_principal')`) is the mechanism used, applying across `orders`' monthly range partitions the same way it applies to an unpartitioned table (PostgreSQL RLS policies on a partitioned parent are inherited by every partition automatically). The MongoDB `order_read_model` projection has no native declarative RLS equivalent (BRD §24.1.4), so the same guarantee is met by routing every read against that collection through this service's own single, unbypassable data-access-boundary query-builder. See database-design.md's "Row-Level Security Policy" subsection for the concrete per-table policies.
- **§24.1.5 (column-level security):** No — Order stores commercial/transactional data (`sku`, `qty`, `unit_price`, `status`, `total_amount`) and an opaque `user_id` owner reference, none of which is the kind of column §24.1.5 exists to mask; there is no phone/address/credential-shaped column on this service's own tables (contrast with User Service's `phone`/address columns, the BRD's own worked example). See database-design.md's "Sensitive / PII Column Classification" subsection for the full reasoning.
- **§24.3 (default audit fields):** every mutable table in database-design.md (`orders`, `order_items`, `order_events`) now carries `created_at`/`updated_at`/`created_by`/`updated_by`, auto-injected by the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor reading the JWT/service-principal/system principal (§24.3's own stated mechanism — one platform-wide implementation, not built locally by this service) — never a value a request handler sets directly. System-initiated mutations (each Saga-step event consumer, the reconciliation sweep, the Outbox poller) stamp a well-known `system:*` principal id, never `NULL` — see ddd-model.md's audit-actor invariant for the full per-trigger mapping.

## 4. Domain Invariants

- **Sole orchestrator invariant**: no service other than Order may drive a cross-service business transaction (BRD §5.1 Boundary Rationale). This is an architectural invariant, not just a functional convenience — it is the entire reason Order exists as a separate service from Inventory/Payment/Shipping. Order's consumption of Delivery Tracking's `DeliveryStatusUpdated` (ADR-0005) does not weaken this invariant: Order still owns the state machine and is the only writer of `orders`/`order_events`; Delivery Tracking never calls Order or writes Order's state directly, it only emits a status event Order chooses to react to.
- **Compensation completeness**: if any Saga step fails, all previously-succeeded steps for that order must be fully compensated before the order is marked in a terminal failure state — an order must never end up `Cancelled` while money remains charged or stock remains held (inferred from BRD §2.2's Payment "must never double-charge" and Inventory "must never oversell" rules, which Order's Saga is the mechanism enforcing across service boundaries).
- **Legal state transitions only**: the order state machine (`Created → Reserved → Paid → Shipped → Delivered → Cancelled/Refunded`, BRD §5.1, extended with `FulfillmentException` per ADR-0015) must only advance along transitions the domain actually permits — no transition may skip states or move backward except into the terminal states. All branches are now fully specified:
  - `Shipped → Delivered`: driven solely by consuming `DeliveryStatusUpdated`'s terminal status value (ADR-0005), with no other trigger.
  - `Cancelled` is reachable only from `Created`, `Reserved`, or `Paid` (i.e., pre-`Shipped`) — via client-initiated `POST /orders/{id}/cancel` or automatic Saga compensation, both routed through the same state machine as single writer. Once `Shipped`, cancellation is rejected; the customer must use the separate returns/refund flow instead (resolved in Open Questions #2 below). `FulfillmentException`'s own resolution-to-`Cancelled` exit (below) is a distinct, manually-triggered path, not this client-facing one.
  - `Paid → FulfillmentException`: driven solely by consuming `ShipmentCreationFailed` (ADR-0015). From `FulfillmentException`, the only legal exits are back to `Paid` (manual ops-triggered shipment retry) or to `Cancelled` (manual ops action, gated on an explicit Payment refund call completing first — see "Post-Confirmation Fulfillment Exception Handling" above). No automatic exit exists.
  - `Refunded` is reached via exactly two paths, both external to Order's own forward Saga logic: (a) the separate refund saga BRD §12 scopes out of Order's ownership, reachable only once an order has reached `Delivered` (BRD §12's "refund-after-dispute" framing is a post-delivery returns/dispute concept, distinct from the pre-`Shipped` compensation refund already covered by Q1's reverse-order compensation); or (b) **consuming `ChargebackReceived` (ADR-0012)**, reachable from **any** state from `Paid` onward — `Paid`, `Shipped`, `Delivered`, or `FulfillmentException` — a deliberate carve-out, since a chargeback is an externally-forced reversal not gated on delivery having happened, and a fulfillment-exception hold does not pause a bank's own dispute timeline (see "Chargeback Reaction" above). Order's own orchestration logic drives transition (b) directly; it never drives (a), only reflects it once that saga reports back.
  - `Cancelled` and `Refunded` are **mutually exclusive terminal states** reached via non-overlapping paths, never a sequence: an order that reached `Cancelled` via pre-confirmation compensation (Q1) or via a resolved `FulfillmentException` already had any charged payment unwound before or at the point of cancellation, so it never additionally becomes `Refunded`; an order that reached `Delivered` can only subsequently become `Refunded` via the external refund saga, never `Cancelled`, since cancellation is already barred from `Shipped` onward; an order reaching `Refunded` via `ChargebackReceived` (path (b) above) can occur from any state `Paid` onward including `FulfillmentException`, but once it does, the order is terminal and no further `Cancelled` transition (including a pending `FulfillmentException` resolution) is legal against it (a duplicate/redelivered `ChargebackReceived` against an already-`Refunded` order is an idempotent no-op, per the existing "Idempotent event consumption" invariant below). (All sub-bullets resolved in Open Questions resolution #2 below, extended by this pass' ADR-0012/ADR-0015 amendments.)
- **Idempotent event consumption**: duplicate delivery of `InventoryReserved`, `PaymentCompleted`, `PaymentFailed`, `ShipmentDispatched`, or `DeliveryStatusUpdated` must not double-advance the saga state (BRD §3 Reliability row: "at-least-once delivery + idempotent consumers"; BRD §13 Failure Simulation explicitly lists "Duplicate Message" as a scenario with the recovery "Idempotency key check at consumer — second delivery is a no-op"). `DeliveryStatusUpdated` is added to this invariant's scope now that Order consumes it (ADR-0005) — a duplicate or replayed terminal-status delivery must not re-publish a second `OrderDelivered` for the same order. `ShipmentCreationFailed` (ADR-0015) and `ChargebackReceived` (ADR-0012) extend this same scope: a duplicate `ShipmentCreationFailed` against an order already in `FulfillmentException` is a state-guarded no-op (does not re-trigger the transition or any notification-adjacent side effect a second time); a duplicate `ChargebackReceived` against an order already `Refunded` is likewise a no-op, never a second Inventory-release attempt or a second `Refunded` transition.
- **Atomic state-transition/event emission**: an order's state transition and the corresponding outbound event must commit atomically in the same database transaction (BRD §11 Outbox Pattern) — the "dual write problem" the Outbox Pattern exists to prevent applies most acutely to Order, since a lost `OrderCreated` stalls the entire downstream Saga.

### Resolved

1. **Payment-succeeds/Shipping-fails compensation ordering — RESOLVED.** Decision: compensate in reverse execution order — release the Inventory reservation first, then issue the Payment refund. This is a single-service Saga-orchestration default (Order is the sole orchestrator, BRD §5.1 Boundary Rationale); it doesn't change any other service's contract, so no ADR is needed. Full reasoning in §2 Saga Orchestration above and in `edge-cases.md`'s "Payment-Success/Shipping-Failure Compensation Ordering" — the trade-off (a slightly longer "compensating" window than parallel compensation) is accepted because it keeps the money-movement step last and most auditable, mirroring the reverse-of-forward-order convention BRD §12.2 already establishes.
2. **State-machine edge transitions — RESOLVED.** `Cancelled` is reachable only from `Created`/`Reserved`/`Paid` (pre-`Shipped`); post-`Shipped`, cancellation is rejected (`409`) and the customer must use the separate returns/refund flow. `Refunded` is never reached via Order's own state machine — it is exclusively the outcome of the separate refund saga (BRD §12), reachable only once an order has reached `Delivered`. `Cancelled` and `Refunded` are mutually exclusive terminal states, never a sequence (see §4 Domain Invariants for the full three-part rule and its justification). This also resolves `edge-cases.md`'s previously-escalated "Client Cancel Request Racing a Terminal `DeliveryStatusUpdated`" edge case: since cancellation is already illegal from `Shipped` onward, terminal delivery always wins that race by construction — there is no live tie-break to design. Single-service state-machine policy grounded in BRD §5.1's FSM and §12's refund-saga scoping note; no ADR needed (does not change any other service's contract — Payment's refund saga, Inventory's release semantics, and Shipping's dispatch semantics are all already scoped exactly as this decision assumes).
3. **Idempotency on `POST /orders` — RESOLVED.** Requires a client-supplied `Idempotency-Key` header, deduplicated via a unique constraint — the same pattern BRD §5.3/§6.1 already specifies for Payment's `/payments/charge`. BRD §24 already mandates "mandatory idempotency keys on all money-moving POSTs" platform-wide, so this is a direct application of an existing platform rule to a service that had simply omitted it, not a new policy choice — single-service, no ADR. See §2 above and `edge-cases.md`'s "Duplicate Order Submission." **Non-blocking carry-forward:** the exact request/response wire shape (header name is fixed as `Idempotency-Key` to match Payment; body/response schema is not) is the API Design Agent's job.
4. **Partial fulfillment / split shipment — RESOLVED, out of scope for v1.** Exactly one shipment per order; no per-item shipment state or backordering-a-subset support. Grounded in the BRD's own consistent singular treatment throughout (`ShipmentDispatched` is singular in §5.1/§5.4/§10, Shipping's own row at §5.4 models one shipment relationship, and no BRD section anywhere mentions partial fulfillment) — this is confirming an assumption the BRD already made implicitly everywhere, not introducing a new constraint. If a future version needs split shipments, the state machine, Saga, and event contracts all change materially and would need a fresh requirement pass across Order, Inventory, and Shipping together at that time.
5. **`order_events`/Outbox relationship — RESOLVED.** `order_events` doubles as the Outbox table: no separate, unlisted `outbox` table exists for Order. It carries the audit-trail role (state-transition history) and the Outbox-specific columns (`published_at`, event-type/routing metadata) together, with an `idx_outbox_unpublished`-equivalent partial index. This reuses BRD §11's generic Outbox schema against BRD §5.1's actual named table rather than inventing a second table BRD §5.1 never lists — single-service schema decision, no ADR.
6. **Per-step Saga timeout budget — RESOLVED, concrete numbers.** Inventory synchronous reserve call: **2s** (tight, because it's a synchronous call directly in the `POST /orders` write path, which itself carries a P95 < 300ms budget — BRD §3). Payment asynchronous completion (`PaymentCompleted`/`PaymentFailed`): **30s** (payment gateways commonly need a few seconds to a low tens-of-seconds window for card-network round trips; async, so it doesn't sit in the client-facing request path). Shipping asynchronous dispatch (`ShipmentDispatched`): **60s** (warehouse/carrier handoff is the slowest legitimate step and, per ADR-0002, is fully decoupled from order confirmation, so it can afford the widest budget). These are defensible engineering defaults derived from each step's position in the Saga (sync vs. async, gating vs. non-gating) and the BRD's own write-path latency target — not arbitrary. This also fixes two downstream thresholds `edge-cases.md` had left unresolved pending this number: the Orphaned/Stuck Saga reconciliation sweep threshold (set to 2× the relevant step's budget as a grace margin over the in-process timeout handler — 4s/60s/120s depending on which step the order is stuck awaiting) and the `DeliveryStatusUpdated`-before-`Shipped` hold/retry window (set to the Shipping dispatch budget, 60s, since that's the longest legitimate wait for the order to reach `Shipped` before the terminal delivery event should be considered genuinely racing ahead rather than just early). Single-service Saga-tuning decision; no ADR — the Architecture Agent may retune these under real load-test data (non-blocking carry-forward for numeric refinement only, not for the decision that a budget exists). **Amendment (this pass, post-ADR-0015):** the sweep's action on the 120s Shipping-await threshold is no longer "force reverse-order compensation" — it now forces entry into `FulfillmentException` instead, the same destination an explicit `ShipmentCreationFailed` drives (see "Post-Confirmation Fulfillment Exception Handling" above). This is the scope-narrowing this pass applies to resolution #1: a silent Shipping timeout is treated identically to an explicit Shipping failure, both requiring manual resolution rather than an automatic refund. The Inventory-await (4s) and Payment-await (60s) thresholds are unaffected — both steps are still pre-confirmation, where resolution #1's original automatic reverse-order compensation (or, for Payment, the existing `PaymentFailed` handling) still applies unchanged.
7. **Retry/DLQ tier for Order's own events — RESOLVED, elevated.** `OrderCreated`/`OrderConfirmed`/`OrderCancelled`/`OrderCompensationTriggered`/`OrderDelivered` are elevated from BRD §10's stated 3x/manual-replay-only tier to the same **5x exponential retry + paged on-call** tier as `Payment*`/`RefundIssued`. Why: BRD §3's Availability NFR row names "the order path" as the 99.99% tier explicitly, and Order is that row's primary subject; a stuck `OrderCreated` blocks Payment from ever being invoked, which is at least as operationally severe as a stuck `PaymentCompleted`, so gating Order's own lifecycle events behind a lower-touch manual-replay tier would undercut the very availability target BRD §3 assigns to Order by name. Trade-off accepted: more on-call paging volume for Order events than the BRD's default table implies. This is a retry-count/paging-tier override made directly in this spec against BRD §10's stated default — single-service operational tuning, not a cross-service contract change (no consumer's payload or behavior changes, only Order's own publish-retry/paging policy), so no ADR is needed.

# Requirement Spec: kart-order-service

## 1. Scope

Covers BRD service **#8, Order Service** (BRD §2.1 row 8: "Order lifecycle, order state machine"). No merge — this is a single BRD service, unlike `kart-offer-service`; no ADR is required to justify scope.

Order is one of the BRD's three named deep-dive services (§5.1, "representative deep-dive — the Saga orchestrator") and is called out explicitly in §2.2's forced-decision list: *"Order Service is the Saga orchestrator touching Inventory, Payment, and Shipping → forces distributed transaction design."* Its boundary rationale (§5.1) is architectural, not just functional: **Order is the only service allowed to drive cross-service business transactions**, specifically to prevent a "distributed monolith" where every service calls every other service directly. Every requirement below is scoped by that single fact — this service's job is orchestration and state ownership, not owning inventory, payment, or shipping logic itself.

## 2. Functional Requirements

### Order Lifecycle & State Machine
- Own the order state machine end-to-end: `Created → Reserved → Paid → Shipped → Delivered → Cancelled/Refunded` (BRD §5.1).
- `POST /orders` — create an order; per the CQRS sequence (BRD §7) this returns `202 Accepted` after the write-side transaction commits, not after the full Saga completes.
- `GET /orders/{id}` — read an order; served from the MongoDB projected read model (BRD §7), not directly from PostgreSQL, per the platform's CQRS split (BRD §6.1: Order is write-heavy, PostgreSQL is source of truth for writes only).
- `POST /orders/{id}/cancel` — client-initiated cancellation (BRD §5.1).
- **`Shipped → Delivered` transition is event-driven, not client- or Saga-initiated (ADR-0005).** Order consumes `DeliveryStatusUpdated`, published by Delivery Tracking Service, and on its terminal "delivered" status value publishes `OrderDelivered`, moving the order into the `Delivered` state (BRD §5.1, §10 — both now list this consume/publish pair, citing ADR-0005). See "Terminal delivery event" under Saga Orchestration below for the full resolution and its provenance.

### Saga Orchestration
- Drive the distributed, compensable transaction across three downstream services (BRD §5.1 Dependencies): Inventory (synchronous reserve call + asynchronous confirm), Payment (asynchronous charge), Shipping (asynchronous shipment creation — see integration-direction note immediately below).
- **Order↔Shipping integration direction — resolved BRD contradiction (ADR-0002).** BRD §12.1's Saga sequence diagram renders `Order->>Shipping: Create shipment` / `Shipping-->>Order: ShipmentDispatched` using the same request/reply arrow notation as a synchronous call, and places `Order->>Order: Mark OrderConfirmed` *after* that exchange — read literally, this implies Order blocks on Shipping before confirming. That reading directly contradicted §5.4 and §10, both of which describe Shipping as a pure async consumer of `OrderConfirmed` with no synchronous inbound endpoint, publishing `ShipmentDispatched` back independently. `docs/services/README.md`'s gap table tracked this as an open cross-service contradiction, and this service's own docs previously did not flag it as unresolved — one of the two findings that disqualified the prior draft of this spec. **ADR-0002 resolves it**: the integration is async, end to end. `OrderConfirmed` is published as soon as `PaymentCompleted` is received — that is the actual confirmation trigger, not shipment creation — and `ShipmentDispatched` flows back to Order (and to Notification, Delivery Tracking) purely as an informational status update, never as a gate on confirmation. `kart-requirements.md` §12.1 now carries a clarifying prose note under the diagram adopting this reading (the same request/reply notation is used for the unambiguously-async Payment step, so the notation itself never carried sync/async meaning — only causal order). This spec adopts ADR-0002's resolution rather than the diagram's literal notation.
- Success flow (BRD §12.1, read per ADR-0002's resolution above — **not** the diagram's literal step order): reserve stock → charge payment → mark `OrderConfirmed` (published as soon as `PaymentCompleted` is received, not gated on shipment creation) → Shipping asynchronously consumes `OrderConfirmed` and creates the shipment, publishing `ShipmentDispatched` back to Order as an informational status update.
- Failure/compensation flow (BRD §12.2, the only compensation path the BRD documents): if `PaymentFailed` arrives after inventory was reserved, release the inventory reservation as a compensating action, then mark `OrderCancelled`.
- Refunds are explicitly **not** modeled as a continuation of the order Saga: BRD §12 notes refund-after-dispute is "tracked as its own saga instance rather than reusing the order saga state machine, since refunds can be partial and asynchronous relative to the original order lifecycle." Order does not own refund orchestration.
- **Terminal delivery event — resolved BRD self-contradiction (ADR-0005).** BRD §5.4 originally named two different terminal-order events with no publisher anywhere in the BRD: Recommendation's row consumed `OrderCompleted`, Review's row consumed `OrderDelivered`, and Order's own Publishes list (§5.1, §10, pre-ADR) contained neither. `docs/services/README.md`'s gap table tracked this contradiction; this service's own prior draft silently omitted `OrderDelivered` entirely rather than flagging it — the second of the two findings that disqualified that draft. **ADR-0005 unifies both names into one event, `OrderDelivered`, published by Order** — chosen over `OrderCompleted` because it matches Order's own state-machine vocabulary exactly (`Delivered` is already a named state; `Completed` is not). Order publishes `OrderDelivered` when its state machine transitions to `Delivered`, triggered by consuming Delivery Tracking Service's `DeliveryStatusUpdated` event and detecting its terminal "delivered" status value (BRD §5.1, §10 now list this consume/publish pair directly, both citing ADR-0005). This adds a **fourth dependency Order did not previously have**: Delivery Tracking Service, alongside Inventory/Payment/Shipping — distinct from those three in that it is not part of the compensable forward transaction; there is no compensation path for a delivery-status event, since it only fires after the order has already reached `Shipped` successfully. **State-machine implication**: `Shipped → Delivered` is the one lifecycle transition Order does not actively drive itself — there is no client-facing API and no Order-Saga-internal trigger for it. For this transition only, Order is a passive listener for `DeliveryStatusUpdated`'s terminal value, and that consumption is the *sole* trigger for both the `Delivered` transition and the `OrderDelivered` publish.

### Persistence & Messaging
- Write side: PostgreSQL tables `orders`, `order_items`, `order_events` (BRD §5.1), range-partitioned by month on `orders`/`order_events` (BRD §6.1) for bounded index size and archival.
- Outbox Pattern (BRD §11, §7): the order state transition and its outbox row must be written in the same DB transaction; a poller publishes from the outbox to RabbitMQ, marking rows published only after broker ack.
- Publish: `OrderCreated`, `OrderConfirmed`, `OrderCancelled`, `OrderCompensationTriggered`, `OrderDelivered` (BRD §5.1; `OrderDelivered` added per ADR-0005 — see "Terminal delivery event" above).
- Consume: `InventoryReserved`, `InventoryReservationFailed`, `PaymentCompleted`, `PaymentFailed`, `ShipmentDispatched`, `DeliveryStatusUpdated` (terminal "delivered" status value only) (BRD §5.1; `DeliveryStatusUpdated` added per ADR-0005 — see "Terminal delivery event" above).

## 4. Domain Invariants

- **Sole orchestrator invariant**: no service other than Order may drive a cross-service business transaction (BRD §5.1 Boundary Rationale). This is an architectural invariant, not just a functional convenience — it is the entire reason Order exists as a separate service from Inventory/Payment/Shipping. Order's consumption of Delivery Tracking's `DeliveryStatusUpdated` (ADR-0005) does not weaken this invariant: Order still owns the state machine and is the only writer of `orders`/`order_events`; Delivery Tracking never calls Order or writes Order's state directly, it only emits a status event Order chooses to react to.
- **Compensation completeness**: if any Saga step fails, all previously-succeeded steps for that order must be fully compensated before the order is marked in a terminal failure state — an order must never end up `Cancelled` while money remains charged or stock remains held (inferred from BRD §2.2's Payment "must never double-charge" and Inventory "must never oversell" rules, which Order's Saga is the mechanism enforcing across service boundaries).
- **Legal state transitions only**: the order state machine (`Created → Reserved → Paid → Shipped → Delivered → Cancelled/Refunded`, BRD §5.1) must only advance along transitions the domain actually permits — though the BRD states the happy-path sequence, it does not enumerate which states can branch to `Cancelled` or `Refunded` (see Open Questions); until that's resolved, the invariant is stated at the level the BRD supports: no transition may skip states or move backward except into the two terminal failure states. The one transition that **is** now fully specified is `Shipped → Delivered`: per ADR-0005 it is driven solely by consuming `DeliveryStatusUpdated`'s terminal status value, with no other trigger — the `Cancelled`/`Refunded` reachability questions below are unaffected by this and remain open.
- **Idempotent event consumption**: duplicate delivery of `InventoryReserved`, `PaymentCompleted`, `PaymentFailed`, `ShipmentDispatched`, or `DeliveryStatusUpdated` must not double-advance the saga state (BRD §3 Reliability row: "at-least-once delivery + idempotent consumers"; BRD §13 Failure Simulation explicitly lists "Duplicate Message" as a scenario with the recovery "Idempotency key check at consumer — second delivery is a no-op"). `DeliveryStatusUpdated` is added to this invariant's scope now that Order consumes it (ADR-0005) — a duplicate or replayed terminal-status delivery must not re-publish a second `OrderDelivered` for the same order.
- **Atomic state-transition/event emission**: an order's state transition and the corresponding outbound event must commit atomically in the same database transaction (BRD §11 Outbox Pattern) — the "dual write problem" the Outbox Pattern exists to prevent applies most acutely to Order, since a lost `OrderCreated` stalls the entire downstream Saga.

## 6. Open Questions / Flagged Ambiguities

**Resolved since the prior draft.** The prior draft's Open Question #1 — `OrderCompensationTriggered` missing from the Event Catalog (§5.1/§5.2 named it as published/consumed, but §10 had no row for it) — is resolved by **ADR-0007** (Event Catalog Completeness), which added a full row: consumers Inventory/Notification/Analytics, 3x retry, `order.dlq`. See §3's Retry/DLQ row and §5's API Surface table. Removed from the list below rather than carried forward as open; remaining genuinely open items renumbered:

1. **Compensation flow only covers one failure ordering.** BRD §12.2 documents exactly one compensation path: Payment fails after Inventory is reserved → release inventory → cancel. It does not document what happens when a *later* step fails after an *earlier* money-moving step already succeeded — e.g., Payment succeeds but Shipping subsequently fails or times out. Does Order refund first or release inventory first? This is the single most safety-critical undefined behavior in the whole spec, since it involves unwinding a completed charge, not just a hold. See `edge-cases.md` for the analysis and a proposed default; this still needs human sign-off since the BRD gives no basis to derive the ordering.
2. **State-machine edge transitions not enumerated.** BRD §5.1 gives the FSM as a single line (`Created → Reserved → Paid → Shipped → Delivered → Cancelled/Refunded`) without specifying: which states `Cancelled` is reachable from (pre-`Paid` only, or also `Paid`/`Shipped` requiring a refund first?); whether `Refunded` is reachable only from `Delivered`, or from any post-`Paid` state; whether `Cancelled` and `Refunded` are mutually exclusive terminal states or `Refunded` always implies a prior `Cancelled`. (The `Shipped → Delivered` edge itself is now resolved — see ADR-0005 and Domain Invariants above; this question is only about the `Cancelled`/`Refunded` edges.) Needed before the DDD Agent can model the `Order` aggregate's transition guard logic.
3. **No documented idempotency mechanism on `POST /orders`.** BRD §24 mandates "mandatory idempotency keys on all money-moving POSTs," and Payment's `/payments/charge` explicitly requires one (§5.3, backed by a unique constraint per §6.1), but Order's own `POST /orders` (§5.1) carries no such requirement despite being the entry point to the same money-moving Saga and sitting on the 99.99%-availability path. Duplicate order submission (double-click, client retry after timeout) is a direct, obvious consequence of this gap — see `edge-cases.md`. Carried to the API Design Agent.
4. **Partial fulfillment / split shipment not addressed.** BRD's Order responsibility, database schema, and event catalog all assume exactly one shipment per order (`ShipmentDispatched` is singular; no per-item shipment state). There is no mention anywhere in the BRD of partial fulfillment, backordering a subset of line items, or an order producing multiple shipments. If the platform needs this later, the state machine, the Saga, and the event contracts all change materially — this should be confirmed in or out of scope explicitly rather than assumed.
5. **`order_events` table's relationship to the Outbox table is unclear.** BRD §5.1 lists Order's tables as `orders`, `order_items`, `order_events` — no table named "outbox." But BRD §7's CQRS sequence shows Order doing "INSERT order + INSERT outbox row (same TX)," and §11 defines a generic `outbox` table schema. Is `order_events` actually serving double duty as the Outbox table (audit trail and transactional relay combined), or is there an unlisted, separate outbox table? This materially affects the DDD/data model and should be confirmed before schema design.
6. **No per-step Saga timeout budget.** The BRD gives an overall write-path latency target (P95 < 300ms) and an overall availability target (99.99%), but never states how long Order should wait on Inventory's synchronous reserve call, or on the asynchronous `PaymentCompleted`/`ShipmentDispatched` events, before treating a step as failed/stuck. This is a prerequisite for both the compensation-ordering question (#1) and the timeout-vs-eventual-success race documented in `edge-cases.md`. Left to the Architecture Agent, but flagged here since it blocks a genuinely undecided piece of Saga design, not an implementation detail.
7. **Retry/DLQ tier for Order's own events may be inconsistent with its availability tier.** Order's published events get 3x exponential retry to `order.dlq` with only a "manual replay tool" (§10) — not the 5x + paged-on-call tier the BRD reserves for `Payment*` events. Yet Order is the one service the BRD's Availability NFR row names directly ("99.99%, order path," §3). Is it intentional that a stuck `OrderCreated` (which blocks Payment from ever being invoked) gets a lower operational-response tier than a stuck `PaymentCompleted`? Carried to the Event Design Agent to confirm or override.

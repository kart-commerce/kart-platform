# DDD Model: kart-order-service

One aggregate root, checked against the transaction-boundary test (`docs/standards/ddd-cqrs-standards.md`: "if two things can't be saved in one transaction, they are two aggregates, not one"). Order's state transition, its line items, and its own outbox/audit row must always commit atomically in one transaction (BRD §11 Outbox Pattern; requirement-spec §4 "Atomic state-transition/event emission") — that single fact is what keeps this a one-aggregate bounded context rather than several.

## Aggregate: Order

**Entity (root):** `Order` — identified by `OrderId`. The single authoritative record of an order's lifecycle/Saga state; no other service, and no in-memory Saga-runner state, is ever more authoritative (requirement-spec §4 "Sole orchestrator invariant").

**Child entity:** `OrderLineItem` — one per `(sku, qty, unitPrice)` line, always created and persisted in the same transaction as its parent `Order` (BRD §5.1 `order_items` table) — never independently addressable or mutable after order creation (no partial-fulfillment/split-shipment support, requirement-spec Open Questions resolution #4).

**Child entity:** `OrderEvent` — one append-only row per state transition, doubling as both the audit trail and the Outbox relay row (requirement-spec Open Questions resolution #5: `order_events` is the Outbox table, not a second unlisted one). Always inserted in the same transaction as the `Order` status write it records — this is the concrete mechanism the transaction-boundary test is built around for this aggregate.

**Value objects:**
- `Money` — `{ amount, currency }`, the shape of `OrderLineItem.unitPrice` and `Order.totalAmount` (BRD §6.1's `orders.total_amount`).
- `OrderStatus` — `Created | Reserved | Paid | Shipped | Delivered | FulfillmentException | Cancelled | Refunded`. The legal-transition graph (see Invariants below) is enforced entirely inside this value object's own transition function, never left to caller discipline.
- `IdempotencyKey` — the client-supplied `Idempotency-Key` header value on `POST /orders`, unique per `Order` (requirement-spec Open Questions resolution #3). **Modeling contrast with `kart-payment-service`:** Payment needed a *separate* `IdempotencyRecord` aggregate specifically because an external, non-transactional gateway call sits between reserving the key and confirming the outcome. Order has no such external call on its creation path — `POST /orders` reserves the key and commits the order in one single transaction — so `IdempotencyKey` is correctly a value object directly on `Order` (a unique-constrained column), not a second aggregate. This is the same transaction-boundary test applied to the opposite conclusion Payment reached, for the opposite reason.
- `OrderEventSequence` — a per-order monotonic integer stamped on every `OrderEvent`, letting Saga-critical consumers detect and buffer out-of-order delivery (edge-cases.md, "Outbox Publish Failure/Reordering After DB Commit").

**Invariants:**
- **Legal transitions only** — the complete graph, matching requirement-spec §4 exactly:
  - `Created → Reserved` (on consuming `InventoryReserved`)
  - `Reserved → Paid` (on consuming `PaymentCompleted`) — this transition is what publishes `OrderConfirmed` (ADR-0002); the state name (`Paid`, BRD vocabulary) and the event name (`OrderConfirmed`, BRD vocabulary) deliberately differ — same transition, two different audiences (internal state vs. external signal).
  - `Paid → Shipped` (on consuming `ShipmentDispatched`)
  - `Shipped → Delivered` (on consuming `DeliveryStatusUpdated`'s terminal value only, ADR-0005) — the one transition Order never actively drives itself.
  - `Paid → FulfillmentException` (on consuming `ShipmentCreationFailed`, **or** on the reconciliation sweep's Shipping-await threshold firing with no such event ever received, ADR-0015) — the one non-terminal hold state; see design-decisions.md. A silent Shipping timeout and an explicit failure signal converge on the same state, never on automatic compensation.
  - `FulfillmentException → Paid` (manual ops-triggered retry) or `FulfillmentException → Cancelled` (manual ops action: conditional idempotent Inventory release, then the Payment refund call, then the terminal write) — the only two legal exits.
  - `{Created, Reserved, Paid} → Cancelled` (client-initiated `POST /orders/{id}/cancel`, or automatic pre-confirmation compensation on `PaymentFailed`) — illegal from `Shipped` onward.
  - `{Paid, Shipped, Delivered, FulfillmentException} → Refunded` (on consuming `ChargebackReceived`, ADR-0012 — reachable from **any** state `Paid` onward, including the non-terminal hold state, since a bank-initiated dispute is not gated on Order's own internal state) **or** `Delivered → Refunded` (via the external refund saga reporting back, BRD §12) — the two non-overlapping paths to this terminal state (design-decisions.md's "Chargeback Reaction Mechanism").
  - Any transition attempted outside this graph (e.g., a duplicate/stale/out-of-order event implying a different one) is rejected as a no-op, never silently applied.
- **Optimistic concurrency (compare-and-swap on `OrderStatus`)** — every transition's write is `UPDATE orders SET status = $new WHERE order_id = $id AND status = $expected`; zero rows affected means a concurrent writer already moved the order, and the caller (Saga step handler or the cancel endpoint) retries or rejects rather than blocking on a lock (design-decisions.md, "Concurrency Control for Order State Transitions"). This is the same mechanism that makes idempotent event consumption possible without a dedicated dedup table for most events (see below).
- **Idempotent creation** — `IdempotencyKey` is unique per `Order`; a retried `POST /orders` with a previously-seen key returns the original order's representation, never creates a second row (requirement-spec Open Questions resolution #3).
- **Idempotent event consumption** — a duplicate/redelivered `InventoryReserved`, `PaymentCompleted`, `PaymentFailed`, `ShipmentDispatched`, `ShipmentCreationFailed`, or `ChargebackReceived` against an `Order` no longer in the expected pre-state is a no-op by construction (the compare-and-swap above rejects it); `DeliveryStatusUpdated`'s terminal value against an already-`Delivered` order, and `ChargebackReceived` against an already-`Refunded` order, are both explicit state-guarded no-ops (requirement-spec §4) rather than relying on the CAS rejection alone, since both are terminal-destination transitions with no other writer.
- **Compensation completeness** — an `Order` must never reach `Cancelled` while money remains charged or stock remains held: pre-confirmation compensation (Inventory release, then mark `Cancelled`) and `FulfillmentException`-resolution-to-`Cancelled` (conditional idempotent Inventory release, then the Payment refund, then mark `Cancelled`) are both gated sequences where the terminal write only happens once the prior step(s) complete (requirement-spec §4).
- **Atomic state-transition/event emission** — every `Order`/`OrderStatus` write and its corresponding `OrderEvent` row commit in the same database transaction, never independently (BRD §11 Outbox Pattern; requirement-spec §4).
- **Monotonic event ordering per aggregate** — `OrderEvent.sequence` strictly increases per `orderId`; the Outbox poller publishes in `created_at` order per the existing index (edge-cases.md).
- **CanRead/CanWrite/CanDelete ownership (BRD §24.1.2 — service-owned, decided here rather than left implicit now that it is a named platform requirement).** Order is the resource owner of its own aggregate, and the BRD's own §24.1.2 worked table already names this exact service's mechanism directly: **`CanWrite` gated by the order's own current `OrderStatus` (an inline state-machine guard clause) — the same "Legal transitions only" invariant above, read as a permission rule rather than only a state-machine rule** — and **`CanDelete` is never exposed to any principal**, including Admin: an order is never hard-deleted, only advanced through the state graph to a terminal `Cancelled`/`Refunded` status. `CanRead` is a plain ownership comparison (`orders.user_id == token.sub`) for the owning Customer, widened to Support Agent/Admin per their coarse role (§24.1.1) for order-support lookups. Beyond the owning Customer's own narrow `POST /orders/{id}/cancel` (itself legal only pre-`Shipped`, per the Legal Transitions invariant above — the ownership check and the state-guard are both required, neither alone is sufficient), every other `CanWrite` path onto `Order` is a Saga-step event consumer (an internal, non-interactively-authenticated context, not a caller-facing permission) or Admin Service's client-credentials principal on `POST /orders/{id}/resolve-fulfillment-exception` (requirement-spec §5) — never a second customer-facing ownership path.
- **Audit-actor invariant (BRD §24.3).** Every mutation to `orders`, `order_items`, or `order_events` (database-design.md) stamps `created_by`/`updated_by` with the resolved acting principal: the owning `userId` itself for `POST /orders` creation and the client-initiated `POST /orders/{id}/cancel` write; Admin Service's client-credentials principal for `POST /orders/{id}/resolve-fulfillment-exception`; or a well-known `system:*` principal id for every Saga-step transition driven by an event consumer with no human/API caller — e.g. `system:order-saga-inventory-consumer` (`Created → Reserved`), `system:order-saga-payment-consumer` (`Reserved → Paid`, `PaymentFailed` compensation), `system:order-saga-shipping-consumer` (`Paid → Shipped`, `ShipmentCreationFailed` → `FulfillmentException`), `system:order-saga-delivery-consumer` (`Shipped → Delivered`), `system:order-saga-chargeback-consumer` (`ChargebackReceived` → `Refunded`), and `system:order-reconciliation-sweep` for the Orphaned/Stuck Saga sweep's own timeout-driven transitions. These columns are auto-injected by the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor (§24.3 — one platform-wide implementation referenced as a NuGet dependency, not built locally by this service; see database-design.md) and are never populated from a client-supplied request value, and never `NULL`.

**Domain events (published — unchanged from the BRD's original five; neither new consumed event adds a new published one, see Modeling Decision #4 below):**
- `OrderCreated` — fired on `Created` (i.e., on the initial commit of `POST /orders`).
- `OrderConfirmed` — fired on the `Reserved → Paid` transition (ADR-0002: as soon as `PaymentCompleted` is received, not gated on shipment creation).
- `OrderCancelled` — fired once the order actually reaches `Cancelled` (the terminal write, after any gated compensation step ahead of it has completed).
- `OrderCompensationTriggered` — fired at the *start* of a compensation sequence, before its terminal write — i.e., when Inventory release is initiated, whether the trigger is pre-confirmation `PaymentFailed`, a client-initiated cancel, or `FulfillmentException` resolving to `Cancelled` (Modeling Decision #3 below), and also when a `ChargebackReceived` reaction initiates its own (idempotent) Inventory-release attempt.
- `OrderDelivered` — fired on the `Shipped → Delivered` transition (ADR-0005).

## Modeling Decisions & Assumptions (resolved here, not escalated — engineering defaults, revisable)

1. **`FulfillmentException` is a value within `OrderStatus`, not a separate aggregate or a boolean flag layered on top of `Shipped`.** It is a genuine, distinguishable lifecycle position (BRD's own state machine, extended per ADR-0015) with its own legal entry/exit edges — modeling it as a flag would let it co-exist with an arbitrary other status, reopening exactly the "which transitions are legal" ambiguity the Legal Transitions invariant exists to close.
2. **`IdempotencyKey` lives directly on `Order`, not in a separate ledger aggregate** — see the Value Objects entry above for the full contrast against `kart-payment-service`'s `IdempotencyRecord`. This is the same DDD test (`ddd-cqrs-standards.md`) applied and reaching the opposite conclusion for a structurally different reason (no external call in between).
3. **`OrderCompensationTriggered` is reused, not duplicated, across all three compensation-initiating triggers** (pre-confirmation `PaymentFailed`, client-initiated cancel, `FulfillmentException`-to-`Cancelled`) — edge-cases.md's "Client Cancel Request Racing an In-Flight Saga" decision already establishes that client cancel routes through the same state machine as any other trigger (single writer); extending the same published-event semantics to `FulfillmentException`'s cancel exit keeps one mechanism instead of inventing a second "cancellation started" event for a case that is functionally identical from every consumer's perspective (Inventory still just needs to attempt a release).
4. **Neither new consumed event (`ChargebackReceived`, `ShipmentCreationFailed`) required a new published event.** Both ADR-0012 and ADR-0015 already name Notification and Analytics as direct consumers of the event *at its source* (Payment, Shipping respectively) — Order reacting internally (state transition, possible Inventory release) does not need to re-signal Notification/Analytics a second time via a new Order-published event. This was verified against both publishing services' own approved `event-contract.md`/`architecture.md` before concluding no gap exists.
5. **No separate `Saga` aggregate.** `Order` *is* the Saga instance — its `OrderStatus` value **is** the Saga's current state, and every Saga-orchestration decision in requirement-spec/edge-cases.md (timeouts, compensation ordering, reconciliation) is phrased in terms of an order's own status, never a separate process-manager record. Introducing a second aggregate to track "the saga" would duplicate `Order`'s own state for no additional invariant it doesn't already enforce.
6. **The Orphaned/Stuck Saga reconciliation sweep (edge-cases.md) is an application service, not a domain aggregate** — it reads `Order` rows past their per-step staleness threshold and drives the same compensation transitions any other trigger would, through the same `Order` aggregate; it introduces no new entity or invariant of its own.

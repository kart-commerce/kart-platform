## Epic: kart-notification-service v1

Platform's broadest single fan-in consumer (ADR-0003) and consumer-only, no-public-API service (`api-contract.yaml`, `architecture.md` Boundary Rationale) — every ticket below is either a consumer trigger, a shared mechanism the consumers all call into, or infrastructure/schema foundation. Two aggregates (`NotificationAttempt`, `NotificationPreference`), two non-aggregate lookup projections (`OrderUserIndex`, `TrackingOrderIndex`, ADR-0020), four PostgreSQL tables, 13 consumed events across 3 criticality tiers each with its own dedicated DLQ, one published event (`NotificationSent`). No API-surface tickets exist in this list — `api-contract.yaml` explicitly documents the absence of any public or internal-ops endpoint, and this is not scope this stage may invent. `ddd-model.md`'s Modeling Decision #10 ("`NotificationAttempt`'s creation path is deliberately producer-agnostic across ADR-0003's entire scoped set... one bounded context with one aggregate-creation path, not six-or-seven producer-specific variants") directly shapes the grouping below: consumer tickets are grouped by criticality tier and shared resolution shape, not split one-per-event, since the underlying processing logic is uniform.

## Flagged Gaps — Not Decomposed Into Tickets (would invent scope, not decompose it)

- **No ops/audit query API.** `api-contract.yaml` affirmatively documents the absence of any public *or* internal-ops endpoint for this service — including the one candidate that could have warranted one (a preference-setting API, explicitly rejected in `architecture.md`'s "Resolved Integration-Contract Question" in favor of `UserNotificationPreferenceUpdated`). `idx_notification_attempts_user_audit` and `idx_notification_attempts_failed` exist in NOTIF-1 to support future internal tooling, but no ticket is created to expose them over an endpoint — that would be new Requirement Agent scope, not a decomposition of what's already approved.
- **`UserDataErased` (GDPR erasure) consumption.** `database-design.md`'s GDPR Erasure section flags, as a non-blocking discrepancy against ADR-0016's introductory framing, that neither `notification_attempts` nor `notification_preferences` holds any directly-identifying field to redact — `user_id` is retained purely as an opaque referential key, exactly what ADR-0016 already permits. No ticket needed unless a future requirements pass adds a PII-bearing field to either table.
- **Retention/archival job for `notification_attempts`'s hash partitions.** `database-design.md`'s Partitioning Rationale explicitly flags this as a non-blocking future gap — hash partitions cannot be dropped wholesale the way time-ranged partitions can, but no retention/TTL policy is stated anywhere upstream for this table to begin with. No ticket until a concrete retention decision exists to build against.
- **`kart-user-service`'s own `UserNotificationPreferenceUpdated` publish-side work.** `architecture.md`'s "Resolved Integration-Contract Question" flags this as a new responsibility for `kart-user-service`'s own next docs pass (parallel to how ADR-0006 required Identity to add `UserAccountUpdated`) — not this service's backlog.

## Notes for Sprint Planner Agent

- NOTIF-1 and NOTIF-2 are the two true root dependencies — pure schema and pure topology, no dependency on each other — can run fully in parallel in sprint 1.
- NOTIF-3 depends only on NOTIF-1 and is the shared uniform pipeline every one of NOTIF-5 through NOTIF-9's consumer tickets calls into (`ddd-model.md` Modeling Decision #10) — sequence it immediately after NOTIF-1, before any consumer ticket.
- NOTIF-4 depends on NOTIF-1 and NOTIF-3 and must land before NOTIF-5, NOTIF-6, or NOTIF-7 (the three consumer tickets whose triggering events don't carry `userId` directly and need the index-resolution mechanism). NOTIF-8 and NOTIF-9 skip NOTIF-4 entirely — `UserRegistered` and `WishlistPriceAlertTriggered` both carry `userId` directly, and `UserNotificationPreferenceUpdated` doesn't touch `NotificationAttempt`/the index tables at all — so NOTIF-8/NOTIF-9 can be built in parallel with NOTIF-4/5/6/7 once NOTIF-2 and NOTIF-3 land.
- NOTIF-5, NOTIF-6, and NOTIF-7 are independent of each other (distinct routing keys/DLQs, same shared pipeline underneath) — can be built in parallel once NOTIF-4 lands. Note: although `order_user_index` is only *populated* at runtime by NOTIF-5's `OrderCreated` consumer, ADR-0020's lookup-miss/requeue handling (built in NOTIF-4) makes NOTIF-6/NOTIF-7 correct regardless of code-deploy order — no circular code dependency, though deploying NOTIF-5 first in a given environment avoids an initial wave of transient requeues.
- NOTIF-7 has one internal ordering note of its own, contained within the ticket, not split out: `DeliveryStatusUpdated`'s two-hop lookup depends on `ShipmentDispatched` having already seeded `tracking_order_index` — the same transient-miss/requeue handling applies, so this does not need a separate ticket.
- NOTIF-10 (channel adapters) can be built in parallel with NOTIF-4 through NOTIF-9 — it only depends on NOTIF-3's pipeline producing a `Pending`-status row to act on, not on which specific triggering event created it.
- NOTIF-11 depends on NOTIF-10 (it wraps the physical delivery outcome NOTIF-10 produces) and NOTIF-2 (the ladder queues it requeues onto).
- NOTIF-12 is the final leaf — depends on NOTIF-3 (suppressed outcome), NOTIF-10 (sent outcome), and NOTIF-11 (failed/exhausted outcome), since it publishes only once any one of those three terminal `DeliveryOutcome` values is reached, regardless of which of NOTIF-5 through NOTIF-9 originated the attempt.
- No circular dependencies in this graph. Longest chain: NOTIF-1 → NOTIF-3 → NOTIF-10 → NOTIF-11 → NOTIF-12, 5 deep.

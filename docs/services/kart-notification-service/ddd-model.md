# DDD Model: kart-notification-service

Two aggregate roots, checked against the transaction-boundary test (`docs/standards/ddd-cqrs-standards.md`: "if two things can't be saved in one transaction, they are two aggregates, not one"): `NotificationAttempt` (the per-`(eventId, channel)` send-attempt/idempotency/audit boundary) and `NotificationPreference` (the per-`userId` local opt-out/reachability projection). Neither ever needs to commit in the same transaction as the other — a preference update commits independently of any particular in-flight send attempt (it is triggered by an unrelated upstream event, `UserNotificationPreferenceUpdated`), and a send attempt only ever *reads* the current preference row at decision time, never writes to it. This is the same split test `kart-delivery-tracking-service/ddd-model.md` applies to its own three-way aggregate split.

Architecture.md's two defining characteristics for this service — no public API (consumer-only) and the platform's broadest single fan-in consumer (ADR-0003) — are both reflected below: `NotificationAttempt` has no synchronous creation path at all (only event consumption creates it), and its creation logic is deliberately producer-agnostic across all seven of ADR-0003's scoped triggering events (see Modeling Decision #10).

## Aggregate: NotificationAttempt

**Entity (root):** `NotificationAttempt` — identified by `NotificationAttemptKey`. Represents one channel-specific send attempt derived from a single consumed triggering event; this is both the platform's `(eventId, channel)` idempotency boundary (edge-cases.md's "Duplicate Notification Delivery Under At-Least-Once Redelivery" decision; design-decisions.md's Idempotency Mechanism decision) and the audit record every consumed event must resolve into (requirement-spec §4 Domain Invariant #1: "either a recorded successful delivery, a recorded suppressed-by-opt-out outcome, or an auditable failure entry... never a silent drop").

**Value objects:**
- `NotificationAttemptKey` — `{eventId, channel}`, enforced as a DB-level unique constraint that is itself the atomic idempotency mechanism, not merely a pre-check (see Modeling Decision #2 for how this closes, not just narrows, the residual race requirement-spec Open Question 4 explicitly deferred to this stage).
- `Channel` — `Email | SMS | Push` (BRD §2.1 item 15; requirement-spec §6 Q2's channel-selection rule).
- `TriggeringEventType` — the name of the consumed event that caused this attempt to exist (`OrderConfirmed`, `PaymentFailed`, etc.) — a reference-only value, never redefining the producing service's own event; carried through because design-decisions.md's Retry & DLQ decision requires the send pipeline to "carry/derive the triggering event's type... through to the retry logic."
- `CriticalityTier` — `{Tier1: 5 attempts + paged, Tier2: 3 attempts non-paged, Tier3: 2 attempts non-paged}`, snapshotted onto the attempt at creation time from `TriggeringEventType` via a static config mapping (requirement-spec §6 Q7 — inherits the triggering event's own already-catalogued BRD §10 tier verbatim, no new tiering scheme invented). See Modeling Decision #3 for why it is snapshotted rather than re-derived per retry.
- `NotificationCategory` — the per-category classification (e.g. order-updates, payment, shipping, marketing/price-alerts, account) an opt-out check is evaluated against; derived from `TriggeringEventType` via a static config mapping — an engineering default, not a BRD-enumerated taxonomy (see Modeling Decision #4).
- `DeliveryOutcome` — `Pending | Sent | Failed | Suppressed`. `Sent`/`Failed`/`Suppressed` are all terminal (mirrors the monotonic terminal-outcome shape `kart-shipping-service/ddd-model.md`'s `ShipmentStatus` and `kart-payment-service/ddd-model.md`'s `PaymentIntentStatus` both use for outcomes confirmed after the fact) and maps directly onto `NotificationSent`'s own `status` payload field values (requirement-spec §10: `sent`/`failed`/`suppressed`).
- `AttemptCount` — the number of physical delivery tries made so far against the TTL-ladder retry ceiling `CriticalityTier` caps it at; incremented in place on the same row, never modeled as a child entity per try (see Modeling Decision #5).

**Fields:** `eventId`, `channel`, `userId` (reference only — owned by `kart-identity-service`; resolved via `OrderUserIndex`/`TrackingOrderIndex` for the eight/nine triggering events that do not carry it directly in their own payload — see ADR-0020 and Modeling Decision #11), `triggeringEventType`, `criticalityTier`, `category`, `status: DeliveryOutcome`, `attemptCount`, `suppressedReason` (nullable, populated only when `status = Suppressed`), `createdAt`, `lastAttemptAt`.

**Invariants:**
- **Uniqueness — at most one `NotificationAttempt` per `(eventId, channel)`.** The row's own insert *is* the idempotency check (Modeling Decision #2), not a separate pre-existing-row lookup — this is what closes the residual race requirement-spec Open Question 4 flagged and design-decisions.md's Idempotency Mechanism decision explicitly deferred to this stage ("a redelivery arriving before the original attempt's row commits can still race past the check... fully closing it is left to the Architecture/DDD Agent stage").
- **Opt-out is checked before any physical delivery try, every time, with no exception by criticality tier.** A `NotificationAttempt` whose `(channel, category)` pair is opted out in the caller's current `NotificationPreference` reaches `status = Suppressed` immediately, with `attemptCount` remaining `0` — never `Pending` first, never attempted even once (requirement-spec §4 Domain Invariant: "must never dispatch... regardless of the triggering event's own criticality tier").
- **Monotonic, terminal `DeliveryOutcome` — `Pending → {Sent, Failed, Suppressed}` only, and only once.** Once a row reaches any terminal value, no further transition or duplicate delivery is attempted; a redelivered triggering event maps to the existing terminal row via the uniqueness invariant above and is a no-op.
- **`AttemptCount` never exceeds `CriticalityTier`'s configured retry budget.** The TTL-ladder mechanism (design-decisions.md's Retry & DLQ decision) retries within this ceiling only (5 for Tier 1, 3 for Tier 2, 2 for Tier 3); exhausting it moves `status` to `Failed` and the row's terminal-failure signal lands in `notification.dlq` — a distinct mechanism from `NotificationSent`'s own separate 1x, fire-and-forget audit-publish tier (requirement-spec §6 Q6; not to be confused with each other).
- **`NotificationSent` is published exactly once per `NotificationAttempt`, only once `status` reaches a terminal value, and its `status` payload field always mirrors the row's own terminal `DeliveryOutcome` exactly** (requirement-spec Domain Invariant #2) — never published while `status = Pending`, and never diverging from the row's own recorded outcome.
- **`CriticalityTier` is fixed at creation time and never re-derived mid-flight**, even if the underlying event-type-to-tier config mapping changes later (Modeling Decision #3) — an attempt's retry budget is stable for its own lifetime, once assigned.
- **CanRead/CanWrite/CanDelete ownership (BRD §24.1.2 — service-owned, decided here rather than left implicit now that it is a named platform requirement) does not apply in the same shape as a client-facing aggregate.** `NotificationAttempt` has no synchronous creation or mutation path of any kind (architecture.md's Boundary Rationale: "no public API... consumer only") — every row is created and transitioned exclusively by consuming one of ADR-0003's scoped triggering events, never by a request a caller's own `CanRead`/`CanWrite`/`CanDelete` claim could be evaluated against. There is no principal — not even the row's own `userId` — that ever directly reads or writes this aggregate through an API; it exists purely as this service's own internal audit/idempotency record. `CanDelete` is never exposed to any principal, including the referenced `userId`, through any code path this service defines.
- **Audit-actor invariant (BRD §24.3).** Every mutation to `notification_attempts` (database-design.md) stamps `created_by`/`updated_by` with a well-known `system:*` principal id — always `system:notification-send-pipeline`, since no human/API caller ever writes this table (the invariant above) — never `NULL`, auto-injected by the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor (§24.3), the same package `order_user_index`/`tracking_order_index` (below) and `NotificationPreference` (below) use.

**Domain events (consumed — triggering, per ADR-0003's resolved scope; each creates one or more `NotificationAttempt` rows per the channel-selection rule, requirement-spec §6 Q2 — see Modeling Decision #6):**
- `OrderCreated` — owned by `kart-order-service`; **Tier 1** (elevated from Tier 2, ADR-0020 — it is now also the sole seed for `OrderUserIndex`, the resolution linchpin for the seven other `orderId`-keyed events below).
- `OrderConfirmed`, `OrderCancelled`, `OrderCompensationTriggered`, `OrderDelivered` — owned by `kart-order-service`; Tier 2. `userId` resolved via `OrderUserIndex` (ADR-0020) — none of these carry `userId` in their own payload.
- `PaymentCompleted`, `PaymentFailed`, `RefundIssued` — owned by `kart-payment-service`; Tier 1. `userId` resolved via `OrderUserIndex` (ADR-0020).
- `ShipmentDispatched` — owned by `kart-shipping-service`; Tier 2. `userId` resolved via `OrderUserIndex` (ADR-0020); also seeds `TrackingOrderIndex`.
- `DeliveryStatusUpdated` — owned by `kart-delivery-tracking-service`; Tier 2. `orderId` resolved via `TrackingOrderIndex`, then `userId` via `OrderUserIndex` (ADR-0020) — the only triggering event carrying neither field directly.
- `WishlistPriceAlertTriggered` — owned by `kart-wishlist-service`; Tier 3. Carries `userId` directly.
- `UserRegistered` — owned by `kart-identity-service`; Tier 2. Carries `userId` directly.

**Domain events (published):**
- `NotificationSent` — existing, BRD §10. Payload `userId, channel, status`. Consumed by Analytics. 1x, fire-and-forget audit-publish tier, distinct from and unaffected by the tiered send-attempt retries above (requirement-spec §6 Q6).

## Lookup Projections: OrderUserIndex, TrackingOrderIndex (ADR-0020)

Not aggregate roots — neither has its own invariants, lifecycle, or transaction boundary beyond a plain upsert-on-consume; both exist solely to resolve `userId` for the eight of thirteen consumed triggering events whose own payload does not carry it (checked directly against each publisher's own approved `event-contract.md`, not assumed — see ADR-0020's Context).

- **`OrderUserIndex`** — `orderId -> userId`. Upserted on every consumed `OrderCreated` (which already carries both fields as part of its own Tier 2 scope, ADR-0003). Read (never written) by the creation path for `OrderConfirmed`, `OrderCancelled`, `OrderCompensationTriggered`, `OrderDelivered`, `PaymentCompleted`, `PaymentFailed`, `RefundIssued`, and `ShipmentDispatched` — each resolves `userId` from this index instead of its own payload.
- **`TrackingOrderIndex`** — `trackingId -> orderId`. Upserted on every consumed `ShipmentDispatched` (which already carries both `orderId` and `trackingId`). Read by `DeliveryStatusUpdated` (which carries neither `userId` nor `orderId`, only `trackingId`) to chain to `orderId`, then to `OrderUserIndex` for `userId`.
- **Lookup-miss handling:** a miss (the seeding event has not yet been consumed — RabbitMQ gives no cross-routing-key ordering guarantee) is treated as transient, not a permanent failure: the dependent event's own message requeues onto its already-modeled `CriticalityTier` TTL-ladder retry, no separate mechanism invented. If the seeding event (`OrderCreated` or `ShipmentDispatched`) is itself permanently lost, every event chained to it exhausts its own retry budget and lands `Failed` — a correct, already-modeled cascade surfaced through existing DLQ/audit tooling, not a silent drop (ADR-0020).
- Both indexes are retained for the same unbounded lifetime as `NotificationAttempt` (no retention window stated upstream) — see Modeling Decision #11.
- **Audit-actor invariant (BRD §24.3).** Neither projection has an API surface or invariant of its own beyond upsert-on-consume, but both still stamp `created_by`/`updated_by` on every row per the platform-wide mandate: `system:notification-order-user-index-consumer` for `OrderUserIndex` (seeded by `OrderCreated`), `system:notification-tracking-order-index-consumer` for `TrackingOrderIndex` (seeded by `ShipmentDispatched`) — never `NULL`, via the same shared `kart-shared` interceptor as `NotificationAttempt`/`NotificationPreference` above.

## Aggregate: NotificationPreference

**Entity (root):** `NotificationPreference` — identified by `UserId` (reference, owned by `kart-identity-service`). The local, eventually-consistent projection of a user's per-channel-per-category opt-out state and app-installed reachability signal, populated only by consuming `UserNotificationPreferenceUpdated` (owned by `kart-user-service` — see architecture.md's "Resolved Integration-Contract Question"; the event itself is already registered in `docs/ddd/ubiquitous-language.md` under "Owned by `kart-user-service`" and is referenced, not redefined, here).

**Value objects:**
- `OptOutMatrix` — the full `{channel -> {category -> optedOut: bool}}` map; replaced wholesale on every consumed `UserNotificationPreferenceUpdated`, never partially patched (see Modeling Decision #7).
- `AppInstalled` — the boolean reachability flag gating whether `Push` is a candidate channel for a given send (requirement-spec §6 Q2's channel-selection rule); populated from the same event.

**Fields:** `userId`, `optOutMatrix: OptOutMatrix`, `appInstalled: AppInstalled`, `lastUpdatedAt`.

**Invariants:**
- A `NotificationPreference` row is created (or, on redelivery, upserted in place — never duplicated) only by consuming `UserNotificationPreferenceUpdated` for that `userId`; before the first such event, no row exists. Absence of a row defaults to opted-in on every channel/category (fail-open toward reachability, never fail-open toward violating the opt-out invariant) — see Modeling Decision #8 for why this default direction is the only one consistent with the stated invariant.
- Every `NotificationAttempt`'s opt-out check reads this aggregate directly via a synchronous, indexed Postgres query at send time (design-decisions.md's Caching Strategy decision — no cache, no TTL, no in-service staleness window). The only staleness this aggregate can ever carry relative to the user's true current preference is the cross-service event-propagation delay from `kart-user-service` — an accepted, explicitly named trade-off (architecture.md's Distributed-Monolith Risk #1), not an in-service caching gap.
- `UserNotificationPreferenceUpdated` redelivery (at-least-once) is idempotent here by construction: replacing the entire `OptOutMatrix`/`AppInstalled` with the same payload twice produces the same resulting row every time — no separate dedup ledger is needed for this aggregate, unlike `NotificationAttempt`'s dedicated `(eventId, channel)` mechanism (see Modeling Decision #9).
- **CanRead/CanWrite/CanDelete ownership (BRD §24.1.2) does not apply in the same shape as a client-facing aggregate**, for the identical reason as `NotificationAttempt` above: this row's only creation/update trigger is consuming `UserNotificationPreferenceUpdated` (owned by `kart-user-service`) — Notification itself exposes no endpoint through which the referenced `userId`, or any other principal, ever reads or writes this aggregate directly. The opt-out check that *reads* it (the send-attempt path above) is an internal query inside this service's own event-processing pipeline, not a request a caller's permission claim is evaluated against.
- **Audit-actor invariant (BRD §24.3).** Every mutation to `notification_preferences` stamps `created_by`/`updated_by` with the well-known `system:notification-preference-sync-consumer` principal id — the only process that ever writes this table — never `NULL`, auto-injected by the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor (§24.3).

**Domain events (consumed):**
- `UserNotificationPreferenceUpdated` — external, owned by `kart-user-service`; Tier 3 retry, non-paged (architecture.md Dependencies table) — sole creation/update trigger for this aggregate.

**Domain events (published):** None. Purely a locally-owned decision projection, never itself a business event source — mirrors the non-publishing role `kart-delivery-tracking-service/ddd-model.md`'s `WebhookDedupEntry` and `kart-search-service`'s `SearchDocument` both play in their own bounded contexts.

## Modeling Decisions & Assumptions (resolved here, not escalated — engineering defaults, revisable)

1. **Two aggregates, not one (transaction-boundary test applied).** `NotificationAttempt` and `NotificationPreference` are separate aggregate roots because neither is ever required to commit in the same transaction as the other: a preference upsert is triggered entirely independently, by an unrelated upstream event (`UserNotificationPreferenceUpdated`), while a send attempt only reads the current preference row at decision time and never writes back to it. Merging them into one aggregate would force every preference update to somehow coordinate with whichever `NotificationAttempt` rows happen to be in flight at that moment — an invariant nothing upstream asks for and the architecture's own no-synchronous-coupling stance (architecture.md's Boundary Rationale, Distributed-Monolith Risk) explicitly avoids.
2. **Closing (not just narrowing) the residual idempotency race requirement-spec Open Question 4 flagged and explicitly deferred to this stage.** The unique `(eventId, channel)` DB constraint is enforced via insert-first semantics (`INSERT ... ON CONFLICT DO NOTHING` or equivalent), not select-then-insert: the first write for a given key is what creates the row and claims the right to proceed with delivery; a concurrent redelivery's insert attempt against the same key fails/no-ops at the DB layer itself and is treated as "already being handled," rather than racing a separate `SELECT` check against a not-yet-committed row. This directly closes the specific race requirement-spec named ("a redelivery arriving before the original attempt's row commits can still race past the check") — the DB's own unique constraint becomes the atomic serialization point, not an application-level check-then-act sequence. This is the concrete, mechanism-level resolution both `edge-cases.md` and `design-decisions.md` deferred to "the Architecture/DDD Agent stage."
3. **`CriticalityTier` is snapshotted onto the `NotificationAttempt` row at creation time, not re-derived from `TriggeringEventType` on every retry.** design-decisions.md's own stated trade-off ("send pipeline must carry/derive the triggering event's type... through to the retry logic") implies the tier travels with the attempt; snapshotting it prevents an attempt's retry budget from silently changing mid-flight if the event-type-to-tier config mapping is edited while attempts against the old mapping are still riding the TTL-ladder.
4. **`NotificationCategory` (order-updates, payment, shipping, marketing/price-alerts, account, etc.) is derived from `TriggeringEventType` via a static, out-of-band config mapping, not a BRD-enumerated business rule.** No BRD section names an actual category taxonomy for Notification — requirement-spec §6 Q3 gives only one illustrative example ("SMS off for marketing, email on for order updates"). This is treated the same way `kart-shipping-service/ddd-model.md` treats `CarrierSelectionPolicy` and `kart-delivery-tracking-service/ddd-model.md` treats its per-carrier status-mapping/SLA tables: operational/config reference data this aggregate reads from, not a modeled business computation or a second aggregate of its own.
5. **`AttemptCount` is a single incrementing field on `NotificationAttempt`, not a child entity per physical delivery try.** The TTL-ladder retry mechanism (design-decisions.md's Retry & DLQ decision) is RabbitMQ-level infrastructure (parking-lot queues, requeue-with-delay); nothing upstream states that each individual physical retry needs its own durable, independently-queryable audit row — only that the attempt's terminal outcome, and the fact retries happened up to a tier-bounded ceiling, need to be recorded. Modeling every retry as a child entity (the way `kart-delivery-tracking-service` models `StatusHistoryEntry` per received carrier update) would invent an audit granularity nothing in this service's requirement-spec or edge-cases actually asks for.
6. **One triggering event can produce more than one `NotificationAttempt` row — one per channel selected by the channel-selection rule (requirement-spec §6 Q2, restated in architecture.md).** E.g. a `PaymentFailed` event yields both an `Email` and an `SMS` `NotificationAttempt` (Tier 1 urgency escalation), and a `UserRegistered` event yields `Email` plus `Push` if `AppInstalled` is true. The channel-selection rule itself (email always; +SMS for the named time-critical/Tier-1 events; +push when `AppInstalled`) is treated as an engineering-default mapping this aggregate's creation logic reads, not a separate modeled aggregate — the same treatment Decision #4 gives `NotificationCategory`.
7. **`OptOutMatrix` is replaced wholesale on every consumed `UserNotificationPreferenceUpdated`, never patched per-field.** architecture.md's own description of the event's payload ("per-channel-per-category opt-out map") states a complete map, not a delta/diff — modeling this as anything other than a full-replace upsert would require inventing a merge/patch semantic the event's own payload shape doesn't support and no upstream doc asks for.
8. **Absence of a `NotificationPreference` row defaults to opted-in on every channel/category, not opted-out.** This is the only direction that keeps the absolute opt-out invariant ("must never dispatch... to a channel a user has explicitly opted out of") satisfiable without contradiction: a user who has never expressed a preference has not explicitly opted out of anything, so defaulting to reachable is the only reading consistent with the invariant's own "explicitly opted out" language (requirement-spec §4). The alternative (fail-closed / suppress-by-default) would silently withhold notifications the BRD's own functional requirement mandates be sent, with no stated justification for treating "no preference on file yet" as equivalent to "opted out."
9. **`UserNotificationPreferenceUpdated` needs no separate idempotency ledger, unlike `NotificationAttempt`'s `(eventId, channel)` mechanism.** A full-map upsert (Decision #7) is naturally idempotent — replaying the same payload twice against the same `userId` produces the same resulting row every time, with no double-apply side effect to guard against. Contrast with a physical email/SMS/push send, which is not naturally idempotent and is exactly why `NotificationAttempt` needs its own dedicated key.
10. **`NotificationAttempt`'s creation path is deliberately producer-agnostic across ADR-0003's entire scoped set.** A single, uniform creation path handles every triggering event — Order, Payment, Shipping, Delivery Tracking, Wishlist, and Identity — each reduced to the same `{eventId, userId, triggeringEventType, criticalityTier, category}` shape before any channel-selection or opt-out logic runs. This keeps the platform's broadest single fan-in consumer (architecture.md's Boundary Rationale item 2) as one bounded context with one aggregate-creation path, not six-or-seven producer-specific variants — extending ADR-0003's scope to a future producer/event only extends the config mappings in Decisions #3/#4/#6, never this aggregate's own shape or invariants.
11. **`OrderUserIndex`/`TrackingOrderIndex` are lookup projections, not aggregate roots (ADR-0020).** Nine of `NotificationAttempt`'s thirteen triggering events do not carry `userId` in their own publisher-approved payload (checked directly, not assumed — `event-contract.md`'s cross-check). Adding `userId` to eight already-approved, signed-off event contracts across four other services was rejected (ADR-0020) in favor of the same "denormalize what you need" pattern this service already applies to `NotificationPreference` (itself citing ADR-0006's precedent): a local `orderId -> userId` index seeded from `OrderCreated`, and a `trackingId -> orderId` index seeded from `ShipmentDispatched` for the one event (`DeliveryStatusUpdated`) that carries neither `userId` nor `orderId`. Neither index is a full aggregate — neither has invariants beyond upsert-on-consume, and neither is ever the transaction boundary for a business decision the way `NotificationAttempt`/`NotificationPreference` are.

No new domain event was required beyond what requirement-spec/architecture.md already name: `NotificationSent`'s existing `status` field (`sent`/`failed`/`suppressed`) already carries the outcome distinction a separate "suppressed" or "failed" event would otherwise exist to communicate, so none is modeled (contrast with `kart-delivery-tracking-service`'s internal-only `CarrierStatusIngested`/`UnmappedCarrierStatusFlagged`, which exist because that service's external event has no equivalent outcome-carrying field).

# DDD Model: kart-review-service

Two aggregate roots plus one non-aggregate lookup projection, checked against the transaction-boundary test (`docs/standards/ddd-cqrs-standards.md`: "if two things can't be saved in one transaction, they are two aggregates, not one"): `Review` (authorship/moderation lifecycle), `ProductRating` (the ADR-0014 canonical rating aggregate), and `VerifiedPurchaseRecord` (the local materialized eligibility-gate projection). None of the three is ever required to commit atomically with another ã `Review`'s own write only ever reads `VerifiedPurchaseRecord`, never writes it; `ProductRating` is updated asynchronously off `Review`'s own published events, exactly the same eventually-consistent "independent projection of the same event stream" relationship design-decisions.md already describes between Review's canonical aggregate and Product's denormalized copy, just applied one level earlier (inside Review's own bounded context, before it ever leaves the service).

**A payload gap surfaced during this pass, resolved here, not escalated:** architecture.md and design-decisions.md both describe `VerifiedPurchaseRecord` as keyed on `(customerId, orderId, sku)`, populated "by consuming `OrderDelivered`." Checked directly against `kart-order-service/event-contract.md`'s own approved Published table (not assumed) ã `OrderDelivered`'s actual key fields are `orderId, deliveredAt` only; it carries neither `userId` nor `sku`. This is the identical shape of gap ADR-0020 already found and closed for `kart-notification-service` (an event missing a field a consumer needs, resolved by that consumer denormalizing the missing field from an *earlier* event in the same producer's stream that already carries it, rather than by changing the producer's own already-approved payload). The same resolution is applied here: see the `VerifiedPurchaseRecord` section and Modeling Decision 3 below. This adds `OrderCreated` to Review's own consumed-event scope ã a new inbound dependency not currently listed in architecture.md's Dependencies table or in `kart-order-service/event-contract.md`'s `OrderCreated` consumer list (today: Payment, Notification, Analytics) ã flagged explicitly for a downstream update, the same way ADR-0020's own Consequences section flagged `kart-order-service/event-contract.md` needing a Notification addition.

## Aggregate: Review

**Entity (root):** `Review` ã identified by `ReviewId`. The single authoritative write-model record of one customer's opinion of one `(orderId, sku)` pair ã submission, edit, retraction, and moderation outcome all resolve onto this one row (BRD ôÏ5.4: "PostgreSQL ã MongoDB read"; requirement-spec ôÏ2). The MongoDB read model is a query-side projection of this aggregate's own persisted, publicly-visible state ã not itself modeled as a separate aggregate here, the same treatment Order gives its own PostgreSQL-write/denormalized-read split, and deliberately *not* the treatment Search gives `SearchDocument` or Delivery Tracking gives `TrackingRecord` (both of which are the *sole* representation of their own state, with no separate normalized write table behind them ã Review's read model is a secondary denormalization of a source of truth that does have one).

**Value objects:**
- `ReviewId` ã the stable identifier assigned at insert time; referenced by `ReviewSubmitted`'s expanded payload (requirement-spec ôÏ6 Q5) and by `ProductRating`'s own per-review ledger key (see Modeling Decision 4 for why that ledger keys on `(orderId, sku)` rather than raw `reviewId`).
- `Rating` ã a `1..5` integer star value. **Flagged assumption:** no BRD section or approved upstream doc states an explicit rating scale; `1..5` is adopted as the industry-standard engineering default (the same kind of unstated-numeric-default treatment `kart-order-service/ddd-model.md` and others give BRD-silent parameters), revisable without any structural change to this model.
- `ModerationStatus` ã `PendingModeration | Published | Rejected | Retracted`. `Published`/`Rejected`/`Retracted` are not uniformly terminal ã see Invariants below for the exact transition graph, which is narrower than a flat terminal/non-terminal split.
- `PendingRevision` (nullable) ã `{ newBodyText, newRating: Rating, submittedAt }`. Staged content for an edit submitted while the review's *current* content is `Published`, held until the automated filter/moderator clears or rejects it; see Invariants and Modeling Decision 7 for why this ã and only this ã case gets a two-phase staging mechanism, deliberately narrower than the reserve/commit option design-decisions.md already rejected for the *original* submission case.
- `IdempotencyKey` ã the client-supplied `Idempotency-Key` header value on `POST /reviews`/`PATCH /reviews/{id}`, unique per request scope. Lives directly on `Review`, not a separate ledger aggregate ã see Modeling Decision 2 for the three-way contrast against both `kart-order-service` (no external call at all) and `kart-payment-service` (an external call with a costly, non-idempotent side effect) that justifies this placement despite Review *also* having a synchronous external call (the classifier) in its write path.

**Fields:** `reviewId`, `orderId` (reference, `kart-order-service`), `sku` (reference, `kart-product-service`), `userId` (reference, `kart-identity-service` ã the author), `rating: Rating`, `bodyText`, `status: ModerationStatus`, `pendingRevision: PendingRevision` (nullable), `firstPublishedAt` (nullable timestamp ã see Modeling Decision 6), `createdAt`, `lastEditedAt`, `retractedAt` (nullable), `idempotencyKey: IdempotencyKey`.

**Invariants:**
- **Verified-purchase gate.** `POST /reviews` is rejected ã not merely flagged ã unless `VerifiedPurchaseRecord` (below) shows this `(orderId, sku)` pair as delivered to this `userId`; checked synchronously against Review's own local projection only, never a live call to Order (requirement-spec ôÏ4; design-decisions.md's Cross-Service Verified-Purchase Check decision).
- **Uniqueness ã at most one `Review` per `(order_id, sku)`, permanently.** Enforced via a PostgreSQL unique constraint (requirement-spec ôÏ4/ôÏ6 Q6). This occupancy is never released, even once a `Review` reaches a terminal state ã a customer who exhausts the edit path or retracts has no path back to a fresh second submission for the same pair (see Modeling Decision 8 for this consequence, not stated explicitly upstream but implied by combining the uniqueness invariant with the edit-is-the-only-revision-path decision).
- **Idempotent creation.** `IdempotencyKey` collapses a literal client retry (double-click, timeout-retry) to the original review's representation; the `(order_id, sku)` unique constraint is the independent second layer catching a duplicate arriving without a matching key (edge-cases.md "Duplicate review submission," design-decisions.md's Idempotency Mechanism decision ã both mechanisms, not either alone).
- **Edit window.** Editable by its author, unlimited times, within 30 days of `createdAt` (not of the last edit) ã requirement-spec ôÏ4/ôÏ6 Q3. Immutability outside the window applies to the *author's* edit path only; a moderator's accept/reject/takedown action (`PATCH /reviews/{id}/moderate`) is never time-boxed by this window (edge-cases.md's "Abusive content" edge case explicitly names post-hoc takedown via this same action as a mitigation with no stated expiry).
- **At most one `PendingRevision` at a time.** A second edit submitted while one is already staged overwrites it (re-running the classifier against the newest content) rather than queuing a second staged revision ã "unlimited edits within the window" is latest-edit-wins, not a queue.
- **Defer-until-outcome, generalized to edits and takedowns, not only original submissions.** No read-model write, no `ProductRating`-affecting event, ever fires for content that has not actually become ã or ceased to be ã publicly visible (design-decisions.md's own stated generalization: "any side-effect... gated by moderation status"). Concretely:
  - Original submission cleared by the classifier ã immediate single-transaction insert as `Published`.
  - Original submission flagged (by the classifier, or fail-safe on classifier timeout/open-circuit, architecture.md) ã immediate single-transaction insert as `PendingModeration`; no read-model write, no event.
  - Edit to a `PendingModeration` or `Rejected` review (nothing currently public to protect) ã overwrites content in place, re-runs the classifier, and follows the same Created-path branching as a fresh submission.
  - Edit to a `Published` review, cleared by the classifier ã applied immediately in place, fires `ReviewUpdated`.
  - Edit to a `Published` review, flagged by the classifier ã staged as `PendingRevision`; the currently-`Published` content is left untouched and visible (see Modeling Decision 7) until the moderator decides.
  - Moderator accept on `PendingModeration` ã transitions to `Published`; moderator accept on a staged `PendingRevision` ã applies the staged content in place, `PendingRevision` cleared, `Published` unchanged.
  - Moderator reject on `PendingModeration` ã transitions to `Rejected`; moderator reject on a staged `PendingRevision` ã discards the staged revision, `Published` content unchanged, no event (edge-cases.md's "Moderation-rejection racing" edge case, "dissolved by construction").
  - Moderator reject/takedown on an already-`Published` review (no staged revision involved ã a later report/re-flag of already-live content, edge-cases.md's "Abusive content" mitigation) ã transitions to `Rejected`, and ã unlike a pre-publish reject ã **does** fire `ReviewUnpublished`, because this content was actually counted while live (see Modeling Decision 11).
  - Author retraction (`DELETE /reviews/{id}`), any time, from any non-`Retracted` status ã transitions to `Retracted`; fires `ReviewUnpublished` only if `firstPublishedAt` was set (i.e., the review was ever actually public) ã never for a `PendingModeration` or `Rejected` (never-published) review being withdrawn.
- **`firstPublishedAt` is the single field governing every publish/removal-event-firing decision.** Set exactly once, the first time `status` transitions into `Published` from a state where it had never been `Published` before; never reset or cleared thereafter, even once the review is later `Rejected` or `Retracted`. `ReviewSubmitted` fires iff this is the transition that sets `firstPublishedAt` for the first time (regardless of whether that transition came from an original clean submission, a moderator-accepted first-time flagged submission, or an edit to previously-never-published content); `ReviewUpdated` fires iff `firstPublishedAt` was already set before this content change; `ReviewUnpublished` fires iff `firstPublishedAt` was set before this removal (see Modeling Decision 6).
- **Monotonic terminal states.** `Rejected` and `Retracted` are both terminal ã no transition of any kind is legal out of either, and no transition into `Retracted` from `Rejected` or vice versa. `Published` and `PendingModeration` are the only non-terminal values. A repeat moderator action against an already-resolved review, or a repeat retraction attempt, is a guarded no-op, never re-applied (mirrors `kart-payment-service/ddd-model.md`'s `PaymentIntentStatus` monotonic-terminal convention).
- **CanRead/CanWrite/CanDelete ownership (BRD ôÏ24.1.2 ã service-owned, decided here rather than left implicit now that it is a named platform requirement).** `Review` is the exact worked example BRD ôÏ24.1.2's own table names by name: "CanWrite (edit) gated by moderation status and an author-only edit window; CanDelete available to the author pre-moderation, to Admin post-moderation | Inline rule combining ownership + status." Concretely, against this aggregate's own already-modeled invariants above: `CanRead` on the write-model row is the author (own edit/retract flow) or a Support Agent/Admin performing moderation (requirement-spec ôÏ3's role-gated `PATCH /reviews/{id}/moderate`) ã a public reader never reads the PostgreSQL row directly at all (only the MongoDB projection, see database-design.md). `CanWrite` (edit) is gated by ownership **and** the 30-day edit window **and** `ModerationStatus` (requirement-spec ôÏ4: editable only by its author, within the window, and ã per the "Defer-until-outcome" invariant above ã what the edit actually *does* additionally depends on whether the review is currently `Published`, `PendingModeration`, or `Rejected`); `CanWrite` for the moderator accept/reject/takedown action is instead gated purely by coarse role (Support Agent/Admin), never ownership, since a moderator is by definition never the row's own author. `CanDelete` (retraction) is available to the author at any time from any non-terminal status (requirement-spec ôÏ4); there is no separate caller-invoked hard-delete for any role, including Admin ã the closest Admin gets to "deleting" already-public content is the moderator takedown path (`Rejected` transition), which is a status write, not a row deletion. **Mechanism chosen:** an inline rule combining ownership comparison, the `ModerationStatus`/edit-window state check, and a coarse-role check for the moderator path ã the same "inline rule combining ownership + status" mechanism BRD ôÏ24.1.2's own table already names for this exact service, extended with the role check its own moderation feature requires.
- **Audit-actor invariant (BRD ôÏ24.3).** Every mutation to `reviews` (database-design.md) stamps `created_by`/`updated_by` with the resolved acting principal ã the owning `userId` for the original submission and every self-service edit/retraction, or the moderating Support Agent's/Admin's own `userId` for an accept/reject/takedown action via `PATCH /reviews/{id}/moderate` ã never `NULL`. `review_outbox` mirrors whichever principal's write produced the published event (author or moderator, per above) for `created_by`, and a well-known `system:review-outbox-poller` for `updated_by` (the poller is the only process that updates a row after insert). `product_ratings`, `product_rating_ledger`, and `verified_purchase_records` are written exclusively by internal asynchronous projectors with no human/API caller of their own (Cross-Aggregate Interaction, below), so their `created_by`/`updated_by` stamp a well-known `system:*` principal id (`system:review-rating-projector`, `system:review-verified-purchase-consumer`) rather than ever being `NULL` or client-suppliable. These columns are auto-injected by the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor (ôÏ24.3 ã one platform-wide implementation referenced as a NuGet dependency, not built locally by this service; see database-design.md).

**Domain events (published):**
- `ReviewSubmitted` ã existing, BRD ôÏ10, expanded payload per requirement-spec ôÏ6 Q5: `orderId, sku, rating, reviewId, userId`. Fires exactly once per `ReviewId`, only on the transition that first sets `firstPublishedAt` (see above) ã never at raw-insert time for flagged content (requirement-spec ôÏ2 Domain Invariant).
- `ReviewUpdated` ã **new**, requirement-spec ôÏ6 Q3, architecture.md payload `orderId, sku, oldRating, newRating`. Fires only for a rating-affecting content change to a review that was already public before the change (see `firstPublishedAt` rule above). **Flag for the Event Design Agent stage** (already carried by architecture.md): retry/DLQ tier not yet finalized; provisionally the same 2x/`review.dlq` tier as `ReviewSubmitted`.
- `ReviewUnpublished` ã **new, introduced at this stage**, not present in any upstream approved doc. Payload: `orderId, sku, rating, reviewId, userId, reason: {AuthorRetraction | ModeratorTakedown}`. Closes a correctness gap this pass found: neither `ReviewSubmitted` nor `ReviewUpdated` gives `ProductRating` (or Product's/Search's own denormalized copies) any way to remove a review's contribution once it stops being public ã without this event, a retracted or taken-down review's rating would remain counted forever. Named and justified the same way `ReviewUpdated` itself was introduced (requirement-spec ôÏ6 Q3's own precedent: "let consumers adjust an aggregate incrementally instead of re-deriving it from a fresh create"), and reused across both trigger paths ã author retraction and moderator post-hoc takedown ã the same way `kart-order-service/ddd-model.md`'s `OrderCompensationTriggered` is reused across its own three distinct trigger paths (Modeling Decision 11). **Flag for the Event Design Agent stage:** retry/DLQ tier not decided here (provisionally 2x/`review.dlq`, same family as its siblings); **flag for a future pass on `kart-product-service`/`kart-search-service`'s own docs:** both currently list only `ReviewSubmitted`/`ReviewUpdated` as consumed ã this new event is a new integration surface for both, out of scope to edit here (different services' already-approved files).

## Lookup Projection: VerifiedPurchaseRecord

Not an aggregate root ã no invariants or lifecycle beyond upsert-on-consume, the same classification `kart-notification-service/ddd-model.md` gives `OrderUserIndex`/`TrackingOrderIndex`. Exists solely to let `POST /reviews`'s hard eligibility gate be a synchronous local read with no runtime dependency on Order Service's availability (architecture.md's Boundary Rationale; design-decisions.md's Cross-Service Verified-Purchase Check decision).

**Key:** `orderId`. **Fields:** `userId` (nullable until `OrderCreated` consumed), `skus: Set<Sku>` (nullable/empty until `OrderCreated` consumed), `deliveredAt` (nullable until `OrderDelivered` consumed).

- Upserted (userId, skus set) on consuming `OrderCreated` ã key fields `orderId, userId, items, total` per `kart-order-service/event-contract.md`'s own approved Published table; `items` is Order's own `OrderLineItem` array (`(sku, qty, unitPrice)` per `kart-order-service/ddd-model.md`), the source of `skus`.
- Upserted (`deliveredAt`) on consuming `OrderDelivered` ã key fields `orderId, deliveredAt` only (see this document's opening note for why `userId`/`sku` cannot come from this event).
- **`POST /reviews`'s gate check:** look up by the client-supplied `orderId`; reject ("no matching delivered order found yet, try again shortly" ã requirement-spec ôÏ6 Q2's own exact wording) if no record exists, or `deliveredAt` is still null, or the caller's authenticated `userId` doesn't match the record's `userId`, or the submitted `sku` is not in `skus`.
- **Idempotent consumption, no separate dedup ledger needed.** Both consumed events are full-field upserts (set `userId`+`skus`, or set `deliveredAt`) ã replaying either produces the same resulting row every time, the identical reasoning `kart-notification-service/ddd-model.md` Modeling Decision 9 gives for why `UserNotificationPreferenceUpdated` needs no dedup ledger of its own.
- **Ordering-race handling (no new mechanism invented).** RabbitMQ gives no cross-routing-key ordering guarantee even between two events from the *same* publisher (`order.order.created` vs. `order.order.delivered` are different routing keys per `kart-order-service/event-contract.md`), so `OrderDelivered` could in principle be consumed before the `OrderCreated` that seeds this record's `userId`/`skus`. Treated as transient, not permanent: an `OrderDelivered` arriving with no existing record simply upserts `deliveredAt` onto a partial row (userId/skus still null), and the gate check above naturally still rejects until `OrderCreated` fills in the rest ã no separate retry-and-requeue mechanism is even needed here, unlike `OrderUserIndex`'s dependent-event case, because `VerifiedPurchaseRecord`'s own two upserts commute regardless of arrival order (ADR-0020's identical race, resolved even more simply here since both writes target the same key/row rather than chaining through a second key).

## Aggregate: ProductRating

**Entity (root):** `ProductRating` ã identified by `Sku` (reference, `kart-product-service`). The canonical rating aggregate ADR-0014 assigns to Review: the system-of-record `avg`/`count` every other bounded context's own copy (Product's `ratingSummary`, Search's `RatingSignal`) is a denormalized projection of.

**Value objects:**
- `RatingAverage` ã the current `avg` (double), maintained by weighted incremental adjustment, never a full recompute (design-decisions.md's Concurrency-Control decision).
- `RatingCount` ã the current `count` (int).
- `ProcessedReviewLedger` ã `Map<(OrderId, Sku), RatingContributionState>` where `RatingContributionState = { lastAppliedRating: Rating | null }` (`null` once a `(orderId, sku)` entry has been unpublished/retracted). The idempotency mechanism design-decisions.md calls "deduplicated by the event's `reviewId`" ã see Modeling Decision 4 for why this ledger is keyed on `(orderId, sku)` rather than raw `reviewId`.

**Fields:** `sku`, `avg: RatingAverage`, `count: RatingCount`, `ledger: ProcessedReviewLedger`.

**Invariants:**
- **Updated only by consuming Review's own published events, never synchronously in the same transaction as the `Review` write that produces them.** This is a deliberate application of design-decisions.md's own framing ã "each maintains its own independent projection of the same event stream" (said there of Review's canonical aggregate *and* Product's denormalized copy, treated identically) ã to Review's internal architecture: `ProductRating` is an eventually-consistent projection of `ReviewSubmitted`/`ReviewUpdated`/`ReviewUnpublished`, exactly the same shape of consumer Product/Search/Analytics are, just running inside this bounded context.
- **Atomic incremental update, never full recompute.** `ReviewSubmitted` ã `count += 1`, `avg` recomputed via the standard weighted-running-average update. `ReviewUpdated` ã `avg` adjusted by `(newRating - oldRating) / count` with `count` unchanged. `ReviewUnpublished` ã `count -= 1`, `avg` recomputed removing the unpublished review's last-known contribution (design-decisions.md's Concurrency-Control decision; edge-cases.md "Rating aggregate recompute race").
- **Idempotent under at-least-once redelivery, deduplicated by `(orderId, sku)`, not by raw `reviewId`.** `ReviewSubmitted`'s own payload does carry `reviewId` directly, but `ReviewUpdated`'s architecture.md-approved payload (`orderId, sku, oldRating, newRating`) does not ã and `(orderId, sku)` is already 1:1 with `reviewId` by construction (`Review`'s own uniqueness invariant above), so it is a functionally equivalent, always-present dedup key across all three event types. A redelivered `ReviewSubmitted` whose `(orderId, sku)` ledger entry already has a non-null `lastAppliedRating` is a no-op; a redelivered `ReviewUpdated` whose ledger entry's `lastAppliedRating` already equals `newRating` is a no-op; a redelivered `ReviewUnpublished` whose ledger entry's `lastAppliedRating` is already `null` is a no-op.
- **Ledger entries are retained indefinitely, never deleted, including after an unpublish.** Deleting an entry on unpublish would make a *redelivered* `ReviewUnpublished` indistinguishable from a `(orderId, sku)` pair never processed at all, reopening exactly the double-decrement race the ledger exists to close ã the same unbounded-retention trade-off `kart-notification-service`'s `OrderUserIndex`/`TrackingOrderIndex` and `kart-search-service`'s `perReviewRatings` map already accept.

**Domain events (consumed):** `ReviewSubmitted`, `ReviewUpdated`, `ReviewUnpublished` ã all owned by this same bounded context's own `Review` aggregate (an internal, same-service consumption relationship, not a cross-service ACL reference).

**Domain events (published):** none ã `ProductRating` is queried directly by Review's own read surface (`GET /reviews/{sku}/summary` or equivalent, ADR-0014 ã final shape is the API Design Agent's job), never itself a further event source.

## Cross-Aggregate Interaction

- **`Review` ã `ProductRating`:** one-directional, asynchronous, via the Outbox relay and Review's own published events ã never a synchronous in-transaction update. This keeps the P95 < 300ms write-path budget (requirement-spec ôÏ3) from ever having to include a full aggregate recompute, and treats `ProductRating`'s own freshness relative to the `Review` write exactly the same way Product's/Search's denormalized copies already treat theirs ã eventually consistent, by design, not a gap.
- **`Review` ã `VerifiedPurchaseRecord`:** one-directional, synchronous, read-only. `POST /reviews` reads this projection at submission time; it never writes to it. All writes to `VerifiedPurchaseRecord` come exclusively from consuming `OrderCreated`/`OrderDelivered`, entirely decoupled from any `Review` aggregate transaction ã textbook justification, per the transaction-boundary test, for why this is not folded into `Review` itself.
- **The synchronous automated classifier call** sits inside `POST /reviews`/`PATCH /reviews/{id}` before any `Review` write (or before a `PendingRevision` is staged), but is not itself a domain aggregate ã a stateless application-service call, the same non-aggregate treatment `kart-payment-service/ddd-model.md` gives the external gateway call inside its own charge-creation flow. Unlike that gateway call, a retry of the classifier call has no harmful side effect (a classification lookup is read-only/idempotent-in-effect) ã this is exactly why `IdempotencyKey` does not need Payment's separate reserve-then-confirm ledger here (Modeling Decision 2).

## Modeling Decisions & Assumptions (resolved here, not escalated ã engineering defaults, revisable)

1. **Two aggregates plus one lookup projection, not one merged bounded context (transaction-boundary test applied).** `Review`, `ProductRating`, and `VerifiedPurchaseRecord` are three separate write boundaries because none is ever required to commit atomically with another: `Review`'s write reads `VerifiedPurchaseRecord` but never writes it; `ProductRating` is only ever written by an asynchronous consumer of `Review`'s own already-committed, already-published events. Merging `ProductRating` into `Review` would force every review submission's write-path latency to absorb the full aggregate-update cost design-decisions.md's own Concurrency-Control decision explicitly avoids by choosing incremental update over full recompute ã the same class of reasoning `kart-notification-service/ddd-model.md` Modeling Decision 1 applies to its own two-aggregate split.
2. **`IdempotencyKey` lives directly on `Review`, not a separate ledger aggregate ã a three-way comparative test, extending `kart-order-service/ddd-model.md`'s own Order-vs-Payment contrast one step further.** Order needed no separate ledger because it has no external call at all between reserving the key and committing. Payment needed a separate `IdempotencyRecord` specifically because its external gateway call has a costly, hard-to-undo side effect (a real charge) that must never be double-executed, forcing a reserve-then-confirm split across two transactions bridged by a non-transactional HTTP call. Review's write path *does* have a synchronous external call (the classifier) sitting in the same position ã but unlike a gateway charge, re-invoking a content-classification lookup on retry has no harmful side effect to guard against, so the specific structural reason Payment's design differs does not apply here. Review is therefore correctly modeled on Order's simpler pattern despite superficially resembling Payment's shape.
3. **`VerifiedPurchaseRecord` additionally consumes `OrderCreated`, not only `OrderDelivered` ã resolving a payload gap this pass found, using the platform's own established precedent (ADR-0020), not a new mechanism.** See this document's opening note for the full cross-check against `kart-order-service/event-contract.md`. This is a self-contained resolution ã it changes nothing about `OrderCreated`'s or `OrderDelivered`'s own already-approved payload shape, exactly the property ADR-0020's resolution for Notification also had ã but it does add a new inbound dependency to Review's own scope, flagged for a downstream update to `kart-order-service/event-contract.md`'s `OrderCreated` consumer list (today: Payment, Notification, Analytics) and to `architecture.md`'s Dependencies table, neither edited here (a different, already-approved file/table is out of this stage's own scope).
4. **`ProductRating`'s idempotency ledger keys on `(orderId, sku)`, not raw `reviewId`, despite design-decisions.md's literal wording ("deduplicated by the event's `reviewId`").** `ReviewSubmitted` does carry `reviewId` directly (requirement-spec ôÏ6 Q5's expansion), but `ReviewUpdated`'s own already-approved architecture.md payload (`orderId, sku, oldRating, newRating`) does not, and adding it now would mean reopening an already-approved payload for a field this model does not actually need ã `(orderId, sku)` is already 1:1 with `reviewId` via `Review`'s own uniqueness invariant, so it is a functionally equivalent key present on every one of the three event types without touching any upstream contract. Contrast with `kart-search-service/ddd-model.md`'s own choice of a full per-`reviewId` map for *full recompute* (chosen there specifically because neither event carries an ordering/version field safe for delta-mutation) ã Review's ledger solves the identical missing-ordering-field problem with a different, equally valid mechanism (idempotent delta-dedup instead of full recompute), because unlike Search, Review's own aggregate update is not a "no other option" recompute-from-scratch design.
5. **No new event is introduced for the classifier's fail-safe-to-queue path.** A classifier timeout/open-circuit routing a submission to `PendingModeration` is modeled as the same `ModerationStatus` value a genuine flag reaches ã an operational/ops-visibility distinction (why a review is queued), not a domain-meaningful state of its own, so no `ClassifierUnavailable`-style value object or event is modeled; this would be over-fitting an infrastructure resilience detail (architecture.md's Resilience Pattern) into the domain model.
6. **`firstPublishedAt` is the single mechanism unifying three separate event-firing decisions.** Rather than three independent flags/branches for "is this the first publish," "was this already public before this edit," and "was this public before this removal," one nullable, set-once timestamp answers all three questions by simple presence/absence ã the same economy-of-mechanism preference `kart-order-service/ddd-model.md` shows in reusing one `OrderCompensationTriggered` event across three distinct triggers (Modeling Decision 3 there) rather than inventing a parallel mechanism per case.
7. **`PendingRevision` is a narrow, edit-scoped two-phase staging mechanism ã not a reintroduction of the reserve/commit option design-decisions.md already rejected for the *original* submission case.** That rejection was about *original* submissions: staging every submission behind a reserve/commit step would cost every submitter latency for the sake of the flagged minority, which the chosen defer-until-outcome design already avoids more cheaply (insert once, gate visibility, not the write itself). `PendingRevision` solves a different, narrower problem the original decision never had to consider: an edit to content that is *already public*, where the alternative (hiding the currently-visible, already-cleared content while the edit is under review) would be strictly worse for the author than doing nothing ã so the existing published content is left untouched, and only the *edit's own effect* (on the read model, on `ProductRating`, on `ReviewUpdated`) is deferred. This is defer-until-outcome applied precisely to the specific side effect at risk, not a second, contradictory mechanism.
8. **Permanent `(order_id, sku)` occupancy through a terminal state is a real, notable consequence of composing two already-resolved decisions, not a new invariant invented here.** requirement-spec ôÏ4/ôÏ6 Q6 states the uniqueness constraint is never released and that edit is the only revision path; it does not separately address what happens once that one review reaches `Rejected` or `Retracted`. Combining the two: a customer who retracts their only review for an order/SKU, or whose only review for that pair is ultimately rejected and never further edited within the window, has no path to a fresh second submission for that pair, ever. This is stated explicitly here because it is a genuine product-facing consequence worth surfacing, not because the model has any other consistent reading available ã the uniqueness invariant, taken literally and permanently as written, allows no other conclusion.
9. **No separate "moderation queue" aggregate.** `PendingModeration` *is* the queue ã `SELECT * FROM reviews WHERE status = 'PendingModeration'` is the entire queue-read mechanism ã the identical reasoning `kart-order-service/ddd-model.md` Modeling Decision 5 gives for why `Order` needs no separate `Saga` aggregate (`OrderStatus` already *is* the Saga's state).
10. **The MongoDB read model is not modeled as its own aggregate**, deliberately contrasting with `kart-search-service/ddd-model.md`'s `SearchDocument` and `kart-delivery-tracking-service/ddd-model.md`'s `TrackingRecord` (both aggregate roots in their own docs). Those two are the *sole* representation of their respective state ã no separate normalized write table exists behind either. Review's read model is a secondary, rebuildable-from-source denormalization of the PostgreSQL `Review` row that already is the source of truth; per `ddd-cqrs-standards.md`'s read-model rule, there is no invariant here for an aggregate boundary to attach to.
11. **`ReviewUnpublished` is reused, not duplicated, across both its trigger paths (author retraction, moderator post-hoc takedown).** Both leave `ProductRating` needing the identical reversal (`count -= 1`, `avg` adjusted to remove the last-known rating) ã the same "functionally identical from every consumer's perspective" reasoning `kart-order-service/ddd-model.md` Modeling Decision 3 already applies to its own reused `OrderCompensationTriggered` across three distinct triggers. The `reason` payload field preserves the *why* for audit/ops purposes without requiring a second event or a second consumer code path.

## Proposed Ubiquitous Language Additions

New terms, to be appended to `docs/ddd/ubiquitous-language.md` under a new "Owned by `kart-review-service`" section (this pass): `Review` (aggregate root), `ReviewId`, `Rating`, `ModerationStatus`, `PendingRevision` (value objects), `ProductRating` (aggregate root), `RatingAverage`, `RatingCount`, `ProcessedReviewLedger` (value objects), `VerifiedPurchaseRecord` (lookup projection), and `ReviewSubmitted`, `ReviewUpdated` (existing BRD/requirement-spec events, previously only forward-referenced from `kart-product-service`'s and `kart-search-service`'s own "Referenced elsewhere" tables ã formally owned here), plus `ReviewUnpublished` (**new** domain event introduced at this stage, see Domain events above). `IdempotencyKey` is deliberately **not** re-registered as a new term ã it is the same generic, already-glossary-registered structural pattern `kart-order-service`'s own `IdempotencyKey` entry already documents, applied locally to a different aggregate, not new cross-service vocabulary (Modeling Decision 2). `UserId` (`kart-identity-service`), `Order`/`OrderId`/`OrderCreated`/`OrderDelivered` (`kart-order-service`), and `Sku` (`kart-product-service`) are referenced only, never redefined.

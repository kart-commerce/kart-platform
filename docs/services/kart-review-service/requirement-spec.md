# Requirement Spec: kart-review-service

## 1. Scope

Covers a single BRD service, **Review Service** (BRD §2.1, item 14): "Ratings, reviews, moderation." No merge with another BRD service is indicated, and no ADR exists proposing one — unlike Offer (see [kart-offer-service](../kart-offer-service/requirement-spec.md)), this is a standalone bounded context.

The BRD's treatment of Review is minimal: one row in the module table (§2.1), one row in the condensed service table (§5.4), and one row in the Event Catalog (§10). It is not one of the domains called out in §2.2 ("Domain Rules That Force Hard Engineering Decisions"), is not mentioned in §24 (Security), and has no dedicated deep-dive section (unlike Order/Inventory/Payment in §5.1–5.3). Where the BRD is silent, this pass closes each gap with an explicit, cited decision rather than leaving it implicit — see §6.

## 2. Functional Requirements

- Accept review submissions via `POST /reviews`, storing them in PostgreSQL as the write model, projected to a MongoDB read model (BRD §5.4: "PostgreSQL → MongoDB read"). **Read path (resolved, engineering default):** the BRD does not specify whether `/reviews` also serves reads; `GET /reviews?sku=...` (list, paginated, served from the MongoDB read model) is adopted as the read counterpart to `POST /reviews` (write), matching the platform's standard CQRS split already used by every other read-heavy service (BRD §6.1's table: Product/Search/Category are "Read-heavy... MongoDB/denormalized is source of truth for reads"). This is a routine REST/CQRS shape, not a business decision, and does not block sign-off; final path/query-param naming is the API Design Agent's job.
- Publish `ReviewSubmitted` **at the point a review becomes publicly visible** (see moderation workflow below — not necessarily at raw write-insert time for flagged content), carrying an expanded payload — see §6 Decision on payload fields (resolved).
- Consume `OrderDelivered`, published by **Order Service** per **ADR-0005** (Unify Order Terminal Event). **Resolved (was blocking Open Question):** consuming `OrderDelivered` is a **hard eligibility gate** on `POST /reviews` — see Domain Invariants and §6.
- **Moderation workflow (resolved, was blocking Open Question).** Two-stage pipeline: (1) an automated profanity/spam/toxicity filter runs synchronously on every `POST /reviews` submission; (2) content the filter clears is **auto-published immediately** (inserted into the write model, projected to the read model, `ReviewSubmitted` fires); content the filter flags is held in a **human moderation queue** and does not enter the read model or fire `ReviewSubmitted` until a moderator accepts or rejects it. Pre-publish gating therefore applies only to the flagged minority, not to every submission — see §6 for the full rationale, which resolves this together with the verified-purchase, moderation-backlog, and abusive-content questions.
- **Rating aggregate ownership (resolved, was blocking Open Question).** Ratings are Review's own domain responsibility (BRD §2.1, item 14). Review computes and owns the canonical rating aggregate (`avg`, `count`) from its own review history; Product's Event-Catalog-listed consumption of `ReviewSubmitted` "for rating recalc" (BRD §10) is a denormalized, eventually-consistent copy for Product's own read-model locality, not a second source of truth. See ADR-0014 (`docs/adr/0014-review-rating-aggregate-ownership.md`) for the full cross-service resolution — this also resolves the same contradiction independently flagged by `kart-product-service/requirement-spec.md`'s own Open Question #2, to be closed there by citation in that service's own next pass.
- **Edit/retraction (resolved, was unaddressed).** See §6 Decision — a bounded edit window is adopted, plus a soft-retract, both engineering defaults.

## 4. Domain Invariants

- `ReviewSubmitted` publication and `OrderDelivered` consumption must be idempotent under RabbitMQ's at-least-once delivery (BRD §3, Reliability row) — a redelivered event must not produce a duplicate review or double-count a rating.
- **Verified-purchase gate (resolved).** A review may only be submitted for an `(orderId, sku)` pair for which Review has previously consumed a matching `OrderDelivered` event for the submitting customer's order. `POST /reviews` is rejected (not merely flagged) if no matching delivered order is found at submission time. See §6 for why a hard gate was chosen over a soft prompt.
- **One review per order/SKU (resolved — previously missing entirely).** A customer may submit at most one review for a given `(order_id, sku)` pair, enforced via a **unique constraint on `(order_id, sku)`** in the PostgreSQL write model (equivalent in effect to a `(order_id, product_id)` constraint — `sku` is the product-identifier field name already used throughout Review's payload and the BRD's Event Catalog). A second distinct submission attempt for the same pair is rejected as a conflict; amending an existing review goes through the edit path instead (see below), not a second insert.
- **Edit window (resolved — previously unaddressed).** A submitted review may be edited by its author within **30 days of submission**, an unlimited number of times within that window; after the window closes, the review becomes immutable except via moderation takedown. A review may be **retracted (soft-deleted)** by its author at any time, which hides it from the read model and read-path API but does not hard-delete the row — preserving an audit trail consistent with Review's moderation responsibility (BRD §2.1). Hard delete is never exposed to customers.
- **Rating aggregate ownership (resolved).** Review is the canonical system-of-record for the rating aggregate (`avg`/`count`) derived from its own reviews; Product's own copy (BRD §6.2's `ratingSummary`) is a denormalized projection, not a second canonical source — see ADR-0014 (`docs/adr/0014-review-rating-aggregate-ownership.md`).
- **Moderation gate (resolved).** Content that the automated filter flags must not appear in the read model, and must not trigger `ReviewSubmitted`, until a human moderator has explicitly accepted it. Content the filter does not flag is published immediately without a human step.

## 6. Open Questions / Flagged Ambiguities

All items from the prior draft have been resolved this pass. Each is recorded below with its resolution path, per the three-tier closure rule (ADR citation / engineering default / new PENDING ADR). Nothing in this section is a residual blocker.

1. **Moderation workflow — RESOLVED (engineering default, single-service).** BRD §2.1 names "moderation" as a Review responsibility but gives zero workflow detail, and no ADR covers it (it is Review-internal, not cross-cutting). **Decision:** a two-stage pipeline — automated profanity/spam/toxicity filter first pass; content the filter clears auto-publishes immediately (no human step); content the filter flags is held in a human moderation queue and is neither visible nor aggregated until a moderator accepts or rejects it. **Why:** this is the industry-standard shape for review moderation at scale (moderating 100% of submissions by hand does not scale to any meaningful review volume; moderating nothing is inconsistent with BRD §2.1 naming moderation a primary responsibility) and it is the only option of the three considered in `edge-cases.md` that resolves the fake-review, moderation-backlog, and abusive-content edge cases without inventing a new cross-service event. **Trade-off accepted:** a small minority of flagged-but-actually-legitimate reviews incur queue latency before becoming visible; deemed acceptable since only flagged content (not all content) is delayed.
2. **Verified-purchase gating — RESOLVED (engineering default, single-service).** **Decision:** consuming `OrderDelivered` is a **hard eligibility gate** — `POST /reviews` is rejected unless a matching delivered order/SKU exists for the submitting customer. **Why:** matches typical e-commerce trust conventions (review authenticity directly affects buyer trust and is explicitly named a platform concern via "moderation" in BRD §2.1), and is consistent with Review's now-resolved verified-purchase invariant (§4). **Trade-off accepted:** a customer cannot leave a review before delivery completes (no "just ordered, in-progress" reviews), and Review takes on a synchronous dependency on having already processed the relevant `OrderDelivered` event before the submission request arrives — addressed operationally by rejecting with a clear "no matching delivered order found yet, try again shortly" response rather than silently failing, since `OrderDelivered` delivery is asynchronous and not instantaneous relative to the customer's own visit to the "leave a review" page.
3. **Edit/delete window — RESOLVED (engineering default, single-service).** **Decision:** 30-day author-edit window (unlimited edits within it), plus an any-time soft-retract; no hard delete. **Why:** 30 days covers the realistic window in which a customer's opinion of a product is still fresh enough to be worth correcting, matches common return/dispute windows used elsewhere in retail without inventing an unprecedented number, and soft-delete preserves the audit trail moderation implies is needed. **Trade-off accepted:** Product and Analytics (consumers of `ReviewSubmitted`) must handle a revision, not just a create — this requires a new `ReviewUpdated` event (payload: `orderId, sku, oldRating, newRating`) to let consumers adjust their aggregates without a full recompute. **Non-blocking, carried forward:** registering `ReviewUpdated`'s formal retry/DLQ tier in the platform Event Catalog (§10) is the Event Design Agent stage's job, the same way `kart-product-service/requirement-spec.md`'s own Open Question #6 (`ProductUpdated`) carries a comparable new-event registration forward rather than resolving it here.
4. **Rating aggregate ownership split — RESOLVED via new PENDING ADR (cross-cutting).** This is not a single-service call: it also independently blocks `kart-product-service/requirement-spec.md`'s own Open Question #2, and resolves a genuine BRD self-contradiction (§5.4's Product Consumes column vs. §10's Event Catalog). See ADR-0014 (`docs/adr/0014-review-rating-aggregate-ownership.md`): Review owns the canonical aggregate; Product's "rating recalc" consumption is a denormalized copy, not a second canonical computation.
5. **`ReviewSubmitted` payload omissions — RESOLVED (engineering default, single-service).** **Decision:** expand the payload from `orderId, sku, rating` to `orderId, sku, rating, reviewId, userId` (review free-text remains out of the event, served only via the `GET /reviews` read path). **Why:** `reviewId` is needed by any consumer that must later reconcile a revision or retraction against a specific review (§3 above's `ReviewUpdated` event and moderator-reject path both need it); `userId` is needed for Analytics' funnel/attribution use cases (ADR-0004's full fan-in explicitly means Analytics consumes every event, so withholding a field it would plausibly need serves no purpose); free-text is excluded to keep the event small and avoid duplicating the read-model as the source of full review content. This is additive and backward-compatible — no existing consumer's behavior changes. **Trade-off accepted:** a marginally larger event payload; judged negligible against the alternative of consumers being unable to reconcile revisions at all.
6. **One-review-per-order/SKU invariant — RESOLVED (engineering default, single-service).** **Decision:** add it as a Domain Invariant (§4), enforced via a unique constraint on `(order_id, sku)`. **Why:** the business intent to allow a *revised* review (not a second, independent one) is now explicit via the edit-window decision (#3 above) — a customer who wants to change their opinion edits their existing review rather than submitting a competing one, so the uniqueness constraint and the edit window are complementary, not in tension. **Trade-off accepted:** none of substance — this is the standard pattern for "one opinion per transaction" domains and was previously simply missing, not a genuine trade-off between two valid readings.

# Architecture: kart-inventory-service

## Boundary Rationale

`kart-inventory-service` is the bounded context that answers "is this SKU actually available, and can we hold it" — it owns stock truth per warehouse/SKU, reservation holds, and the oversell-prevention invariant as one cohesive domain (requirement-spec §5.2 Boundary Rationale: isolated "so oversell logic ... can be scaled, locked, and load-tested independently"). `docs/PLATFORM_BLUEPRINT.md` independently names it **the highest-contention service in the whole system**, citing its row-level `SELECT ... FOR UPDATE` mechanism and Order's Saga dependency on it as the reason it cannot be folded into Order or Payment — a distinct scaling/locking profile from both.

Unlike `kart-offer-service`, Inventory is a **mandatory, blocking participant in the Order Saga** (BRD §12), not an adjacent checkout service: it is BRD §12.1's first saga step (`Reserve stock`) and BRD §12.2's compensating action (`Release reservation`). This drives its availability posture — it sits in the 99.99% order-path tier (requirement-spec Decision 8), the same tier as Order itself, because a slow or unavailable Inventory directly blocks Order from ever starting the saga, not just degrading a secondary read path.

`ADR-0009` is the load-bearing resolution for this boundary: it confirms the Order→Inventory reserve call is genuine synchronous RPC (`POST /inventory/reserve`), correcting an overreaching BRD note that briefly implied every saga arrow was async. Everything below treats that reserve/release pair as synchronous, and every other integration (event publish/consume) as asynchronous — there is no third, ambiguous category left for this service.

### API Surface Consistency Note — Resolved

Two peers reach Inventory's read path by two different mechanisms, and this is intentional, not an inconsistency: Recommendation calls the existing public `GET /inventory/{sku}` REST endpoint directly (`kart-recommendation-service/architecture.md`) — a good fit since Recommendation's own read is not on `kart-cart-service/design-decisions.md`'s stated high-throughput gRPC-shaped path. Cart's own `design-decisions.md` already made a firm, reasoned decision to use gRPC for its checkout-time validation call, citing the reusable API standard's own illustrative example ("an inventory reserve check") verbatim — that decision is honored here rather than re-litigated: `api-contract.yaml` now defines `InventoryAvailabilityService.CheckAvailability`, a minimal read-only gRPC RPC (current available quantity per SKU, no reservation side effect) distinct from the public REST read endpoint. Both surfaces read the same underlying stock data through the same short-TTL cache-aside path (`design-decisions.md`'s caching decision) — this is two protocols for the same read concern, not two sources of truth.

## Internal-Only Concerns Confirmed Out of Scope for This Boundary

The following are stated here only to confirm they do **not** create additional cross-service edges, since they were flagged as blocking questions upstream and are now fully closed inside the requirement-spec/edge-cases docs, with no residual integration-contract question left for this stage:

- **Locking strategy** (pessimistic `SELECT ... FOR UPDATE`, requirement-spec Decision 1): purely an internal PostgreSQL concurrency mechanism; no peer service observes or depends on it.
- **Reservation TTL (15 minutes, 60s sweep)** (requirement-spec Decision 2): the sweep is deliberately self-contained inside this service's own boundary (no heartbeat/renew/query-back to Order), which is itself the boundary rationale for keeping TTL ownership here rather than in Order's saga — this is a *reinforcement* of the boundary, not a new edge. The one informational, non-blocking dependency edge-cases.md carries forward (Order's own per-step Saga timeout for `PaymentCompleted` should stay comfortably shorter than this 15-minute TTL) is `kart-order-service`'s own parameter to set — confirmed already set to 30s there (`kart-order-service/requirement-spec.md` §2), comfortably inside the 15-minute window — no action needed on this service's side.
- **Multi-warehouse allocation** (single-warehouse-first, ordered-lock fallback, requirement-spec Decision 3): confirmed above as invisible outside this boundary.
- **Backorder handling (none)** (requirement-spec Decision 4): entirely an internal write-path concern; no cross-service edge.
- **Replenishment trigger** (threshold-based automated reorder, requirement-spec Decision 5): the automated half is entirely internal (detected inside the same locked write that decrements stock); the manual half is **not** purely internal — it is now a confirmed inbound edge from Admin (see Dependencies table above, added per `kart-admin-service/architecture.md`'s follow-up flag). Either trigger converges on the same write path and the same externally-visible `InventoryReplenished` event; which trigger fired is invisible on that event's payload (`sku, qtyAdded, warehouseId`, per ADR-0007) regardless.

## Distributed-Monolith Risk

**One real, but intentional and already-scoped, risk: Order's synchronous reserve call makes Inventory's availability a hard gate on the Order Saga starting at all.** This is not a chatty or accidental coupling — it is a single, named synchronous call explicitly required by the BRD (§5.1's "sync reserve call," §12.1's first saga step) and already assigned the matching 99.99% order-path availability tier (requirement-spec Decision 8) precisely because of this dependency. It is flagged here, per this stage's responsibility, rather than left implicit:

- **Blast radius is bounded to one hop, not a chain.** Inventory itself has **no synchronous outbound dependency on any other service** — it never calls Order, Payment, or Shipping synchronously; every synchronous edge in the Dependencies table above (Order's reserve/release, the client/Recommendation read, Cart's lazy validation, Admin's manual replenishment) is **inbound to Inventory**, never outbound from it. The sync coupling is one-directional (peer → Inventory) and shallow, so this is not the multi-hop "distributed monolith" failure mode (a chain of synchronous calls each waiting on the next); the Order reserve/release edge specifically is a single, load-bearing dependency the BRD itself designates as necessary, and the other three inbound edges are either low-frequency control-plane calls (Admin) or already required to degrade/fail open on the caller's own side (Recommendation, Cart) rather than gating on Inventory's health.
- **Mitigations already designed, not left to hope:** a bounded, fail-fast `lock_timeout` on the reservation transaction (design-decisions.md's Resilience Budget decision) prevents one hot SKU's lock contention from degrading into unbounded queuing that could starve Order's connection pool; `kart-order-service`'s own NFR requires circuit breakers on all outbound calls, including this one, so a genuinely down Inventory fails Order's reserve step fast rather than hanging the whole saga indefinitely.
- **The release/compensation side is deliberately decoupled from Order's health**, partially offsetting the reserve-side coupling: the TTL sweep (requirement-spec Decision 2) self-heals a dangling reservation without requiring Order to ever call back, and release is reachable via three independent paths (explicit call, `OrderCancelled`, `OrderCompensationTriggered`) plus the sweep, all converging on one idempotent path — so Inventory's own correctness under an unhealthy/slow Order does not degrade further than the initial reserve call already implies.
- **Nothing to escalate beyond what's already recorded.** The coupling is real, named, bounded to one hop, and already has a matching availability tier and resilience budget — this does not meet the bar for re-opening the boundary (per this agent's failure-condition guidance) since it does not conflict with any existing bounded context in `service-boundaries.md`; it is simply the correct, BRD-mandated shape of this dependency, recorded here for the DDD/Database Design Agents downstream.

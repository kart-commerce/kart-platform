# Edge Cases: kart-inventory-service

## Edge Case: Oversell under concurrent reservation

- **What happens:** Two or more concurrent `POST /inventory/reserve` calls against the same SKU each see sufficient available stock and both succeed, driving stock negative / overselling the SKU.
- **Why it happens:** Read-then-write race on the same stock row with no serialization point between the availability check and the reservation write (requirement-spec Domain Invariant 1; BRD §2.2's "Inventory must never oversell" forcing function).
- **Solutions available (3):** Pessimistic row lock (`SELECT ... FOR UPDATE`) held for the transaction · Optimistic concurrency (version column, retry-on-conflict at commit) · Atomic conditional decrement (single `UPDATE ... SET qty = qty - N WHERE qty >= N` statement, no explicit lock)
- **Decision (4 bullets):**
  - Chosen: Pessimistic row-level locking (`SELECT ... FOR UPDATE`)
  - Why: requirement-spec §3/§6.1 already names this as the BRD's stated mechanism for Inventory stock rows
  - Trade-off accepted: Reduced concurrent throughput on hot SKUs (lock contention held for the transaction) vs. optimistic locking's higher throughput but retry-storm risk under heavy contention
  - Escalate: The BRD contradicts itself here (§2.2 says "optimistic locking," §6.1 prescribes pessimistic locking for the same rule — requirement-spec Open Question 1); this decision follows §6.1's concrete design table, but the wording conflict itself is unresolved and carried to the Architecture Agent

## Edge Case: Replenishment racing an active reservation

- **What happens:** A warehouse replenishment (stock increase) for a SKU commits concurrently with an in-flight reservation attempt against the same SKU/warehouse stock row. If replenishment's write path doesn't serialize through the same lock reservation uses, the replenishment's increment can be silently lost (overwritten) by the reservation's write, or the reservation can be evaluated against a stale pre-replenishment quantity — leaving stock truth wrong in either direction.
- **Why it happens:** requirement-spec Domain Invariant (§4): "Replenished quantity must be reflected in stock truth before it can be reserved by a new request" states an *ordering* requirement but not a *mechanism*. The Oversell edge case's chosen `SELECT ... FOR UPDATE` locking decision (requirement-spec §6.1) was scoped to the reservation write path only — nothing in requirement-spec FR §2's Replenishment section, or Open Question 5 (replenishment's trigger — manual admin vs. automated feed — still unstated), confirms replenishment's write goes through that same lock, so the two operations touching the same row are not guaranteed to serialize against each other.
- **Solutions available (3):** Require replenishment to acquire the same `SELECT ... FOR UPDATE` lock on the stock row before crediting quantity, inside one transaction (symmetric extension of the existing reservation-side decision) · Single atomic conditional `UPDATE stock SET qty = qty + N WHERE sku=... AND warehouse_id=...` statement with no explicit lock, relying on Postgres's implicit per-statement row lock to serialize against any concurrent `FOR UPDATE` holder · Optimistic version-column check on the replenishment write, retrying on conflict (same pattern already rejected for reservation in the Oversell edge case, for the same throughput/retry-storm trade-off)
- **Decision (4 bullets):**
  - Chosen: Same-row locking discipline extended symmetrically to replenishment — the replenishment write either opens with `SELECT ... FOR UPDATE`, or is expressed as a single atomic `UPDATE ... SET qty = qty + N` statement (functionally equivalent under Postgres's per-statement row locking); no separate concurrency mechanism for replenishment.
  - Why: This is not a new concurrency-control *decision* so much as confirming the existing one (requirement-spec §6.1's pessimistic locking) actually covers the row, not just the reservation *operation*. A replenishment write path implemented as an application-level read-then-write (read qty, compute qty+N, write) would **not** be safely covered by the existing decision even though both operations touch the same row, since the read step could use a stale pre-lock snapshot — the Domain Invariant's ordering guarantee only holds if both writers use a lock-compatible mechanism.
  - Trade-off accepted: Replenishment writes now contend for the same row lock as hot-path reservation attempts on the same SKU, adding latency to warehouse restock writes on high-contention SKUs — judged acceptable since replenishment is presumably far less frequent than reservation traffic, though the BRD gives no volume figures to confirm this.
  - Escalate: The Oversell edge case's locking *decision* generalizes to cover this race in principle, but is not automatically satisfied unless the replenishment code path is confirmed to use the same mechanism — and Open Question 5 (replenishment's trigger still unstated) means it isn't yet known which service/code path performs this write at all. This decision must be re-confirmed once Open Question 5 is answered; carried to the Architecture/DDD Agent stage alongside it.

## Edge Case: Reservation left dangling by a failed/abandoned Saga step

- **What happens:** Order reserves stock (`InventoryReserved` published) but then crashes, times out, or never issues a subsequent release/compensation call — the reservation hold is never released and permanently locks stock away from other customers.
- **Why it happens:** Saga orchestration failure or partition between Order and Inventory after a successful reservation but before the next saga step or compensation trigger runs (requirement-spec §5.2 TTL requirement exists precisely because this can happen; exact TTL/expiry mechanics are unstated — Open Question 2).
- **Solutions available (3):** TTL-based auto-expiry sweep (background job releases holds past TTL) · Saga-side timeout watchdog in Order Service that force-triggers compensation after N seconds of silence · Manual/admin reconciliation tooling only, no automated expiry
- **Decision (4 bullets):**
  - Chosen: TTL-based auto-expiry sweep, owned entirely inside Inventory
  - Why: Reservation holds already carry a TTL per requirement-spec §5.2; keeping cleanup inside Inventory's own boundary matches its Boundary Rationale of being independently scalable/lockable without depending on Order staying healthy
  - Trade-off accepted: Gives up a saga-side watchdog as the sole safety net — if Order's own compensation call arrives after Inventory's TTL sweep already released the hold, release must be idempotent (see Compensating-transaction edge case below) rather than relying on a single authority to release exactly once
  - Escalate: Exact TTL duration and whether expiry publishes an event (and which one) is unresolved — requirement-spec Open Question 2

## Edge Case: TTL firing while Payment is merely slow, not actually failed

- **What happens:** Inventory's TTL sweep releases a reservation (publishing `InventoryReleased`) because the hold has exceeded its TTL, while Payment for that same order is still legitimately processing — not failed or abandoned — and Payment then succeeds moments later, publishing `PaymentCompleted` for an order whose stock hold Inventory has already released (and which may already have been re-reserved/sold to another customer in the interim).
- **Why it happens:** requirement-spec Open Question 2 — reservation TTL duration and expiry mechanics are unspecified — means Inventory's TTL sweep is a blind, unconditional timer with no visibility into Payment's actual status. This races against Order's own (likewise unspecified) per-step Saga timeout for `PaymentCompleted`, and the two timers are uncoordinated. This is explicitly distinct from the "Reservation left dangling by a failed/abandoned Saga step" edge case above, which covers *permanent* failure/abandonment — here the Saga step is alive and will complete successfully, just slower than Inventory's TTL. `kart-order-service/edge-cases.md`'s "Saga Step Timeout vs. Eventual-Success Race" edge case documents the analogous pattern from Order's side (a fired timeout racing an in-flight downstream success).
- **Solutions available (4):** Accept the race and require Order to treat a late `PaymentCompleted` (arriving after TTL-release) as needing a fresh reservation attempt, refunding/cancelling if the SKU has meanwhile sold out (mirrors kart-order-service's "compensate-the-compensation" resolution for its analogous race) · Inventory extends/renews the TTL on a heartbeat/keep-alive signal from Order while Payment is still in-flight · Inventory's TTL sweep queries Order/Payment for live status before hard-releasing ("soft expiry"), mirroring kart-order-service's rejected "two-phase timeout" option for the same kind of race · Size the TTL generously above Payment's worst-case latency SLA so the race becomes statistically rare, without eliminating it
- **Decision (4 bullets):**
  - Chosen: Accept the race as a designed consequence of a self-contained TTL sweep (no heartbeat/status-query back into Order or Payment), paired with a mandatory "late success" handling path on Order's side: a `PaymentCompleted` arriving for an order whose reservation has already TTL-expired must trigger a fresh reservation attempt, with refund-and-cancel as the fallback if that reservation fails.
  - Why: The heartbeat/renew and query-before-release options would reintroduce exactly the cross-service dependency the "Reservation left dangling" edge case's decision explicitly rejected — TTL cleanup was deliberately kept "owned entirely inside Inventory... without depending on Order staying healthy" (that edge case's Decision, above). Staying consistent with that means Inventory's TTL sweep must remain a blind, unconditional timer; the coordination burden shifts to Order's saga-completion handling instead, matching how Order already handles the symmetric race on its own side.
  - Trade-off accepted: A legitimately in-flight-but-slow Payment can lose its stock hold and, if the SKU sells out before the fresh reservation attempt succeeds, force a customer-facing refund on an order whose payment actually succeeded — a real business/customer-experience cost being knowingly accepted, not a free engineering trade-off.
  - Escalate: How often this actually happens is entirely a function of the TTL's duration, which requirement-spec Open Question 2 leaves unset — this can't be sized or judged "acceptable" until a human sets that value. Carried forward together with Open Question 2; it also depends on Order's own per-step saga timeout budget being set to a compatible value, a second unresolved input to the same race.

## Edge Case: Multi-warehouse split-stock allocation races

- **What happens:** An order line's requested quantity isn't available at any single warehouse but is available when summed across two or more warehouses; concurrent reservation attempts against the same SKU across warehouses can independently see "enough combined stock" and over-allocate the same units, or fail unnecessarily if warehouse rows are checked one at a time without coordination.
- **Why it happens:** Stock truth is scoped per warehouse (requirement-spec §5.2 Responsibility: "Stock levels, reservations, warehouses"), so a single stock-row lock no longer serializes an allocation spanning multiple rows across multiple warehouses — the BRD defines no cross-warehouse allocation strategy at all (requirement-spec Open Question 3).
- **Solutions available (3):** Lock all candidate warehouse rows for the SKU in one transaction, in a fixed deterministic order (e.g. warehouse-id ascending) to avoid deadlock, then allocate greedily · Single-warehouse-only reservation (reject/fail if no one warehouse fully satisfies the line; push any splitting decision up to Order/Cart) · Pre-aggregated cross-warehouse stock view with optimistic allocation plus a compensating re-check
- **Decision (3 bullets):**
  - Chosen: Unresolved — escalated
  - Why: Whether the platform supports split-shipment orders at all is a product/business call the BRD never states, not an engineering default (requirement-spec Open Question 3)
  - Trade-off accepted: N/A — no option chosen; this must be decided by a human before the Architecture/DDD Agents can shape the reservation aggregate boundary, since "one warehouse vs. many per reservation" changes the aggregate's identity

## Edge Case: Compensating-transaction rollback correctness if Payment fails after Inventory already reserved

- **What happens:** Order successfully reserves stock (`InventoryReserved`), Payment then fails, and Order triggers Inventory's compensating release — but the compensation call/event can be delivered more than once (at-least-once delivery), or arrive after the same reservation has already been auto-expired by its TTL, risking either a double-credit of the released quantity back into available stock, or a duplicate `InventoryReleased` publish for a reservation already released.
- **Why it happens:** Saga compensation runs asynchronously and independently of the original reservation's lifecycle (BRD §12.2 Failure & Compensation Flow), and at-least-once delivery (BRD §3 Reliability NFR) plus TTL auto-expiry (see dangling-reservation edge case above) means two independent triggers can target the release of the same reservation. requirement-spec's Domain Invariant (§4) now states this explicitly for `InventoryReleased` by name — release "must not cause a duplicate `InventoryReleased` to be published for a hold already released" — rather than this being inferred only from the platform-wide idempotency NFR, since `InventoryReleased` itself was entirely absent from the prior draft of the spec (§2, §6 "Resolved since the prior draft," per ADR-0007).
- **Solutions available (3):** Idempotent release keyed on reservation ID (release is a no-op returning success if already released/expired) · Explicit reservation state machine with terminal states (`RESERVED → RELEASED`/`EXPIRED`) that rejects any transition out of a terminal state · Distributed lock/mutex around release per reservation ID to serialize concurrent release attempts
- **Decision (3 bullets):**
  - Chosen: Idempotent release keyed on reservation ID, backed by an explicit reservation state machine with terminal states
  - Why: Directly satisfies requirement-spec's release-idempotency Domain Invariant (§4) — including its now-explicit no-duplicate-`InventoryReleased`-publish clause — and resolves the double-release race against TTL expiry without new infrastructure
  - Trade-off accepted: Gives up the stronger single-point-of-truth guarantee a distributed lock would provide, in favor of a state-machine check inside the same PostgreSQL transaction — consistent with Inventory's PostgreSQL-as-source-of-truth NFR (requirement-spec §3)

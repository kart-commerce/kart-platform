# Requirement Spec: kart-inventory-service

## 1. Scope

Covers a single BRD service: **Inventory** (BRD §2.1 item 6: "Stock levels, reservations, warehouses"). No merge, no ADR needed — unlike `kart-offer-service`, this is a 1:1 mapping from BRD to service.

This is one of the BRD's three deep-dive services (§5.2, alongside Order §5.1 and Payment §5.3), not a condensed row (§5.4), so the BRD gives it real structural detail: a dedicated Responsibility/API/Database/Events/Boundary-Rationale table, an explicit concurrency-control mechanism (§6.1), a role as a mandatory participant in the Order Saga's both success and compensation flows (§12), and its own line in the domain-forcing-functions list (§2.2). This spec reflects that depth rather than condensing it.

`docs/PLATFORM_BLUEPRINT.md` independently flags `kart-inventory-service` as a **missing Phase 1 repo** and calls Inventory "the highest-contention service in the whole system," directly citing §5.2's `SELECT ... FOR UPDATE` oversell prevention and Order's Saga dependency on it as the reason it cannot be folded into Order or Payment. That framing (not itself a BRD requirement) is included here only as scope justification, not as a source of functional requirements.

## 2. Functional Requirements

### Stock & Reservation

- Maintain stock truth per warehouse, per SKU (§5.2 Responsibility; §2.1 item 6).
- Reserve stock synchronously when Order Service calls `POST /inventory/reserve` as the first step of the Order Saga (§12.1 Success Flow: `Order->>Inventory: Reserve stock`; §5.1 Dependencies: "Inventory Service (sync reserve call + async confirm)" — i.e. the call is synchronous, but saga advancement is confirmed via the async `InventoryReserved`/`InventoryReservationFailed` events). ADR-0009 independently confirms this reading — it corrects an overreaching BRD note that briefly implied every saga arrow (including this one) was async, and narrows that note to exclude the Inventory reserve step, which remains genuine synchronous RPC. No change to this spec's reading is required.
- Reservation holds must carry a TTL (§5.2: "reservation holds with TTL"). Duration and expiry behavior are not stated in the BRD — resolved directly as a single-service engineering default (Decision 2, §6): **15 minutes** from reservation creation, auto-expired by a periodic sweep (running every 60 seconds) owned entirely inside Inventory, which drives any expired-but-still-`RESERVED` hold through the same idempotent release path as an explicit `POST /inventory/release` call, publishing the same `InventoryReleased` event rather than a distinct expiry event.
- On successful reservation, publish `InventoryReserved` (orderId, sku, qty), consumed by Order to advance the saga (§10).
- On failed reservation (insufficient stock), publish `InventoryReservationFailed` (orderId, sku), consumed by both Order (to trigger saga compensation) and Cart (§10) — Cart's consumption implies cart-level customer feedback (e.g. item no longer available), though the BRD does not specify Cart's exact reaction.
- Never oversell: stock must not go negative under concurrent reservation attempts against the same SKU — called out explicitly as a platform-level forcing function (§2.2), and as the reason this service exists as an isolated boundary (§5.2 Boundary Rationale: "so oversell logic ... can be scaled, locked, and load-tested independently").
- Enforce the oversell invariant via row-level pessimistic locking — `SELECT ... FOR UPDATE` on stock rows, held for the duration of the reservation transaction, in PostgreSQL (§5.2 Database: "PostgreSQL with row-level `SELECT ... FOR UPDATE` on stock rows"; §6.1's concrete design table: "`SELECT ... FOR UPDATE` | Inventory stock rows | Prevents oversell under concurrent reservation"). §2.2's "optimistic locking / reservation patterns" phrase, resolved (Decision 1, §6): read as high-level forcing-function shorthand written before the concrete database design was finalized, superseded by §5.2/§6.1's specific and unambiguous mechanism — not a competing engineering directive. This spec, and every reference to Inventory's concurrency control below, consistently states pessimistic row-locking via `SELECT ... FOR UPDATE`.
- Multi-warehouse allocation (§2.1 item 6: "Stock levels, reservations, **warehouses**"): resolved directly (Decision 3, §6) as single-warehouse-first — a reservation attempts to satisfy the full requested quantity from one warehouse; only if no single warehouse can fully satisfy it does Inventory fall back to a cross-warehouse split, locking all candidate warehouse rows for that SKU in one transaction, in ascending `warehouse_id` order (deadlock avoidance), and allocating greedily until the requested quantity is met or exhausted (failing the whole reservation atomically otherwise).
- Backorder handling: resolved directly (Decision 4, §6) — insufficient stock always yields a hard `InventoryReservationFailed`; there is no backorder/waitlist path in this service's domain.

### Release / Compensation

- Release a held reservation via `POST /inventory/release` (§5.2 API).
- Consume `OrderCancelled` and release the associated reservation (§5.2: "Consumes: `OrderCancelled` (release)"; confirmed in Event Catalog §10, where `OrderCancelled`'s consumers are "Inventory, Coupon, Notification, Analytics").
- Consume `OrderCompensationTriggered` and release the associated reservation as the Order Saga's compensating action when a later saga step (Payment) fails (§5.2; §12.2 Failure & Compensation Flow: `Order->>Inventory: Release reservation (compensation)`). `OrderCompensationTriggered` previously had no Event Catalog entry at all despite being named as consumed here and published by Order (§5.1) — **now resolved by ADR-0007** (Event Catalog Completeness Pass): §10 gives it a full row (publisher Order; consumers Inventory, Notification, Analytics; payload `orderId, reason`; 3x retry; `order.dlq`). See §6's Decisions Log for the full resolution record.
- **On releasing a reservation — via the explicit API call, `OrderCancelled`, or `OrderCompensationTriggered` — publish `InventoryReleased` (orderId, sku, qty)**, the compensating counterpart to `InventoryReserved`, closing the loop back to Order so the Saga knows the compensating release actually completed before marking `OrderCancelled` (§12.2's diagram: `Order->>Inventory: Release reservation (compensation)` immediately followed by `Inventory-->>Order: InventoryReleased`, then `Order->>Order: Mark OrderCancelled`). This event is now stated directly in §5.2's own Publishes row and given a full Event Catalog entry at §10 (consumers Order, Analytics; payload `orderId, sku, qty`; 2x retry; `inventory.dlq`), both added per **ADR-0007**. The prior draft of this spec described the release *action* in full (API endpoint, `OrderCancelled`/`OrderCompensationTriggered` consumption) but never once stated that Inventory publishes an event when it happens — a gap this redraft closes.
- Release must be idempotent: a redelivered release call/event under at-least-once delivery (§3 Reliability NFR, platform-wide) must not double-credit the same hold back into available stock, and correspondingly must not re-publish a duplicate `InventoryReleased` for a hold already released. This is inferred from the platform-wide idempotency NFR, not stated specifically for Inventory release — flagged as a Domain Invariant below rather than assumed silently.

### Replenishment

- Publish `InventoryReplenished` when warehouse stock is restocked (§5.2). The prior draft of this spec flagged a double gap here: no listed consumer/payload/retry policy in the Event Catalog, and no stated trigger. **The Event Catalog portion is resolved by ADR-0007**: §10 gives `InventoryReplenished` a full row (publisher Inventory; consumer Analytics; payload `sku, qtyAdded, warehouseId`; 2x retry; `inventory.dlq`). **The trigger — neither §5.2 nor §10 nor any other BRD section states whether replenishment is a manual admin action, an automated upstream supplier/warehouse feed, or something else — is now resolved directly as a single-service engineering default (Decision 5, §6):** a threshold-based automated reorder signal (default reorder point: available quantity for a SKU/warehouse drops below 20% of that warehouse's target stocking level, per-SKU override supported), detected inside the same locked write that decrements stock (reservation or sale), surfaced to procurement/admin tooling; a manual admin-initiated replenishment (stocktake corrections, ad hoc supplier deliveries) remains supported through the identical replenishment write path regardless of trigger source, and both converge on the same lock-compatible write decided in edge-cases.md's "Replenishment racing an active reservation" edge case.

### Read

- Serve current stock level for a SKU via `GET /inventory/{sku}` (§5.2).

**Cross-cutting Security obligations added by BRD §24.1.2, §24.1.4, §24.1.5, §24.3 (cross-reference only — the detailed "why" lives in the BRD itself and in this service's own ddd-model.md/database-design.md, not re-derived here):**

- **§24.1.2 (CanRead/CanWrite/CanDelete, service-owned):** Inventory is the resource owner of its own `WarehouseStock`/`Reservation` data and defines its own fine-grained permission mechanism. `CanRead` on `GET /inventory/{sku}` is unconditional — a stock level is not personal data tied to any individual, the same public-read posture `kart-product-service`/`kart-category-service` take for their own catalog reads, and customer-facing "in stock" UI needs an unauthenticated-reachable read. `CanWrite` on `POST /inventory/reserve`/`POST /inventory/release` is instead gated by an **inline check that the caller is Order Service's own client-credentials principal** — the same "narrow, per-endpoint, service-principal-only" mechanism `kart-identity-service`'s internal lock/unlock endpoints and `kart-user-service`'s erasure-tombstone endpoint already use, since these two endpoints are synchronous service-to-service calls (§2/§5 above), never invoked directly by an end customer or Admin. No ownership comparison applies (a `Reservation` correlates to an `orderId`, not a `userId` — see §24.1.4 below) and no persisted grant table is needed (there is exactly one legitimate caller, Order Service, not a set of principals requiring individually-revocable grants the way Admin Service's own categories do). `CanDelete` is never exposed — a `Reservation` only ever transitions `Reserved → Released/Expired` (ddd-model.md's terminal-state machine), never a hard delete. The concrete mechanism is recorded as a Domain Invariant in ddd-model.md.
- **§24.1.4 (row-level security):** Inventory's write model is 100% PostgreSQL (`warehouse_stock`, `reservations`, `reservation_allocations`, `inventory_outbox_events` — database-design.md), no MongoDB read model (only a short-TTL Redis cache-aside). None of these tables carry a `user_id`-shaped ownership column: `reservations` keys on `order_id`, structurally identical to the BRD's own §24.1.4 worked example ("Payment's `payment_intents`... never stores `user_id` directly — it keys only on the opaque `order_id`, so a `user_id`-shaped row-level policy would not even apply"); `warehouse_stock` and `reservation_allocations` key on warehouse/SKU/reservation identifiers with no end-user dimension at all. Native RLS is therefore not applicable to any table here, for the same reason the BRD itself already gives for Payment. See database-design.md's "Row-Level Security Policy" subsection.
- **§24.1.5 (column-level security):** No sensitive/PII columns exist. `warehouse_stock`/`reservations`/`reservation_allocations` hold stock quantities, warehouse/SKU identifiers, an opaque `order_id` foreign reference, and reservation status/timing — none of it personal data about an individual, and `order_id` alone (with no name/address/contact data alongside it) carries no more sensitivity here than it does in Payment's own non-PII classification (BRD §24.1.5's own worked example: "No unmasked PCI data exists to protect... payment credentials are tokenized"). See database-design.md's "Sensitive / PII Column Classification" subsection.
- **§24.3 (default audit fields):** every mutable table in database-design.md (`warehouse_stock`, `reservations`, `reservation_allocations`, `inventory_outbox_events`) now carries `created_at`/`updated_at`/`created_by`/`updated_by`, auto-injected by the shared `kart-shared` `Kart.Shared.Auditing` package's `SaveChanges` interceptor reading the client-credentials/event-consumer principal (§24.3's own stated mechanism — one platform-wide implementation, not built locally by this service) — never a value a request handler sets directly. The acting principal is Order Service's client identity for a direct reserve/release call, a well-known `system:*` id for the TTL sweep and the `OrderCancelled`/`OrderCompensationTriggered` consumers, and either Admin's operator identity or `system:inventory-replenishment-trigger` for a replenishment write (Decision 5, §6) — never `NULL`.

## 4. Domain Invariants

- Stock must never go negative / be oversold under concurrent reservation attempts for the same SKU (§2.2) — the platform's canonical example of a domain rule forcing a hard concurrency-control decision, and the direct justification for this service's isolated boundary (§5.2).
- A reservation hold is time-bounded (TTL) and must not persist indefinitely once created (§5.2). Resolved (Decision 2, §6): TTL = 15 minutes from creation; a periodic sweep (every 60s) owned entirely inside Inventory auto-releases expired holds through the same idempotent release path as an explicit release, publishing the same `InventoryReleased` event.
- Reservation release — whether triggered by explicit `POST /inventory/release`, `OrderCancelled`, `OrderCompensationTriggered`, or the TTL sweep above — must be idempotent, so that at-least-once delivery (or the sweep racing an in-flight explicit release) cannot double-release the same hold back into available stock, nor cause a duplicate `InventoryReleased` to be published for a hold already released (inferred from the platform-wide idempotency NFR at §3, resolved as applying here per Decision 6; the BRD does not state this specifically for Inventory release or for `InventoryReleased`, which itself was entirely absent from this spec prior to an earlier redraft — see §2 and §6).
- `InventoryReserved` must only be published once the stock-row lock and reservation write are durably committed, never before, so Order cannot advance the saga on a reservation that could still fail to persist (inferred from the oversell invariant plus the Outbox pattern at §11; not stated explicitly in the BRD for this event).
- `InventoryReleased` must only be published once the compensating release (crediting the hold's quantity back to available stock) is durably committed, mirroring `InventoryReserved`'s atomicity requirement above — so Order cannot mark `OrderCancelled` on a compensating release that could still fail to persist (inferred from the same Outbox-pattern reasoning at §11; not stated explicitly in the BRD for this event, which was previously missing from this spec entirely — see §2 above and ADR-0007).
- Replenished quantity must be reflected in stock truth before it can be reserved by a new request (inferred from the "Stock truth per warehouse" responsibility at §5.2; resolved as a same-row-locking-discipline requirement in edge-cases.md's "Replenishment racing an active reservation" edge case — replenishment writes, threshold-triggered or manual, must go through the same lock-compatible write path as reservation).
- A reservation attempt resolves to a single all-or-nothing allocation decision inside one transaction, whether satisfied at one warehouse or split across several with candidate warehouse rows locked in ascending `warehouse_id` order (Decision 3, §6) — never a partial success that leaves some units allocated and others not.
- Insufficient stock at reservation time always yields a hard `InventoryReservationFailed`; no backorder/waitlist state exists in this service's domain (Decision 4, §6).

## 6. Decisions Log (formerly "Open Questions / Flagged Ambiguities")

This spec previously carried eight items as open questions, plus three gaps closed by ADR-0007. **All eleven are now resolved.** This section records what was decided and why, using the same three-tier closure approach applied platform-wide: (a) cite an existing ADR if one already answers the question, (b) decide directly as a single-service engineering default if the question has no cross-cutting impact, (c) raise a new ADR only if the question is genuinely cross-cutting or resolves a BRD self-contradiction and no existing ADR already covers it. None of the items below needed tier (c) — see the closing note.

**Closed by ADR-0007 (Event Catalog Completeness Pass)** — carried over from the prior draft, unchanged:

- *`OrderCompensationTriggered` had no Event Catalog entry.* §5.2 already listed it as consumed by Inventory and §5.1 listed it as published by Order, but §10 omitted it entirely. **ADR-0007** added a full row: publisher Order; consumers Inventory, Notification, Analytics; payload `orderId, reason`; 3x retry; `order.dlq`. See §2's Release/Compensation FRs and §5's API Surface table.
- *`InventoryReplenished` had no Event Catalog entry.* **ADR-0007** added a full row: publisher Inventory; consumer Analytics; payload `sku, qtyAdded, warehouseId`; 2x retry; `inventory.dlq`. (Its trigger — separate from the catalog entry — is resolved below as Decision 5.)
- *`InventoryReleased` was missing entirely* — not stated as a publish obligation anywhere in the prior draft despite being named in §12.2's Saga diagram. **ADR-0007** added it to BRD §5.2's Publishes row and to §10 (consumers Order, Analytics; payload `orderId, sku, qty`; 2x retry; `inventory.dlq`). Now a stated FR (§2), Domain Invariant (§4), and API Surface entry (§5).

**Decision 1 — Locking strategy: pessimistic, not optimistic.**
- Tier: BRD-citation fix (document-internal contradiction, not a new call).
- Resolved: row-level pessimistic locking (`SELECT ... FOR UPDATE`) on Inventory stock rows, held for the reservation transaction's duration.
- Why: BRD §5.2 ("PostgreSQL with row-level `SELECT ... FOR UPDATE` on stock rows") and §6.1's concrete database-design table ("`SELECT ... FOR UPDATE` | Inventory stock rows | Prevents oversell under concurrent reservation") both state this explicitly and concretely. §2.2's "optimistic locking / reservation patterns" phrase is high-level forcing-function shorthand written at the domain-rules level, before the concrete database design was finalized — it names *that* a concurrency decision is forced, not a competing mechanism. `docs/PLATFORM_BLUEPRINT.md` independently corroborates the pessimistic reading, citing §5.2's `FOR UPDATE` mechanism by name as the reason Inventory is "the highest-contention service in the whole system."
- Trade-off accepted: none new — this was always the BRD's concrete design. This closure pass only makes the spec state it consistently (FR §2, NFR §3, Domain Invariants §4) instead of leaving §2.2's shorthand to read as a live contradiction.
- Not an ADR: resolved directly by explicit, unambiguous BRD text with nothing left to adjudicate; affects no other service's contract.

**Decision 2 — Reservation TTL: 15 minutes, self-swept inside Inventory.**
- Tier: single-service engineering default.
- Resolved: TTL = 15 minutes from reservation creation. A periodic sweep (every 60 seconds), owned entirely inside Inventory, drives any expired-but-still-`RESERVED` hold through the same idempotent release path as an explicit `POST /inventory/release` call, publishing the same `InventoryReleased` event (no distinct expiry event).
- Why: 15 minutes comfortably covers a normal checkout window (cart review, address/payment entry, gateway round-trip) without indefinitely locking hot-SKU stock away from other customers on an abandoned checkout. Owning the sweep entirely inside Inventory (rather than a saga-side watchdog in Order) matches Inventory's own Boundary Rationale — independently scalable/lockable without depending on Order staying healthy.
- Trade-off accepted: a legitimately slow-but-succeeding Payment can lose its stock hold if it exceeds 15 minutes (see edge-cases.md "TTL firing while Payment is merely slow, not actually failed" — mitigated by Order's late-success handling, not eliminated by this choice). A shorter TTL would reduce that exposure further at the cost of more false-positive releases on merely-slow-but-fine checkouts; a longer TTL ties up contended stock longer under genuine abandonment. No BRD value exists to contradict.
- Not an ADR: the TTL number is invisible to any other service's contract — Order only observes `InventoryReserved`/`InventoryReservationFailed`/`InventoryReleased`, never the TTL value itself.

**Decision 3 — Multi-warehouse allocation: single-warehouse-first, fallback split with ordered locking.**
- Tier: single-service engineering default.
- Resolved: a reservation first attempts to satisfy the full requested quantity from one warehouse. Only if no single warehouse can fully satisfy it does Inventory fall back to a cross-warehouse split — locking all candidate warehouse rows for that SKU in one transaction, in ascending `warehouse_id` order (standard deadlock-avoidance ordering), allocating greedily until the quantity is met or exhausted, and failing the whole reservation atomically otherwise (Domain Invariant, §4: one all-or-nothing allocation decision per reservation, never a partial success).
- Why: matches BRD §2.1's "warehouses" framing and §5.2's per-warehouse stock truth without inventing a shipment-splitting product feature the BRD never authorizes. Keeping the split inside one transaction, with the same lock-ordering discipline as Decision 1, avoids introducing a second concurrency-control mechanism for the same stock rows. Inventory's published events carry no warehouse field (`InventoryReserved`/`InventoryReservationFailed` payload is `orderId, sku, qty` — BRD §10), so whether a reservation was satisfied from one warehouse or several is invisible to Order/Shipping; this is a reservation-transaction implementation detail entirely inside Inventory's own boundary, not a product decision about split shipments.
- Trade-off accepted: locking multiple warehouse rows in the fallback case adds latency/lock contention versus the single-warehouse path, accepted because the fallback only triggers when a single warehouse genuinely can't cover the line.
- Not an ADR: confined entirely to Inventory's own reservation-transaction design; Order/Shipping's contracts are unaffected since no warehouse-allocation detail crosses the API/event boundary. If a future need arises for Shipping to know about multi-warehouse fulfillment (e.g. to split a shipment into multiple packages), that would be a new, separate cross-cutting question for Order/Shipping's own docs — not a gap in this decision.

**Decision 4 — Backorder handling: none; hard fail only.**
- Tier: single-service engineering default.
- Resolved: insufficient stock at reservation time always yields a hard `InventoryReservationFailed`. No backorder/waitlist state exists in this service's domain.
- Why: §2.2 states only "must never oversell" — no BRD section describes a backorder/waitlist domain concept (queue state, partial-fulfillment saga steps, customer notification-on-restock flow), and Order/Cart's saga and event contracts (§10, §12) are built entirely around a binary reserve success/fail outcome. Introducing backorder support would require new domain concepts and saga steps the BRD never asked for.
- Trade-off accepted: customers hitting an out-of-stock SKU get a hard failure with no wait-list option — accepted as the simpler, BRD-consistent default; adding backorder support later remains possible as a genuinely new feature, not a retrofit of a broken assumption.
- Not an ADR: a scope-limiting default entirely internal to Inventory's own failure handling; Order/Cart already only ever needed to handle `InventoryReservationFailed` as a hard outcome.

**Decision 5 — Replenishment trigger: threshold-based automated reorder, plus manual admin path through the same write.**
- Tier: single-service engineering default.
- Resolved: a threshold-based automated reorder signal — default reorder point of 20% of a warehouse's target stocking level for a SKU, with per-SKU override support — detected inside the same locked write that decrements stock (reservation or sale), surfaced to procurement/admin tooling. Manual admin-initiated replenishment (stocktake corrections, ad hoc supplier deliveries) remains supported through the identical replenishment write path regardless of trigger source.
- Why: BRD §10 already implies replenishment happens (an `InventoryReplenished` event with payload `sku, qtyAdded, warehouseId` exists) without ever stating what triggers it. Automated threshold/reorder-point restocking is the standard inventory-management default: it needs no external system dependency to function, and routing both automated and manual replenishment through one write path directly satisfies edge-cases.md's "Replenishment racing an active reservation" resolution — a second, un-audited manual code path could otherwise race the locked decrement differently than the automated path does.
- Trade-off accepted: a single fixed default (20%) may not fit every SKU's sales velocity — mitigated by the stated per-SKU override rather than blocking sign-off on tuning it; the precise thresholds and any procurement/supplier integration are Architecture/Admin-service design details, not this spec's job.
- Not an ADR: a replenishment-trigger default lives entirely inside Inventory's own write path; it doesn't change any other service's API or event contract (the `InventoryReplenished` payload and consumer, resolved by ADR-0007, are unaffected by which internal trigger produced it).

**Decision 6 — Reliability guarantee applies uniformly to all of Inventory's own events.**
- Tier: single-service engineering default (direct extension of an existing platform-wide rule).
- Resolved: the platform-wide "at-least-once delivery + idempotent consumers" reliability guarantee (§3) applies to `InventoryReserved`, `InventoryReservationFailed`, `InventoryReleased`, and `InventoryReplenished`, not only to the Order/Payment events §3 names by example.
- Why: §3's Reliability row states the delivery *mechanism* platform-wide; naming Order/Payment is illustrative, not exhaustive — nothing in §3 scopes the guarantee away from other services' events. The Domain Invariants in §4 already require idempotency on Inventory's own release path regardless of whether §3 names it, so stating this explicitly here removes any ambiguity rather than introducing a new requirement.
- Trade-off accepted: none — this adds no new engineering work, it only makes an already-required guarantee explicit.
- Not an ADR: a clarifying restatement of an existing platform-wide rule, not a new cross-cutting decision.

**Decision 7 — Retry/DLQ tier: standard 2x/`inventory.dlq`, not Payment's 5x/paged tier — confirmed intentional.**
- Tier: single-service engineering default (risk acceptance).
- Resolved: `InventoryReserved`, `InventoryReservationFailed`, `InventoryReleased`, and `InventoryReplenished` all keep the standard 2x-retry/`inventory.dlq` tier (§10; ADR-0007). This is not upgraded to Payment's 5x-retry/paged tier.
- Why: Inventory's correctness under message loss already has an independent, non-messaging safety net that Payment lacks. The synchronous `/inventory/reserve` call fails fast for the caller regardless of the async event's fate, and the TTL sweep (Decision 2) self-heals a lost release whether or not `InventoryReleased` itself was ever successfully delivered. Payment has no equivalent self-healing path for a lost `PaymentCompleted`/`PaymentFailed`, which is precisely why it needs the heavier tier.
- Trade-off accepted: a DLQ'd Inventory event could sit unprocessed longer before a human notices than a paged Payment event would — accepted because Inventory's PostgreSQL row remains the durable source of truth regardless of whether the *event* describing it is ever delivered (Domain Invariants, §4).
- Not an ADR: confirms existing Event Catalog values (§10, ADR-0007) are intentional rather than proposing new ones; a single-service risk-acceptance call, not a cross-service contract change.

**Decision 8 — Availability tier: 99.99% (order-path tier).**
- Tier: BRD-citation fix (direct application of an existing rule, not a new call).
- Resolved: Inventory sits in §3's "order path" 99.99% availability tier.
- Why: §3 defines exactly two tiers by criticality position, not by service name ("order path" 99.99% vs. "secondary" 99.9%), and ties the tier to critical-path status. Inventory sits synchronously on the Order Saga's first step (§5.1: "sync reserve call"; §12.1's first saga arrow is `Reserve stock`) — unlike Offer, which BRD §5.5 explicitly places on a secondary path. Applying "order path" to Inventory is a mechanical application of §3's own stated rule, not an inference.
- Trade-off accepted: none — this is not a competing option, it is the correct application of an already-stated rule.
- Not an ADR: resolved directly by BRD text; no other service's tier assignment is affected.

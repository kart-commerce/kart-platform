# Requirement Spec: kart-payment-service

## 1. Scope

Covers the single BRD service **Payment Service** (BRD §2.1 item 9): "Payment intents, gateways, refunds." No merge; no ADR applies.

Payment is one of only three services the BRD treats as a dedicated deep-dive (§5.1 Order, §5.2 Inventory, §5.3 Payment), rather than folding it into the condensed "Remaining Services" table (§5.4). It is also the platform's explicitly named money-moving-critical service: BRD §2.2 states "Payment must never double-charge → forces idempotency keys + Saga compensation," and §10's Event Catalog gives `Payment*` events the platform's highest retry budget and only paged-on-call DLQ tier ("Money-moving events (`Payment*`) get the highest retry budget and human paging"). This spec draws on §5.3's deep-dive table plus every other BRD section that names Payment (§2.2, §3, §6.1, §10, §11, §12, §18, §22, §24, §27) — see §6 for gaps the BRD leaves unresolved despite this depth.

## 2. Functional Requirements

- Execute an idempotent charge against a payment gateway (`POST /payments/charge`, requiring a mandatory `Idempotency-Key` header, BRD §5.3).
- Enforce idempotent charge-attempt semantics via a unique constraint on the `idempotency_keys` table (BRD §5.3, §6.1: "guarantees exactly-once charge attempt semantics") — this is the concrete mechanism the BRD's domain rule at §2.2 ("must never double-charge") forces. **Resolved concrete semantics (was Open Question #1, single-service engineering default — BRD names the mechanism but not its parameters):** unique constraint scoped on `(idempotency_key, endpoint)`, not on the key alone, so the same key value can't accidentally collide across `charge` and `refund` calls; a 24-hour TTL-scoped replay window (BRD gives no number — this is a defensible industry-standard default, noted as such rather than left blank); replaying the same key with an identical request payload (hash match) within the TTL returns the original stored response with no second gateway call; replaying the same key with a **different** payload hash returns `409 Conflict`. After the TTL expires the key may be reused as a brand-new logical attempt.
- Require the same mandatory `Idempotency-Key` header on `POST /payments/{id}/refund`, not only on `charge` — BRD §24's "mandatory idempotency keys on all money-moving POSTs" names the general rule and a refund is money-moving; the BRD's §5.3 prose only calls this out explicitly for `charge`, but scoping it to charge alone would leave the platform's own §24 rule unmet for refunds. Same `(key, endpoint)`-scoped mechanism applies.
- Issue refunds against a completed charge (`POST /payments/{id}/refund`, BRD §5.3), supporting partial refunds. **Resolved (was Open Question #6, single-service engineering default):** `payment_intents` gains a `captured_amount` column; a new child table `refunds(id, payment_intent_id, amount, status, requested_at)` allows multiple partial refunds per intent; a DB-enforced constraint keeps `SUM(refunds.amount WHERE status='succeeded') <= payment_intents.captured_amount` (the concrete mechanism for Domain Invariant #6, below), checked in the same transaction that inserts a new refund row. Each partial refund publishes its own `RefundIssued` with its own `refundId`/`amount` — this fits the Event Catalog's existing payload shape (`orderId, refundId, amount`, BRD §10) without any payload change.
- Consume `OrderCreated` to initiate a charge for a newly created order (BRD §5.3 Consumes row; payload includes `orderId, userId, items, total` per §10).
- Publish `PaymentCompleted` on a successful charge, carrying `orderId, txnId` (BRD §10), consumed by Order and Analytics.
- Publish `PaymentFailed` on a failed or declined charge, carrying `orderId, reason` (BRD §10), consumed by Order to drive Saga compensation (§12.2). **Resolved gateway-call-vs-event-delivery retry distinction (was Open Question #9, single-service engineering default):** the gateway call itself gets its own bounded retry — up to 3 attempts, exponential backoff, only for transient/network-classified errors (timeouts, gateway 5xx) — a definitive decline/rejection is terminal immediately and is never retried; a circuit breaker trips after a threshold of consecutive transient failures. If the outcome is still ambiguous after retries are exhausted, the `payment_intent` is left `pending`/ambiguous rather than speculatively publishing `PaymentFailed` — resolution comes from the webhook confirmation path or a background reconciliation poll (see the new webhook endpoint below), consistent with `edge-cases.md`'s "Saga Compensation Against a Pending/Unknown Gateway State" decision. `PaymentCompleted`/`PaymentFailed` is published exactly once, only once the intent reaches a terminal state. This is independent of the 5x/`payment.dlq`/paged-on-call policy in §10, which governs redelivery of the already-decided event, not the upstream gateway call.
- Publish `RefundIssued` on a successful refund, now fully specified in the Event Catalog: `orderId, refundId, amount`, consumed by Order, Notification, Analytics, 5x retry, `payment.dlq`, paged on-call (BRD §10, gap closed by **ADR-0007** — see Open Questions).
- Ingest asynchronous gateway confirmation via a new inbound webhook endpoint, `POST /payments/webhooks/{gateway}` (gateway-facing only, not part of the client-facing surface). **Resolved (was Open Question #3, single-service engineering default, matches `edge-cases.md`'s existing decision):** ingestion is idempotent-by-gateway-event-ID with a monotonic state-ordering guard on `payment_intents` that rejects any transition older than the currently stored state. This is also the ingestion path chargeback/dispute notifications arrive through — see below.
- Consume gateway-reported chargebacks/disputes and publish a new event, `ChargebackReceived` (`orderId, paymentIntentId, chargebackId, amount, reason`; 5x retry, `payment.dlq`, paged on-call — same money-critical tier as `PaymentCompleted`/`PaymentFailed`/`RefundIssued`). **Resolved (was Open Question #7, genuinely cross-cutting — see ADR-0012 (`docs/adr/0012-payment-chargeback-handling.md`)):** on a chargeback notification, Payment marks the `payment_intent` `disputed` (blocking any new refund against it — the mechanism `edge-cases.md`'s "Refund Racing a Chargeback/Dispute" edge case already decided) and publishes `ChargebackReceived` so Order can react via its own compensation logic (hold/cancel, release inventory) without Payment separately initiating a refund for funds the gateway/bank has already reversed. `ChargebackReceived` is routed under the existing `payment.*` convention (§9), so it automatically falls under Notification's already-resolved consumed scope (ADR-0003) and Analytics' full fan-in (ADR-0004) with no change needed to either ADR.
- Integrate with one or more payment gateways, tokenizing card data at the gateway rather than storing it (BRD §2.1 item 9 names "gateways" plural; BRD §24: "payment data never stored — tokenized by the gateway provider"). **Resolved (was Open Question #2, single-service engineering default):** build a gateway-agnostic adapter interface (e.g. `IPaymentGatewayAdapter`) internal to Payment; ship exactly one concrete gateway implementation behind it at launch. Multi-gateway routing/failover is explicitly out of scope for v1 — the BRD states no routing rule, no failover SLA, and no per-gateway selection logic anywhere, so the plural "gateways" wording is read as future-proofing the abstraction, not a stated multi-provider requirement. Specific provider selection is an infrastructure/vendor choice, deferred to the Architecture Agent — a normal downstream decision, not a blocking gap.
- Track a refund as its own Saga instance, distinct from the Order Saga's state machine, because refunds "can be partial and asynchronous relative to the original order lifecycle" (BRD §12.2, "Payment Saga" note).
- Participate as a required step in the Order Saga's success flow (`Order->Payment: Charge` → `PaymentCompleted`, BRD §12.1) and as the trigger for compensation on the failure flow (`PaymentFailed` → Order releases the Inventory reservation, BRD §12.2).
- Never persist raw card data; mandatory idempotency keys on all money-moving POSTs (BRD §24, Security/API Security rows) — this applies platform-wide but is stated with Payment as the motivating case.

## 4. Domain Invariants

- A charge must never be executed twice for the same logical payment attempt (double-charge prevention, BRD §2.2: "Payment must never double-charge"), enforced via the unique constraint on `idempotency_keys` (BRD §6.1).
- `payment_intents` is the authoritative record of a charge's state; the BRD's own convention (write-heavy, PostgreSQL source of truth, §6.1) implies no other store or in-memory saga state should be treated as more authoritative than it.
- A refund is a distinct lifecycle from the original charge — "can be partial and asynchronous relative to the original order lifecycle" (BRD §12.2) — and therefore must not be modeled as a simple reversal step inside the Order Saga's own state machine.
- Raw card/payment data must never be persisted by this service; only gateway-issued tokens are stored (BRD §24, §5.3 Boundary Rationale).
- `OrderCreated` consumption and `PaymentCompleted`/`PaymentFailed`/`RefundIssued` publication must be idempotent under RabbitMQ's at-least-once delivery (BRD §3 Reliability row; §13's general Duplicate Message scenario: "Idempotency key check at consumer — second delivery is a no-op").
- A refund amount must not exceed the amount actually captured on the originating charge — inferred from standard payment domain logic and the BRD's general correctness posture (§6.1), not stated explicitly anywhere in the BRD. **Now given a concrete enforcement mechanism** (§2, Open Question #6 resolution): `SUM(refunds.amount WHERE status='succeeded') <= payment_intents.captured_amount`, DB-enforced in the same transaction as the refund insert — no longer just a flagged assumption.
- A `payment_intent` marked `disputed` (chargeback received, §2 Open Question #7 resolution) must reject any new refund attempt until the dispute is resolved — prevents a refund and a chargeback from independently reversing the same funds twice.
- Idempotency keys are scoped `(key, endpoint)` with a 24-hour replay window (§2 Open Question #1 resolution); a replayed key with a different request payload is a conflict (`409`), never a silent overwrite of the stored result.

## 6. Open Questions / Flagged Ambiguities

**Closure pass complete.** All eight items previously listed here as blocking or escalated have been resolved — six as single-service engineering defaults (decided directly below, in §2/§4/§5 above), one via an existing platform ADR, and one via a new cross-cutting PENDING ADR. Full resolution log:

1. **Idempotency-Key semantics — RESOLVED (single-service engineering default).** Was: BRD §5.3 requires the header and §6.1 states a unique constraint, but never defines TTL, scope, or same-key-different-payload behavior. Now: `(key, endpoint)`-scoped unique constraint, 24h replay window, identical-payload replay returns the stored result, mismatched-payload replay returns `409`. See §2 and Domain Invariants above.
2. **Payment gateway(s) not named — RESOLVED (single-service engineering default).** Was: BRD says "gateways" (plural) but names no provider and doesn't state whether multi-gateway routing/failover is required. Now: gateway-agnostic adapter interface, one concrete implementation at launch, multi-gateway routing explicitly deferred (not a v1 requirement) since the BRD states no routing rule or failover SLA to design against. Specific provider selection remains a normal downstream (Architecture Agent / infra) decision, not a blocker.
3. **Async gateway confirmation (webhook) ingestion absent — RESOLVED (single-service engineering default, matches `edge-cases.md`'s existing decision).** Now: new `POST /payments/webhooks/{gateway}` endpoint, idempotent-by-gateway-event-ID with a monotonic state-ordering guard. See §5.
4. **`RefundIssued` missing from the Event Catalog — RESOLVED, already covered by an existing ADR.** This was found to already be fixed in the current BRD: **ADR-0007** (Event Catalog Completeness Pass) added `RefundIssued`'s row (`orderId, refundId, amount`; 5x; `payment.dlq`, paged on-call) — the prior draft of this spec was written against a stale reading. No new ADR needed; citation updated throughout this doc.
5. **Refund SLA/timing unspecified — RESOLVED (single-service engineering default).** No BRD-stated override exists. Adopted: the general write-path NFR (P95 < 300ms) governs the synchronous accept/enqueue of `POST /payments/{id}/refund` (validate + write refund intent + Outbox row); actual gateway-side settlement and the resulting `RefundIssued` publication remain asynchronous and are deliberately **not** bounded by this NFR, consistent with BRD §12.2's own framing of refunds as "asynchronous relative to the order lifecycle." This is a considered trade-off, not a gap.
6. **Partial refund / partial capture unaddressed — RESOLVED (single-service engineering default).** Now: `captured_amount` on `payment_intents`, a child `refunds` table, DB-enforced `SUM(refunds.amount) <= captured_amount`, one `RefundIssued` per partial refund. See §2 and Domain Invariants above.
7. **Chargeback/dispute handling entirely absent — RESOLVED (genuinely cross-cutting; new PENDING ADR).** See ADR-0012 (`docs/adr/0012-payment-chargeback-handling.md`): new `ChargebackReceived` event, Payment marks the `payment_intent` `disputed` and blocks new refunds against it, Order reacts via its own existing compensation logic (hold/cancel, release inventory) without a duplicate Payment-side refund. Kept consistent with `kart-order-service`'s own resolved compensation-ordering decision (its edge-cases.md: release Inventory then refund, reverse execution order) — a chargeback is treated as an externally-forced reversal, not a trigger for a second, Payment-initiated refund.
9. **Gateway-call retry policy conflated with event-delivery retry — RESOLVED (single-service engineering default).** Now: gateway-call retry (up to 3 attempts, exponential backoff, transient errors only, circuit breaker) is explicit and distinct from the 5x event-delivery retry in §10; an ambiguous outcome after retries leaves the intent `pending` rather than speculatively publishing `PaymentFailed`, resolved later via the webhook path or reconciliation. See §2.

**Non-blocking, carried forward to the Architecture Agent (not a sign-off blocker):**

8. **Availability tier not assigned by name.** BRD §3 splits Availability only generically into "order path" (99.99%) vs. "secondary" (99.9%) and never places Payment in either explicitly — though Payment's presence in both Order Saga diagrams (§12.1, §12.2) and Order's declared Dependencies row (§5.1) makes the 99.99% read considerably stronger here than for a service the BRD doesn't name in any saga flow. This is a confirmation the Architecture Agent makes when it finalizes NFR targets, not an unresolved requirements gap — §3's NFR table above already states 99.99% as the working target.

# DDD Model: kart-shipping-service

One aggregate root, checked against the transaction-boundary test (`docs/standards/ddd-cqrs-standards.md`: "if two things can't be saved in one transaction, they are two aggregates, not one"). Everything this service does — persisting a shipment intent, calling a carrier out-of-band, and recording that carrier interaction's terminal outcome — happens against a single row keyed on `orderId` (requirement-spec §4 Domain Invariant #1; edge-cases.md's "Duplicate `ShipmentDispatched`" decision); no second entity in this bounded context is ever written in coordination with it, which is what keeps this a one-aggregate bounded context rather than several (contrast with `kart-payment-service`'s three-aggregate split — see Modeling Decision #1 below).

## Aggregate: Shipment

**Entity (root):** `Shipment` — identified by `ShipmentId`. The single authoritative record of one order's fulfillment attempt — carrier selection through label generation (BRD §2.1 item 16) — created only by consuming `OrderConfirmed` (requirement-spec §2/§5, ADR-0002); never created by a synchronous inbound call from Order (ADR-0002; architecture.md's Boundary Rationale).

**Value objects:**
- `OrderId` — reference field only (owned by `kart-order-service`); the natural, pre-existing business key this aggregate is uniquely keyed on. See Modeling Decision #1 for why this is what lets `Shipment` absorb its own idempotency guarantee directly, unlike `kart-payment-service`'s separate `IdempotencyRecord` aggregate.
- `Carrier` — the carrier that actually produced a valid label for this shipment; nullable until `ShipmentStatus` reaches `Dispatched`. This is the `carrier` field `ShipmentDispatched`'s BRD §10 payload carries — set exactly once, by the carrier attempt that succeeds, never speculatively before that (requirement-spec §4 Domain Invariant #2).
- `TrackingId` — the carrier-issued tracking identifier captured from the carrier's own label-generation response; nullable until `Dispatched`. This is the `trackingId` field `ShipmentDispatched`'s payload carries, and the same value `kart-delivery-tracking-service`'s own `TrackingId` value object treats as "first seen in `ShipmentDispatched`'s payload; [Tracking] never generates one itself" (`docs/ddd/ubiquitous-language.md`). Shipping is where that value is first captured on the platform — but see Modeling Decision #5 for why this pass does not re-claim the term's glossary ownership from Tracking.
- `ShipmentStatus` — `Pending | Dispatched | Failed`. `Pending` is the only non-terminal value, entered the instant the shipment-intent row and its Outbox marker commit in the same transaction as `OrderConfirmed` processing (design-decisions.md's Transactional Outbox decision); `Dispatched`/`Failed` are both terminal, reached only once the out-of-band carrier interaction resolves — mirroring the same monotonic-terminal shape `kart-payment-service/ddd-model.md`'s `PaymentIntentStatus` uses, for the analogous reason (an external call's outcome is confirmed after the fact, not synchronously).
- `FailureReason` — nullable, populated only when `ShipmentStatus` reaches `Failed`; carries `ShipmentCreationFailed`'s own `reason` payload field (ADR-0015) — set once all configured carrier options are exhausted, whether because the address fails carrier-side validation or no configured carrier services the destination (ADR-0015 §2; edge-cases.md's "Address validation failure" edge case).
- `CarrierSelectionPolicy` — the ordered, externally-configured carrier priority list (primary, then secondary on the primary's circuit tripping) a shipment's carrier-selection attempt walks through. See Modeling Decision #2 for why this is deliberately not a computed cheapest/fastest/contracted-rate business rule — requirement-spec Open Question #4's resolution.

**Invariants:**
- **Uniqueness — at most one `Shipment` per `Order`.** Enforced via a DB-level unique constraint on `order_id` plus a pre-carrier-call existence check performed before ever invoking the carrier API (requirement-spec §4 Domain Invariant #1; edge-cases.md's "Duplicate `ShipmentDispatched`" decision, design-decisions.md's Idempotency decision) — the same row this aggregate's eventual `Dispatched`/`Failed` state is written to is what a duplicate `OrderConfirmed` delivery is checked against, with no second ledger table involved (Modeling Decision #1).
- **`ShipmentDispatched` is published only once, and only once genuinely durable.** It fires exactly when `ShipmentStatus` reaches `Dispatched`, which itself requires both `Carrier` and `TrackingId` to be non-null (requirement-spec §4 Domain Invariant #2) — the transition function refuses to move to `Dispatched` with either field still unset; the event's own stated payload fields (BRD §10) are the only signal the BRD gives for what "dispatched" means, and this model enforces exactly that, nothing more.
- **Monotonic, terminal transitions — `Pending → {Dispatched, Failed}` only, and only once.** Once `ShipmentStatus` reaches either terminal value, no further transition is accepted; a redelivered or duplicate resolution of the out-of-band carrier call against an already-terminal `Shipment` is a no-op, never reapplied (mirrors `kart-payment-service/ddd-model.md`'s `PaymentIntentStatus` monotonic-transition invariant, applied here to the structurally identical async-confirmation shape).
- **`Dispatched` has no compensating/reversing action — ever.** Once published, `ShipmentDispatched` represents an external, carrier-side fact (a label exists); requirement-spec §4 Domain Invariant #3 and edge-cases.md's "Saga compensation must void/cancel an already-generated label" edge case both conclude this is structurally unreachable: per ADR-0002, shipment creation happens strictly *after* `OrderConfirmed`, past the point BRD §12.2's compensation flow (Inventory release, Payment refund) ever runs. No `ShipmentCancelled`/void-label transition exists in this model at all — not merely "unused," but genuinely not modeled (Modeling Decision #3).
- **A shipment that can never be created must still reach a terminal, signaled state.** Exhausting every configured carrier option (`CarrierSelectionPolicy`'s full priority list, each carrier already circuit-breaker/bulkhead-isolated per design-decisions.md) without producing a valid label moves `ShipmentStatus` to `Failed` and publishes `ShipmentCreationFailed` — a `Shipment` is never left indefinitely `Pending` with no terminal outcome (requirement-spec §4 Domain Invariant #4; ADR-0015). `Failed` is exactly as terminal as `Dispatched`: this aggregate has no automatic `Failed → Dispatched` retry transition of its own — whatever resolution a `Failed` shipment gets (manual ops retry, address correction, cancellation) is a new shipment-creation attempt or Order Service's own downstream fulfillment-exception workflow (ADR-0015 §3), never a mutation of the existing terminal row.
- **Idempotent event consumption.** A duplicate/redelivered `OrderConfirmed` for an `orderId` that already has a `Shipment` row — in any status, including still-`Pending` — is a no-op at the pre-carrier-call existence check: it never re-attempts a carrier call and never re-publishes an event for an order that already has one in flight or resolved (requirement-spec §3 Reliability NFR; edge-cases.md's "Duplicate `ShipmentDispatched`" decision).

**Domain events (consumed):**
- `OrderConfirmed` — existing, BRD §5.4/§10. Payload `orderId, address`. Sole shipment-creation trigger, consumed fully asynchronously — Shipping has no synchronous inbound endpoint Order invokes (ADR-0002).

**Domain events (published):**
- `ShipmentDispatched` — existing, BRD §10. Payload `orderId, carrier, trackingId`. Fired exactly once, only when `ShipmentStatus` reaches `Dispatched`. Consumed by Order, Notification, Delivery Tracking, Analytics.
- `ShipmentCreationFailed` — **new**, ADR-0015. Payload `orderId, reason`. Fired exactly once, only when `ShipmentStatus` reaches `Failed`. Consumed by Order, Notification, Analytics.

## Modeling Decisions & Assumptions (resolved here, not escalated — engineering defaults, revisable)

1. **No separate `IdempotencyRecord`-style ledger aggregate, unlike `kart-payment-service`.** Contrast directly with Payment's Modeling Decision #2 (`kart-payment-service/ddd-model.md`): Payment needed a second aggregate because a caller-supplied `idempotencyKey` has no natural connection to any resulting `PaymentIntent` until a charge actually succeeds, and the external gateway call sits between reserving that key and confirming the intent's own resulting state — two writes that structurally cannot share one transaction. Shipping's situation differs in the one respect that actually matters for the transaction-boundary test: `orderId` is already the natural, pre-existing unique business key this aggregate is keyed on, known *before* the carrier call is ever made, not derived from its outcome. The pre-carrier-call existence check (edge-cases.md, design-decisions.md's Idempotency decision) is a lookup against `Shipment`'s own unique-constrained `order_id` column — the same row the eventual `Dispatched`/`Failed` state is written to — so a duplicate `OrderConfirmed` delivery is caught with no second, separately-coordinated ledger table required. This is the same transaction-boundary test Payment's aggregate split applied, reaching the opposite conclusion because the identifying key's provenance is structurally different here.
2. **`CarrierSelectionPolicy` is an externally-configured ordered priority list, not a computed cheapest/fastest/contracted-rate business rule** — resolving requirement-spec Open Question #4, explicitly handed off to this stage by both requirement-spec ("carried to the Architecture/DDD Agent stages as normal downstream design work") and architecture.md ("a domain-model/strategy question... left to the DDD Agent stage"). The BRD (§2.1 item 16) names "carrier selection" by title only and gives zero selection criteria; edge-cases.md/design-decisions.md only ever fix the *failure-handling* shape (circuit breaker, primary→secondary fallback on trip), never a *business* selection algorithm. Modeling a cost- or SLA-comparison rule inside the `Shipment` aggregate would invent a BRD fact that does not exist — this model deliberately does not do that. Instead, carrier selection is modeled as walking a configured priority-ordered list (primary attempted first, secondary only once the primary's own circuit breaker trips), recording only the terminal winning `Carrier` or, on exhaustion, `FailureReason` — the same treatment `kart-delivery-tracking-service/ddd-model.md` gives its own per-carrier status-mapping/SLA tables (Modeling Decision #5 there): operational/configuration data, not a domain computation this aggregate owns. The actual priority order per deployment/destination, and the carrier roster itself, are left to configuration — consistent with architecture.md's own OQ5 resolution (at least two carrier integrations required structurally, exact roster deferred).
3. **`Dispatched` is a true terminal state with no compensating action — no `ShipmentCancelled`/void-label event is modeled at all.** requirement-spec Domain Invariant #3 and edge-cases.md's "Saga compensation must void/cancel an already-generated label" edge case both conclude this is structurally unreachable given ADR-0002 (shipment creation happens strictly after `OrderConfirmed`, past the point BRD §12.2's compensation flow ever runs). Building a void-label mechanism against a transition that can never legitimately occur under the BRD's own stated Saga design would be speculative engineering, not a defensible domain invariant. If a genuinely new capability is later introduced — e.g. a customer cancelling an already-confirmed, already-shipped order via Order's own `POST /orders/{id}/cancel` — that is Order Service's scope to raise first (edge-cases.md's own stated deferral), not a gap in this model to pre-emptively fill.
4. **No Outbox-relay domain entity, unlike `kart-order-service`'s `OrderEvent`.** design-decisions.md's Transactional Outbox decision persists a shipment-intent row plus an Outbox marker in one transaction, but nothing in this service's requirement-spec/architecture states that marker doubles as a domain-meaningful audit trail the way requirement-spec explicitly resolves for Order's own `order_events` table (`kart-order-service/requirement-spec.md` Open Questions resolution #5). Absent that explicit doubling, the Outbox marker is treated as generic technical infrastructure, out of this DDD model's scope — the same treatment `kart-payment-service/ddd-model.md` implicitly gives its own Outbox writes (no Outbox entity appears there either, despite Payment needing the identical publish-after-persist guarantee).
5. **`TrackingId` is not re-registered as a `kart-shipping-service`-owned glossary term, even though Shipping is where the value is first captured on the platform.** The existing, already-approved `docs/ddd/ubiquitous-language.md` registers `TrackingId` under "Owned by `kart-delivery-tracking-service`" ("the opaque tracking identifier first seen in `ShipmentDispatched`'s payload; this service never generates one itself"). That phrasing correctly implies Shipping (via the carrier) is the value's origin, but the term's glossary ownership was already assigned to Tracking before this pass ran. Per this agent's own rule ("every ubiquitous language term is owned by exactly one bounded context... do not redefine it"), this pass does not move or duplicate that ownership — it is flagged here explicitly, not silently worked around, as a known naming/ownership asymmetry for future glossary curation to revisit if it ever becomes a real conflict (it hasn't: both services' models remain internally consistent under the existing assignment).
6. **`Carrier` (this aggregate's value object) is a distinct term from `kart-delivery-tracking-service`'s own `CarrierId`, not a re-owned or renamed version of it.** Tracking's `CarrierId` identifies *which carrier adapter/webhook-routing client* owns a `TrackingRecord` (an integration/routing concern); Shipping's `Carrier` identifies *which carrier actually fulfilled a given shipment* (a business fact captured at label-generation time, part of `ShipmentDispatched`'s own payload). Both concepts are inspired by the same real-world entity (a carrier company) but serve different bounded-context purposes with different lifecycles — this is a deliberate, checked non-collision, not an oversight.
7. **`ShipmentStatus`'s legal transitions are exactly `Pending → {Dispatched, Failed}`, both terminal, and no other value or edge is modeled.** Derived directly from design-decisions.md's Transactional Outbox shape (persist a `Pending` intent fast, resolve out-of-band) and requirement-spec/ADR-0015's failure handling; no other transition is meaningful given this service's stated scope, so none is modeled speculatively — the same reasoning `kart-payment-service/ddd-model.md`'s Modeling Decision #5 applies to its own `PaymentIntentStatus`.
8. **The internal/administrative `/shipments` endpoint (requirement-spec §5, architecture.md's unconfirmed inbound row) introduces no new aggregate.** Whatever shape the API Design Agent gives it (query-only, or a manual-shipment-creation path for ops), it reads or writes through this same `Shipment` aggregate and its existing invariants — it is a second access path onto one aggregate, not a second bounded-context concept.

## Proposed Ubiquitous Language Additions

New terms, appended to `docs/ddd/ubiquitous-language.md` under a new "Owned by `kart-shipping-service`" section (this pass): `Shipment` (aggregate root), `ShipmentStatus`, `Carrier`, `CarrierSelectionPolicy`, `FailureReason` (value objects), `ShipmentDispatched` and `ShipmentCreationFailed` (domain events, the latter new per ADR-0015). `Order`/`OrderId` (`kart-order-service`) and `OrderConfirmed` (`kart-order-service`) are referenced only, never redefined. `TrackingId` is deliberately **not** added here despite Shipping being its value's origin — see Modeling Decision #5 above; it remains owned by `kart-delivery-tracking-service` in the existing, already-approved glossary. No other new term conflicts found.

# Edge Cases: kart-wishlist-service

## Edge Case: Alert Storm on Sitewide Price Drop

- **What happens:** A sitewide sale or catalog-wide repricing drops the price of thousands of SKUs at once; every affected SKU is wishlisted by many users, so the fan-out of `WishlistPriceAlertTriggered` (and downstream notification sends) spikes far above normal volume in a short window.
- **Why it happens:** `ProductPriceChanged` is consumed per-SKU (requirement-spec §2, Price-Drop Alerts) with no batching — one price event can map to N users × M wishlisted SKUs, and a sale touches many SKUs simultaneously, multiplying fan-out the same way flash-sale traffic multiplies fan-out for other event consumers on this platform.
- **Solutions available (3):** per-user alert batching/digest (collapse multiple triggers into one outbound notification within a time window) · rate-limit outbound `WishlistPriceAlertTriggered` publication with a queue-depth-driven consumer autoscale (same HPA-on-queue-depth pattern used elsewhere on this platform) · defer alert fan-out to an async digest job decoupled from the triggering event entirely.
- **Decision (3-5 bullets max):**
  - Chosen: per-user alert batching/digest with a **15-minute rolling window per user** (flush immediately if no new qualifying trigger arrives within the window; hard-cap flush at 60 minutes elapsed even if triggers keep arriving), backed by consumer autoscale on queue depth as a second line of defense.
  - Why: 15 minutes keeps alert latency reasonable for the common case (a handful of drops) while the 60-minute hard cap bounds worst-case fan-out during a sitewide repricing event, without requiring a redesign of the notification path; both numbers are plain engineering defaults (no cross-cutting impact on another service), consistent with this platform's existing HPA-on-queue-depth pattern.
  - Trade-off accepted: a user who wishlisted many sale items gets one grouped notification instead of N individual ones — acceptable since the BRD does not specify per-alert delivery granularity, and Notification's consumption of `WishlistPriceAlertTriggered` is confirmed (requirement-spec §2, §5; BRD §5.4 line ~192; ADR-0007), so batching is purely a Wishlist-side publish-cadence choice, not a Notification-contract question.
  - Note: the window value (15/60 min) is a config default, revisitable once real flash-sale traffic data exists — not a blocking sign-off item.

## Edge Case: Duplicate Alert Delivery from At-Least-Once Redelivery

- **What happens:** The same `ProductPriceChanged` message is delivered more than once for the same price change, causing `WishlistPriceAlertTriggered` to fire — and the user to be notified — twice for one actual price drop.
- **Why it happens:** RabbitMQ's at-least-once delivery guarantee (requirement-spec §3, Reliability row) means a consumer crash or nack-then-redeliver after processing but before ack replays the same event; the requirement-spec's domain invariant on idempotent alert publication (§4) exists precisely because the BRD does not itself state whether a redelivered event for an already-alerted price should be suppressed — resolved directly in that same invariant rather than left open.
- **Solutions available (3):** idempotency/dedup table keyed on `(userId, sku, priceObserved)` checked before publishing · consumer-side event-ID deduplication (dedupe on the inbound `ProductPriceChanged` message ID, same pattern this platform uses for other idempotent consumers) · outbox pattern on the publish side so `WishlistPriceAlertTriggered` itself is only ever emitted once per committed state transition.
- **Decision (3-5 bullets max):**
  - Chosen: dedup table keyed on `(userId, sku, priceObserved)` combined with the outbox pattern for publishing `WishlistPriceAlertTriggered`.
  - Why: matches the platform's existing Outbox-Poller-RabbitMQ pipeline (already used elsewhere in kart-platform) rather than introducing a one-off dedup mechanism, and directly satisfies the domain invariant that alert publication must be idempotent.
  - Trade-off accepted: requires a persisted dedup record per (user, sku, price) rather than pure event-ID dedup, which costs storage but survives the case where the *same* price is re-announced via a *different* message ID (e.g., a republish/backfill).
  - Relation to the alert-throttling decision (requirement-spec §4/§6): this dedup table exists to stop the *same* underlying event from re-firing an alert; it is a different mechanism from the separately-resolved 24-hour per-`(userId, sku)` cooldown, which throttles *distinct* qualifying price-drop events. The two compose without conflict — a redelivered event is suppressed by this table regardless of cooldown state, and a genuinely new qualifying drop within the cooldown is suppressed by the cooldown regardless of this table.

## Edge Case: Stale Wishlist Entry for a Discontinued Product

- **What happens:** A wishlisted product is discontinued or removed from the catalog, but the wishlist entry persists indefinitely and can still be evaluated against (nonexistent) future `ProductPriceChanged` events, or shown to the user as if still purchasable.
- **Why it happens:** The requirement-spec (§6, item 7 — resolved with an interim default this pass) notes the BRD states no event for product deletion/discontinuation that Wishlist consumes — only `ProductPriceChanged` — so there is no signal that would let Wishlist mark or prune an entry when the underlying product goes away. `kart-product-service`'s proposed `ProductDiscontinued` event would close this properly, but it is still only a proposal (no ADR, not in the BRD's Event Catalog), so this spec cannot depend on it existing yet.
- **Solutions available (3):** consume a broader Product lifecycle event if one is defined upstream (depends on `ProductDiscontinued` or equivalent being approved) · periodic reconciliation job that queries Product Service for wishlisted SKUs still active and flags/removes the rest · leave stale entries as-is and surface staleness only at read time via a synchronous existence check against Product Service on `/wishlist` reads.
- **Decision (3-5 bullets max):**
  - Chosen: periodic reconciliation job — Wishlist queries Product Service (`GET /products/{id}`, BRD §5.4) on a scheduled cadence for every SKU currently wishlisted by any user, and flags/prunes entries whose product no longer exists or is marked unavailable.
  - Why: this is buildable today against API surface the BRD already defines, rather than blocking on an upstream event proposal that may never be approved; it also degrades gracefully — even if `ProductDiscontinued` is approved later, this job keeps working as a second line of defense (e.g., for entries created before event-consumption was wired up, or if an event delivery is missed).
  - Trade-off accepted: staleness window is bounded by the reconciliation cadence (an implementation-level cron/schedule choice for the Architecture Agent, e.g. hourly or daily) rather than instant — acceptable because this is a read-experience rough edge (seeing a stale item briefly), not a correctness or money-moving concern like Inventory/Payment's edge cases.
  - Forward path — **now implemented, not merely forward-looking:** `kart-product-service`'s `ProductDiscontinued` proposal is now approved (`kart-product-service/ddd-model.md`/`event-contract.md`, both `status: approved`; payload `sku`, `discontinuedAt`; 3x retry). Wishlist consumes it directly as a second, event-driven invalidation path alongside the reconciliation job (requirement-spec §2, §4, §5, §6 item 7) exactly as this decision anticipated, rather than needing to revisit this decision from scratch — the reconciliation job is unchanged and remains the backstop for any discontinuation this event misses.

## Edge Case: Residual Wishlist State After a `UserDataErased` Event

- **What happens:** A user submits a verified GDPR erasure request (processed by User Service per ADR-0016); if Wishlist does nothing further, that user's wishlist entries, alert reference-price/cooldown state, and the redelivery-dedup table rows (all keyed by `userId`) persist indefinitely — including inside the Redis-backed per-user batching accumulator (`design-decisions.md`, "State-Store Mechanism for the Per-User Alert Batching/Digest Window") if a digest happens to be mid-flight for that user at erasure time.
- **Why it happens:** ADR-0016 already names Wishlist directly as an expected `UserDataErased` consumer (Context, Consequences), because Wishlist's own PostgreSQL write side and Redis accumulator both hold userId-linked data as a normal part of the price-alert feature (requirement-spec §2's GDPR Erasure Consumption FR, §4's matching Domain Invariant) — but neither the requirement-spec's FR nor any prior edge-case decision enumerated *every* place that data lives, and a batching window in flight at the moment of erasure is a genuinely distinct race this service's own Alert Storm mitigation introduced.
- **Solutions available (3):** consume `UserDataErased` and delete only the PostgreSQL-side wishlist entries, leaving the Redis accumulator to expire naturally via its existing 60-minute TTL (`design-decisions.md`) · consume `UserDataErased` and synchronously purge every store this service holds for that `userId` — PostgreSQL wishlist entries, the alert-cooldown/reference-price record, the dedup table, and the Redis accumulator key, in one handler — · defer erasure to the next scheduled reconciliation/batch job rather than an immediate event-driven purge.
- **Decision (3-5 bullets max):**
  - Chosen: option 2 — on consuming `UserDataErased`, synchronously delete (in the same handler, one pass) the user's wishlist entries and alert-cooldown/reference-price rows from PostgreSQL, the `(userId, sku, priceObserved)` dedup-table rows for that user, and the Redis accumulator key for that user, if present.
  - Why: ADR-0016 item 7's compliance-critical retry/paging tier for `UserDataErased` reflects that "an erasure event silently swallowed... is a compliance failure, not a tolerable staleness window" — letting the Redis accumulator merely expire on its own 60-minute TTL (option 1) would leave a real, if short-lived, window where erased-user data still exists and could still be flushed as an outbound digest; deferring to a batch job (option 3) reintroduces exactly that same avoidable window for no benefit, since nothing about erasure is cheaper to batch than to do immediately.
  - Trade-off accepted: the erasure handler must reach three different stores (PostgreSQL, the dedup table, Redis) instead of one, adding a small amount of handler complexity — accepted because a compliance-critical event is exactly the case where "erasure is fully synchronous and complete, not eventually consistent" is the correct bias, consistent with ADR-0016's own tombstone-now, not tombstone-eventually, framing for User Service's own erasure workflow.
  - Idempotency: a redelivered `UserDataErased` for an already-erased `userId` finds nothing left to delete in any of the three stores and is a no-op, not an error — satisfies the at-least-once-delivery discipline requirement-spec §4 already requires of every consumer here.
  - Relation to other decisions above: this is orthogonal to the "Duplicate Alert Delivery" dedup table and the "Alert Storm" batching window — both of those mechanisms continue to operate exactly as decided for every user who has *not* been erased; this decision only adds a teardown path for the one user who has.

## Edge Case: Noise Alerts from Undefined Price-Drop Threshold

- **What happens:** Trivial price movements (e.g., a $0.01 rounding adjustment or currency-conversion jitter) trigger `WishlistPriceAlertTriggered` the same as a meaningful sale price, producing alerts users learn to ignore.
- **Why it happens:** Requirement-spec §6, item 2 previously noted the BRD defines no minimum threshold (absolute or percentage) for what counts as a "drop" — every `ProductPriceChanged` that reflects any decrease was a candidate trigger under the FR as literally stated (requirement-spec §2, Price-Drop Alerts).
- **Solutions available (2):** enforce a minimum percentage/absolute delta before evaluating a price change as alert-worthy · enforce a minimum delta *and* a per-(user, sku) cooldown window so a slow multi-step drop doesn't re-trigger repeatedly.
- **Decision (3-5 bullets max):**
  - Chosen: option 2 — a minimum 5% price-decrease threshold from the tracked reference price, combined with the 24-hour per-`(userId, sku)` cooldown (requirement-spec §4, §6 items 2–3).
  - Why: a percentage floor eliminates rounding/currency-jitter noise regardless of the product's price point, and pairing it with the cooldown also closes the "slow multi-step drop re-triggers repeatedly" half of this same edge case in one mechanism rather than two independent ones.
  - Trade-off accepted: a genuine but sub-5% drop (or a second qualifying drop inside the 24h window) will not alert; acceptable given this platform's general bias toward bounding notification volume (see the Alert Storm edge case above) over maximum alert sensitivity, and 5%/24h are revisitable config values, not hardcoded architecture.

## Edge Case: Price Rebound During the Batching/Digest Window

- **What happens:** A price drop is detected and queued for per-user digest delivery under the batching window chosen as this file's own Alert Storm mitigation; before that window elapses and the notification actually sends, the price rebounds — either back to/above its pre-drop level, or drops further and then partially recovers — so the notification the user eventually receives cites an `oldPrice`/`newPrice` pair that is no longer true at delivery time.
- **Why it happens:** The domain invariant that `WishlistPriceAlertTriggered` must reflect the actual current price "at trigger time, not a stale cached price" (requirement-spec §4) only constrains the moment of trigger/publish; it says nothing about the gap between trigger and actual delivery to the user. That gap did not exist as a distinct race until this file's own "Alert Storm on Sitewide Price Drop" edge case introduced a per-user batching/digest window as the chosen mitigation — the batching window itself is what creates the race between a price that was true when captured and a price that may no longer be true by the time the digest is delivered. This is distinct from "Duplicate Alert Delivery from At-Least-Once Redelivery" above: that edge case is about the same event being redelivered and reprocessed; this one is about a single, correctly-processed alert going stale between trigger and delivery purely because of the batching delay.
- **Solutions available (3):** re-check the current price against the latest known price state immediately before digest send, and suppress the alert if the price has rebounded to/above the baseline it was triggered on · re-check at send time and, if rebounded, still send the digest but with corrected current-price content rather than the originally captured `oldPrice`/`newPrice` · do nothing — accept the batching window's inherent staleness risk as a known consequence of the Alert Storm decision, and deliver the digest with whatever price was true at trigger time regardless of what happens before delivery.
- **Decision (3-5 bullets max):**
  - Chosen: re-check the current price immediately before digest send. If the current price has rebounded to/above the original baseline (the reference price the alert was triggered against), suppress that item from the digest entirely — there is no longer a drop to report. If it rebounded partially but is still below baseline, send the digest with the corrected current price rather than the originally captured `oldPrice`/`newPrice`.
  - Why: this extends the existing trigger-time accuracy invariant (requirement-spec §4 — "must reflect the actual current price ... not a stale cached price") to cover the delivery gap the batching window itself introduced, rather than leaving that gap uncovered; a "price-drop alert" that tells a user a price is lower than it actually is at delivery time would undermine trust in the feature more than the added latency of one extra price read costs.
  - Trade-off accepted: adds a synchronous price re-check to the digest-send path (small added latency/complexity per digest) in exchange for never delivering a materially wrong price to the user; this is a single-service implementation cost, not a cross-cutting one, so it's decided here rather than escalated.
  - Note: this composes with the 5%/24h throttling decision (requirement-spec §4, §6 items 2–3) rather than replacing it — throttling controls *whether* an alert is queued at trigger time; this re-check controls *what price is shown* at send time for whatever was queued.

## Edge Case: Wishlist Entry Added After the Price Drop Already Happened

- **What happens:** A user adds a product to their wishlist after a `ProductPriceChanged` event for that product has already fired and already been processed for every user who had it wishlisted at that time. The newly-added entry did not exist when the event fired, so it is unclear whether this user should receive a retroactive alert for a drop that already happened, or whether their entry only starts reacting to price changes from the moment it is added, forward.
- **Why it happens:** The Price-Drop Alerts FR (requirement-spec §2) states only that Wishlist "consumes `ProductPriceChanged` to detect price movement on wishlisted products" — an event-driven, forward-looking consumption model with no stated temporal scope. Neither this FR nor any domain invariant in §4 defines what a wishlist entry's price baseline is at add-time, or whether Wishlist should look backward at prior `ProductPriceChanged` events for a product once a new entry starts watching it. This is a real product/UX question the BRD does not address at any level.
- **Solutions available (3):** no backfill — the entry's baseline is the price at add-time; only `ProductPriceChanged` events after that timestamp are evaluated for this user/SKU, matching the FR's literal forward-consumption model · retroactive backfill — at add-time, compare the current price against a reference price (e.g., the product's price immediately before its most recent drop) and immediately fire `WishlistPriceAlertTriggered` if the product is already discounted relative to that reference · read-time-only surfacing — show the product's current discount as UI state at the moment of add (e.g., an "already on sale" badge) without publishing a retroactive alert event at all.
- **Decision (3-5 bullets max):**
  - Chosen: combine option 1 and option 3 — no retroactive `WishlistPriceAlertTriggered` at add-time (the entry's reference price is captured as the price observed at add-time, per requirement-spec §4, and only subsequent `ProductPriceChanged` events are evaluated against it going forward); the product's already-discounted state, if any, is surfaced as read-time UI state (e.g., an "already on sale" badge) on the wishlist view rather than as a published event.
  - Why: matches the Price-Drop Alerts FR's literal forward-consumption wording (requirement-spec §2) rather than inventing a backfill requirement the BRD never states (the Requirement Agent's own rule against inventing requirements); it also avoids compounding the alert-volume risk the 5%/24h throttling decision above was sized against — backfilling an alert for every add of an already-discounted item would inflate alert volume in a way that decision didn't account for; and it still avoids leaving the user with zero information, since the discount is visible the moment they view their wishlist.
  - Trade-off accepted: a user who wishlists an already-discounted item gets no proactive push/email for that existing discount, only the passive UI badge on next view — acceptable because over-notifying at add-time is the higher-risk failure mode given this service's existing alert-fatigue concerns (Alert Storm, Noise Alerts edge cases above), and under-notifying here is recoverable (the user can still see and act on the badge) where over-notifying is not.
  - Implementation note (non-blocking, DDD Agent): this decision does confirm a wishlist entry needs a captured "reference price at add-time" field — carried to the DDD Agent alongside requirement-spec §6 item 6 (wishlist size limits) as an aggregate-shape input, not a new open question.
